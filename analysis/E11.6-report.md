# E11.6 Report

**Status:** implemented, typecheck clean, runtime smoke verified.

## Scope Executed

Surfaces the E11.1–E11.7 task system to tenant users via a new
**My Work** inbox page in the platform portal. Users can:

* see every task currently assigned to them (across products/workflows),
* filter by status,
* start a task,
* complete a task — which atomically advances the owning workflow via
  the E11.7 binding and surfaces the resulting workflow step,
* cancel a task,
* refresh the list (manual + automatic after every action).

No backend logic was duplicated in the UI. No workflow-edit, fan-out,
team-queue, or bulk-action surfaces were built.

## Assumptions

* **"Tenant portal" = the `(platform)` route group in `apps/web`.**
  That is the existing post-login shell (TopBar + Sidebar via
  `AppShell`, guarded by `requireOrg`). The other portal — `app/portal`
  — is the **injured-party** portal (a separate authenticated context)
  and is out of scope.
* **Reuse the BFF proxy convention.** Every existing product
  (`/api/lien`, `/api/care…`, `/api/fund`, …) reaches its service
  through a Next.js catch-all route handler that injects the
  `platform_session` cookie as a `Bearer` header. We add an analogous
  `/api/flow/[…path]` proxy for Flow rather than calling Flow
  directly from the browser. The browser never holds the bearer.
* **Gateway already routes Flow.** `appsettings.json` defines
  `flow-protected` matching `/flow/{**catch-all}` → `flow-cluster`
  with `PathRemovePrefix: /flow`. So `/api/flow/api/v1/tasks/me`
  → gateway `/flow/api/v1/tasks/me` → Flow `/api/v1/tasks/me`. No
  gateway change required.
* **Backend is the source of truth.** The UI never invents valid
  transitions. Buttons are *hidden* (not just disabled) for invalid
  source states, and every action is followed by a refetch — no
  optimistic updates.
* **Server-side default ordering is good enough.** `MyTasksService`
  already orders Open / InProgress before Completed / Cancelled, then
  by recency. The UI renders rows in server order so the queue feels
  like a queue, not a history.
* **Page size 50 is sufficient for v1.** Pagination controls are
  intentionally not built in this phase (the spec lists "page,
  pageSize" as supported on the API, not as required UI surfaces).
  When usage demands it, we can add a "Load more" button without
  changing the data layer.
* **`platform_session` JWT is acceptable to Flow's auth.** Flow's
  `/api/v1/tasks/me` and `/api/v1/workflow-tasks/{id}/{start|complete|cancel}`
  use the platform `Policies.AuthenticatedUser` policy, which the
  same JWT satisfies for the other services in this monorepo.

## Repository / UI Architecture Notes

* Web app: **Next.js 15 App Router**, route groups
  `(platform)`, `(control-center)`, `(admin)`. Platform pages live
  under `apps/web/src/app/(platform)/...` and inherit
  `requireOrg()` + `<AppShell/>` from
  `(platform)/layout.tsx`.
* Sidebar nav is data-driven from `apps/web/src/lib/nav.ts`:
  `PRODUCT_NAV` for per-product sections, `GLOBAL_BOTTOM_NAV` for the
  always-visible "ACCOUNT" group at the foot of every product
  sidebar. We add **My Work** to `GLOBAL_BOTTOM_NAV` so it is reachable
  from every product context, matching the cross-cutting nature of
  the inbox (a task in any product still belongs to one user).
* API access pattern: client components call `apiClient` (in
  `lib/api-client.ts`) with a relative path under `/api/...`. The
  Next.js dev/prod proxy / rewrites send those to the gateway, which
  is itself behind `GATEWAY_PREFIX = '/api'`. That is why the new
  task adapter prefixes paths with `/flow/api/v1/...` — the leading
  `/api` is added by `apiClient`, the `/flow` is consumed by the new
  BFF route.

## Task List / Component Notes

Three small components, all under `apps/web/src/components/my-work/`:

* `status-badge.tsx` — pill matching the visual language already used
  by the notifications and workflow surfaces: `Open` blue,
  `InProgress` indigo, `Completed` emerald, `Cancelled` gray.
* `priority-badge.tsx` — lower-emphasis row treatment with a remix-icon
  glyph (`Low` arrow-down, `Normal` dash, `High` arrow-up, `Urgent`
  alarm) and a colour ramp (`text-gray-500` → `text-red-700`).
* `task-row.tsx` — one card per task. Owns its per-action button
  loading state, but does NOT own the list — on success or on a
  stale-state error it calls `props.onChanged()` to ask the parent
  to refetch. Action buttons are *gated by status*:
  * `Open` → Start, Cancel
  * `InProgress` → Complete, Cancel
  * `Completed` / `Cancelled` → no actions
* `my-work-client.tsx` — root client component that owns the filter
  state, the list, the loading / empty / error shells, and the
  page-level "refetch" callback. Uses a `useRef`-backed sequence
  number so a slow prior fetch can't overwrite a newer one (rapid
  filter changes are safe).

The list is rendered as a `<ul>` of bordered cards (consistent with
the notifications page treatment) rather than a table, because each
row's metadata block (workflow / product / step / created / started /
completed) is variable-width and awkward in a fixed table layout.
Cards remain scannable thanks to the leading title + status badge +
priority cluster.

## API Integration Notes

* New BFF proxy: `apps/web/src/app/api/flow/[...path]/route.ts`. Mirrors
  the existing `lien` proxy verbatim except for the gateway path
  prefix (`/flow/...` instead of `/liens/...`). Forwards the
  `platform_session` cookie as `Authorization: Bearer ...`, preserves
  `X-Correlation-Id`, returns 204 / JSON / text faithfully.
* New typed adapter: `apps/web/src/lib/tasks/tasks.api.ts`, exported
  through `lib/tasks/index.ts`. Surface:
  ```
  tasksApi.listMine({ status?, page?, pageSize? })
                                → ApiResponse<PagedTasks>
  tasksApi.start(taskId)        → ApiResponse<TaskTransitionResult>
  tasksApi.complete(taskId)     → ApiResponse<TaskCompletionResult>
  tasksApi.cancel(taskId)       → ApiResponse<TaskTransitionResult>
  ```
  Status repeats are encoded the way Flow expects:
  `?status=Open&status=InProgress`.
* Type contracts in `tasks.types.ts` mirror the Flow DTOs exactly
  (`MyTaskDto`, `PagedResponse<T>`, `WorkflowTaskTransitionResult`,
  `WorkflowTaskCompletionResult`). `TaskCompletionResult` is declared
  as `extends TaskTransitionResult` so the additive E11.7 contract
  shape is enforced at the type level.
* Errors flow through the existing `ApiError` class. Callers branch
  on `err.isNotFound`, `err.isConflict`, `err.status === 422`, etc.

## State Management Notes

Deliberately minimal. No Zustand store, no SWR / React-Query — just
local `useState` + `useEffect` in the root client component:

* The list, loading flag, error message, and active filter all live
  in `MyWorkClient`.
* Each row owns only its per-action `busy` state, which is
  short-lived.
* After every mutation (success OR a stale-state error), the row asks
  the parent to refetch. The parent's `fetchTasks` uses a
  `useRef`-backed sequence counter so a stale response can't overwrite
  a newer one if the user rapidly toggles filters or re-clicks
  Refresh.

This matches the spec's "mutation → refetch" recommendation and keeps
the surface honest about workflow progression — the next-step task (if
any) appears on the very next render rather than via clever optimistic
patching that could drift from the engine.

## UX / Interaction Notes

* **Header.** Title + one-line subtitle; right-aligned Refresh button
  (with a spinning icon while loading).
* **Filter chips.** Five chips: `All`, `Open`, `In Progress`,
  `Completed`, `Cancelled`. The active chip is dark; inactive chips
  show a parenthesised page-local count when "All" is selected so the
  user has a soft cue of how many of each status are visible (precise
  per-status totals would cost N requests — out of scope).
* **Loading.** Skeleton cards on the first load. Subsequent fetches
  (after an action or filter change) keep the existing rows visible
  while the Refresh icon spins, so the surface doesn't flash.
* **Empty.** "You're all caught up" + secondary copy; reads "No tasks
  match this filter" + "Try a different status or clear the filter."
  when filtered.
* **Error.** Tinted red panel with the message + a "Try again" button
  that re-runs the fetch.
* **Toasts.** All actions go through the existing `useToast` hook
  (`lib/toast-context.tsx`). Successful Complete shows the workflow
  outcome explicitly:
  * advanced + still active → `Task completed — workflow moved to "<step>".`
  * advanced + workflow completed → `Task completed — workflow finished.`
* **Action error mapping** (no raw error dumps):
  * 422 Invalid state → `This task cannot be {started|completed|cancelled} from its current state. Please refresh.`
  * 409 on Complete → `The workflow has already moved forward, or another tab finished this task. Please refresh.`
  * 409 on Start/Cancel → `This task was already updated. Please refresh.`
  * 404 → `This task no longer exists. Please refresh.`
  * 401 → `Your session expired. Please sign in again.`
  * For any of `404` / `409` / `422` we **also** trigger an
    immediate refetch — by definition the local row is stale.
* **Action gating** is by *visibility*, not just disabled state, so
  the UI cannot offer an invalid transition. Backend enforces the
  same rules so the two stay in sync.

## Workflow Context Navigation

A first-class "Open workflow" deep-link is intentionally **not**
included in this phase: the existing workflow-detail UI is keyed on
`(caseId, workflowInstanceId)` (via the lien BFF), so the "open
workflow" target depends on which product owns the row. The MyTasks
DTO surfaces `productKey` and `workflowName`, but **not** a
context/case id (engine-internal field, intentionally hidden by
E11.5). The `productKey` and `workflowName` fields are rendered in
the row's metadata block so the user has the context they need to
find the workflow manually for now. Surfacing the right deep-link
shape is a follow-up that should land alongside a proper
cross-product context-id contract.

## Known Issues / Gaps

* **No deep-link to the owning workflow.** As described above —
  blocked on a portable cross-product context-id surface.
* **No in-page pagination.** Page size is hard-set to 50; rare power
  users with > 50 active tasks will see a soft "Showing 50 of N"
  hint but no "Load more" control. Easy follow-up when needed.
* **Per-status counts are page-local.** Counts beside the inactive
  chips reflect what is on the current page only. Precise tenant-wide
  totals would require either a dedicated stats endpoint or N
  per-status fetches; deliberately deferred.
* **No SSE / polling.** A user with two tabs open will see staleness
  until they refresh. Acceptable for v1 (the action error mapping
  surfaces the conflict and triggers a refetch). A live channel
  would be a separate phase.
* **No "open task drawer" detail view.** The card already shows every
  field on the MyTaskDto; there's nothing else to drill into without
  the workflow deep-link above.

## Recommendation

**Production-ready for the scoped slice.** Tasks are visible,
actions are correct, completion is bound to workflow progression
(via the existing E11.7 backend), the UI cannot offer invalid
transitions, error mapping is user-friendly, and there is no
regression in unrelated portal surfaces. Recommended follow-ups:
(1) cross-product workflow deep-link, (2) "Load more" pagination
when a tenant outgrows 50 tasks, (3) live updates / push when a
notification channel is wired in.

## API Integration Details

* Browser → Next.js BFF: `apiClient.{get,post}('/flow/api/v1/...')`
  → `/api/flow/api/v1/...`.
* BFF → gateway: `GATEWAY_URL/flow/api/v1/...` with
  `Authorization: Bearer <platform_session cookie>`.
* Gateway → Flow.Api: `flow-protected` YARP route strips `/flow`,
  forwards to `http://localhost:5012` (dev cluster).
* Calls used:
  * `GET  /api/v1/tasks/me?status=...&page=...&pageSize=...`
  * `POST /api/v1/workflow-tasks/{id}/start`
  * `POST /api/v1/workflow-tasks/{id}/complete`
  * `POST /api/v1/workflow-tasks/{id}/cancel`

## State Management Approach

`useState` + `useEffect` in `MyWorkClient` for filter, list, loading,
and error; `useState` in `TaskRow` for per-action busy. A `useRef`
sequence counter discards superseded fetch responses so concurrent
filter changes don't race. No global store, no caching layer — every
mutation triggers an explicit refetch so the list always reflects the
backend (including new tasks created by E11.7's workflow advance).
