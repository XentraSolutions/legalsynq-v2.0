# E12.2 Report

## Scope Executed

Wired the remaining lifecycle / task / admin events into the canonical
`NotificationEnvelope` contract from E12.1 so every system notification
in Flow is now produced from a single, template-keyed shape and routed
through the existing `INotificationAdapter` seam.

Concretely:

- `OutboxDispatcher.DispatchAdminAsync` now emits a notification envelope
  for `workflow.admin.retry`, `workflow.admin.force_complete`, and
  `workflow.admin.cancel` (previously audit-only). Recipient is the
  workflow instance assignee when known; the existing audit row is
  unchanged.
- `FlowEventDispatcher.MapToNotification` is rewritten to build a
  `NotificationEnvelope` and translate it via
  `NotificationContractTranslator.ToLegacySubmit`, matching the pattern
  used inside `OutboxDispatcher`. This brings the in-process
  task-assigned / workflow-completed paths onto the same contract +
  metadata (severity, category, correlationId, outboxId/templateKey
  pass-through).
- `TaskCompletedEvent` now produces a notification (template
  `task.completed`) when a completer user id is known. Previously this
  event only wrote audit.

No new producers were added in `TaskService` / `WorkflowEngine` /
`WorkflowSlaEvaluator` — all required hooks already exist from
E11.2 / E11.7 / E10.3 / E10.2. This task closed the contract-shape gap
on the dispatch side rather than scattering new producers.

## Assumptions

1. The outbox pattern is the canonical pipe for **workflow / SLA /
   admin** events; `FlowEventDispatcher` is the in-process pipe for
   **task** events. Both ultimately land in `INotificationAdapter` and
   both are now built from `NotificationEnvelope`. The split is
   pre-existing and out-of-scope to redesign per the spec
   ("do not redesign outbox or workflow engine").
2. Recipient resolution for admin actions is the current workflow
   assignee (`WorkflowInstance.AssignedToUserId`). No fan-out to roles
   or orgs is performed for admin actions in this phase — the
   role/org broadcast path is deliberately reserved for the
   role-aware producers added in earlier merges (#36/#40).
3. The legacy `NotificationMessage.EventKey` continues to carry the
   original outbox event-type string (e.g. `workflow.admin.retry`)
   so the wire payload remains bit-identical to pre-E12.2 sends. The
   canonical template key is published alongside as
   `Data["templateKey"]`.
4. Idempotency is delegated to the existing `outboxId` metadata: every
   send carries it and downstream collectors dedupe on it. No new
   dedup table introduced.
5. Notification adapter failures must not break the originating
   workflow / task operation — already enforced by the try/catch in
   `FlowEventDispatcher` and by the outbox processor's failure
   handling.

## Repository / Architecture Notes

- Producers stay close to where the event happens:
  - `WorkflowEngine` → outbox row → `OutboxDispatcher` (workflow
    lifecycle).
  - `WorkflowSlaEvaluator` → outbox row → `OutboxDispatcher` (SLA
    transitions).
  - `AdminWorkflowInstancesController` → outbox row → `OutboxDispatcher`
    (admin overrides).
  - `TaskService` (create / status update / assign) →
    `IFlowEventDispatcher` → `FlowEventDispatcher` (task events).
- No notification call is issued from a controller or from business
  code directly. All adapter sends are funneled through either
  `OutboxDispatcher` (durable) or `FlowEventDispatcher` (in-process).
- The canonical envelope and translator live in
  `shared/contracts/Contracts/Notifications/`; both producers reference
  the same constants from `NotificationTemplateKeys`.

## Event Producer Notes

| Source | Trigger | Producer |
| --- | --- | --- |
| `WorkflowEngine.CompleteAsync` | Workflow reaches terminal state | Outbox row → `OutboxDispatcher.DispatchLifecycleAsync` |
| `WorkflowSlaEvaluator` | Step crosses dueSoon/overdue/escalated boundary | Outbox row → `OutboxDispatcher.DispatchSlaTransitionAsync` |
| `AdminWorkflowInstancesController` | Operator retry / force-complete / cancel | Outbox row → `OutboxDispatcher.DispatchAdminAsync` |
| `TaskService.CreateAsync` / `AssignAsync` | Task created with assignment / re-assigned | `IFlowEventDispatcher.PublishAsync(TaskAssignedEvent)` |
| `TaskService.UpdateAsync` (Done) / `UpdateStatusAsync` | Task transitions to Done | `IFlowEventDispatcher.PublishAsync(TaskCompletedEvent)` |

Producers pass structured `BodyVariables` (workflow id, task id, step
key, dueAt, etc.) — no per-product strings are baked into producer
code. Subject/body fallbacks remain so the legacy adapter has a
sensible default until the notifications service starts rendering from
the template registry.

## Delivery Flow Notes

```
[engine / sla / admin] --> [Outbox row] --> [OutboxProcessor tick]
                                              |
                                              v
                                  [OutboxDispatcher.DispatchAsync]
                                              |
                          +-------------------+-------------------+
                          |                                       |
                          v                                       v
              [IAuditAdapter.WriteEventAsync]      [INotificationAdapter.SendAsync]
                                                              |
                                                              v
                                        [HttpNotificationAdapter -> notifications svc]

[task service] --> [IFlowEventDispatcher.PublishAsync]
                                |
                +---------------+---------------+
                v                               v
   [IAuditAdapter.WriteEventAsync]   [INotificationAdapter.SendAsync]
```

For both pipes the `INotificationAdapter` payload is now built from
`NotificationEnvelope` → `NotificationContractTranslator.ToLegacySubmit`.
Severity, category, correlationId, outboxId, and the canonical
template key all flow into `NotificationMessage.Data` so downstream
consumers can migrate at their own pace.

## Channel Support Notes

- `ChannelHints = ["system"]` on every envelope — in-app / system
  inbox is the only first-class channel today. Email is left as a
  responsibility of the notifications service (no producer-side
  channel routing).
- The legacy `NotificationMessage.Channel` is filled by the translator
  from the envelope's channel hints. No producer hardcodes a channel
  string.

## Validation Results

- Read-only verification of the producer side: every existing
  `INotificationAdapter.SendAsync` call site in Flow now goes through
  `NotificationEnvelope` + translator. Grep for
  `new NotificationMessage(` in `Flow.Application` /
  `Flow.Infrastructure` returns only the two private translator
  helpers.
- Backend build: `Flow.Application` + `Flow.Infrastructure` compile
  clean (typecheck only — full solution build is run by the workflow
  on restart).
- Runtime smoke (against the running `Start application` workflow):
  outbox dispatcher processes admin/lifecycle/SLA rows without throw;
  task assignment / completion paths log the new envelope translation
  metadata.
- No regression observed in My Work UI flows that depend on
  task notifications.

## Known Issues / Gaps

1. **Task events still bypass the outbox.** `TaskAssignedEvent` and
   `TaskCompletedEvent` are dispatched in-process by
   `FlowEventDispatcher`. If the notifications adapter is down at the
   instant the event is raised, the task notification is lost (audit
   is also best-effort). Promoting the task pipe to outbox is a
   follow-up — out of scope per "do not redesign outbox".
2. **Admin notifications only address the current assignee.** No
   role/org fan-out for admin overrides yet. The role/org broadcast
   plumbing from #36/#40 is wired for engine producers; admin path
   should be promoted in a follow-up once recipient policy is agreed.
3. **No template rendering on the producer side.** Subjects/bodies
   are still pre-rendered as a safety net for the legacy adapter.
   Once the notifications service consumes `templateKey` +
   `BodyVariables` natively, the producer-side strings can be
   dropped.
4. **Force-complete severity is producer-set, not config-driven.**
   Severity tiers per template should eventually live in the
   template registry, not in the producer.
5. **No automated tests.** Unit coverage for the new admin envelope
   and the rewritten `FlowEventDispatcher` mapping is left to
   follow-up task #44.

## Recommendation

Ready for production for the in-app / system channel. The change is
strictly additive on the wire (`NotificationMessage` shape unchanged),
template-key driven on the producer side, and idempotent through the
existing `outboxId` metadata. Follow-ups #44 (tests) and the task-pipe
outbox promotion should be sequenced before we depend on these
notifications for SLA-driven escalations in production tenants.
