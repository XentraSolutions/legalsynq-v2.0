# E11.2 Report — Task Creation from Workflow Automation

> Status: implementation complete; pending architect re-review.

## Scope Executed

E11.2 makes the workflow execution layer auto-create a `WorkflowTask` row
every time a `WorkflowInstance` enters an actionable step, so that operators
have a unit-of-work to act on without anyone having to call the task API
manually. Three call sites were wired:

1. `WorkflowEngine.StartAsync` — task created when the instance first lands
   on its initial step.
2. `WorkflowEngine.AdvanceAsync` — task created on every non-terminal
   transition.
3. `AdminWorkflowInstancesController.HandleAdminActionAsync` — task created
   only on `retry`; `force_complete` and `cancel` flip status off `Active`,
   so the factory's eligibility gate naturally short-circuits and no task is
   produced for them.

Out of scope (deferred): notification fan-out, assignee resolution, SLA-on-task
seeding, unique partial DB index for stricter dedup. The bidirectional
terminal-status↔timestamp invariant on `WorkflowTask.EnsureValid()` shipped
as part of E11.1 and is a load-bearing precondition for this work.

## Assumptions

- A `WorkflowInstance` is "actionable" iff `Status == Active` AND
  `CurrentStepKey != null`. Terminal statuses (`Completed`, `Cancelled`,
  `Failed`) are never actionable; `Pending` is treated as not-yet-actionable
  and waits for `StartAsync` to flip it to `Active`.
- A step with an existing Open or InProgress `WorkflowTask` for the same
  `(WorkflowInstanceId, StepKey)` already has someone working on it; we do
  NOT create a second task. Re-entry after all prior tasks at that step
  reached a terminal status (Completed / Cancelled) is treated as a fresh
  unit of work and DOES create a new task.
- `WorkflowInstance.TenantId` is authoritative — task rows copy that value
  explicitly rather than depending on `ITenantProvider`, because background
  flows (outbox replay, SLA evaluator) may execute outside a request scope.
- Admin retries can only re-enter the current step; they never advance, so
  the factory call after retry only ever produces a task at the existing
  `CurrentStepKey`.

The factory is a stateless service injected as scoped (matches DbContext
lifetime). It does NOT call `SaveChangesAsync`; it stages the new entity on
the change tracker and relies on the caller's existing unit-of-work commit.
This is what allows the task to land atomically with the lifecycle outbox
row in a single SQL transaction.

## Task Creation Logic Notes

The factory exposes a single async method:

```
EnsureForCurrentStepAsync(WorkflowInstance instance, CancellationToken ct)
```

The flow inside is:

1. **Eligibility gate** — return immediately if
   `instance.Status != Active` OR `instance.CurrentStepKey is null`.
2. **In-memory dedup** — scan the change tracker's `Local` collection for
   any pending insert with the same `(InstanceId, StepKey)` and a non-terminal
   status. This catches re-entrant calls inside the same unit-of-work
   (e.g. an automation that advances twice and lands back on the same step).
3. **DB dedup** — `AnyAsync` on `WorkflowTasks` filtered by
   `(InstanceId, StepKey, Status in (Open, InProgress))`. Skip if a row
   already exists.
4. **Construct & stage** — build a `WorkflowTask` with:
   - `Id = Guid.NewGuid()`
   - `TenantId = instance.TenantId`
   - `WorkflowInstanceId = instance.Id`
   - `StepKey = instance.CurrentStepKey!`
   - `Title = "{FlowDefinition.Name ?? ProductKey} — {StepKey}"`,
     truncated to 512 chars
   - `Status = WorkflowTaskStatus.Open`
   - `CreatedAtUtc = DateTime.UtcNow`
   Then `_db.WorkflowTasks.Add(task)`. No `SaveChangesAsync`.

The title falls back to `ProductKey` (always non-null on the instance) when
the loaded definition has no `Name`, so we never produce a `" — step_key"`
title with leading-orphan separator.

## Actionable Step Rules

A step is actionable iff:

- The owning instance has `Status == Active`.
- `CurrentStepKey` is non-null and non-empty.

Terminal advances (where `nextStage.IsTerminal == true`) are handled by
`WorkflowEngine.AdvanceAsync` setting `Status = Completed` BEFORE calling
the factory, so the eligibility gate suppresses task creation for the
terminal step. This means a workflow can have a final "Completed" step in
its definition without producing a phantom task that no one can ever close.

`Pending` instances are not actionable — `StartAsync` flips them to
`Active` first, then calls the factory, so the very first task is created
at the moment the workflow truly begins.

## Duplicate Prevention Notes

### Tenant query-filter caveat (load-bearing)

The `FlowDbContext` applies a global query filter
`e.TenantId == _tenantProvider.GetTenantId()` to every entity, including
`WorkflowTask` and `FlowDefinition`. The `AdminWorkflowInstancesController`
loads the parent instance with `IgnoreQueryFilters()` so platform admins
can act on instances outside their ambient tenant claim (or with no claim
at all).

If the factory's dedup `AnyAsync` ran with that filter active, a
cross-tenant retry would **silently miss** an existing Open / InProgress
task and produce a duplicate. Both DB reads in the factory therefore use
`IgnoreQueryFilters()` and add an explicit `TenantId == instance.TenantId`
predicate. This pins the read to the instance's authoritative tenant
regardless of the ambient request scope.

The `Add()` call deliberately stays under the normal change-tracker path —
the `WorkflowTask` is constructed with `TenantId = instance.TenantId` so
the row is tagged correctly even when `_tenantProvider.GetTenantId()`
returns a different value.

### Two-pass application-level dedup

- Pass 1 (`Local`) protects against the same logical operation calling the
  factory twice within one unit-of-work. Without this, two `Add` calls
  inside the same `SaveChangesAsync` would both succeed and produce
  duplicate rows.
- Pass 2 (DB `AnyAsync`) protects against the typical case: a task is
  already Open/InProgress because someone is actively working it and a
  separate retry/advance fires.

A unique partial index `(WorkflowInstanceId, StepKey) WHERE Status IN
('Open','InProgress')` would be a stronger guarantee against a narrow race
between two concurrent admin retries that read a Failed instance, both miss
the dedup query, and both insert. This is documented as a known-acceptable
race for this phase — admin retry is a low-frequency, single-operator action,
and the worst-case outcome is a duplicate Open task that an operator can
cancel, not data corruption.

## Transaction / Consistency Notes

The factory does not own a transaction. It stages the new entity and lets
the caller's `SaveChangesAsync` commit it atomically with everything else
in the unit of work:

- `StartAsync`: instance status flip + lifecycle outbox row + new task
  commit in one transaction. If the save fails, the task is rolled back
  with the rest — no orphan rows.
- `AdvanceAsync`: instance step/status mutation + outbox row + new task
  commit via `SaveWithConcurrencyAsync`, which uses the conditional UPDATE
  helper. A 409 from a stale `CurrentStepKey` rolls the task back too.
- Admin retry: instance status flip + audit + outbox row + new task in one
  `SaveChangesAsync`.

If a future change introduces an explicit `IDbContextTransaction` around
the engine call, the factory will continue to work unchanged because it
participates in whatever unit-of-work the caller has set up.

## Admin Retry Interaction Notes

Only the `retry` action invokes the factory in `HandleAdminActionAsync`.
`force_complete` and `cancel` both flip `Status` off `Active`
(`Completed` and `Cancelled` respectively), so even if they accidentally
called the factory, the eligibility gate would suppress creation. The
explicit `if (action == ActionRetry)` is defense-in-depth + intent
documentation.

Retry semantics in practice:

- **Failed → retry**: instance flips back to `Active`, factory creates a
  fresh Open task for `CurrentStepKey` (or skips if one survived the
  failure as Open/InProgress).
- **Active with stuck error → retry**: dedup query finds the existing
  task; factory is a no-op. Operator continues working the same task.
- **Pending → retry**: not a defined transition; the controller's pre-checks
  reject it before the factory is ever called.

## Known Issues / Gaps

- **No unique partial DB index** on `(WorkflowInstanceId, StepKey) WHERE
  Status IN ('Open','InProgress')`. The narrow race window (two concurrent
  admin retries on the same Failed instance) is documented as acceptable
  for this phase and would be addressed by a follow-up migration.
- **Notification fan-out is NOT wired**. Creating a task does not yet
  publish a `task.created` event for downstream notification or assignment
  workers. Out of scope for E11.2.
- **No assignee resolution**. New tasks land with no `AssignedToUserId`;
  assignment will be handled by the assignment epic.
- **No SLA seeding on the task itself**. The instance still owns SLA
  state; per-task SLA is a future epic.
- **Flow.Api still does not auto-run migrations at startup** — pre-existing
  gap from E10.3, unrelated to this change.

## Recommendation

Ship E11.2 as-is and proceed to architect re-review. Two follow-ups should
be queued:

1. Add a unique partial index to harden dedup against concurrent admin
   retries.
2. Emit a `task.created` outbox event from the factory (still no
   `SaveChangesAsync` — just stage an `OutboxMessage` alongside the task)
   to unblock downstream notification work.
