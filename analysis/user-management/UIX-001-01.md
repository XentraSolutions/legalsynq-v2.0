## Interface Boundary Clarification

**Ports listed in this document (5000, 5004) are development runtime ports only.** They describe where each Next.js process listens in the local dev environment. They are not gateway-facing ports and do not represent how production traffic is routed.

| Interface | Dev Port | Production Boundary | Notes |
|---|---|---|---|
| Tenant Portal | 5000 | Tenant hostname / subdomain | e.g. `acme.legalsynq.com` |
| Control Center | 5004 | Platform-internal hostname | e.g. `admin.legalsynq.com` |
| Identity Service | 5001 | Not public — gateway only | |
| Notifications Service | 5008 | Not public — gateway only | |

**The hostname is the real runtime boundary**, not the port. Port 5004 does not conflict with any service port (Identity = 5001, Notifications = 5008, CareConnect = separate). All inter-service traffic routes through the API gateway. Frontend-to-backend calls use relative paths or the `GATEWAY_URL` environment variable.

## 6. Invitation Flow Design

Resend invite and admin-triggered onboarding require a dedicated invitation model. This is separate from `PasswordResetTokens`.

### 6.1 UserInvitations Table

| Column | Type | Notes |
|---|---|---|
| `Id` | uuid | Primary key |
| `UserId` | uuid FK | The invited user |
| `TenantId` | uuid FK | Scoping — invite only valid for this tenant |
| `InvitedByUserId` | uuid FK | The admin who created/resent the invite |
| `TokenHash` | varchar | SHA-256 hash of the raw token sent in email |
| `Status` | enum | `PENDING`, `ACCEPTED`, `EXPIRED`, `REVOKED` |
| `PortalOrigin` | enum | `TENANT_PORTAL` or `CONTROL_CENTER` — determines invite URL base |
| `ExpiresAtUtc` | datetime | 72-hour window recommended |
| `AcceptedAtUtc` | datetime nullable | Set when user completes onboarding |
| `RevokedAtUtc` | datetime nullable | Set when admin revokes or resends |
| `CreatedAtUtc` | datetime | |

### 6.2 Invite Token Mechanics

- Token is generated as a cryptographically random 32-byte value.
- Raw token is embedded in the invite email URL only; `TokenHash` (SHA-256) is stored in the DB.
- Invite link format: `https://{portal-hostname}/accept-invite?token={rawToken}`
- On accept: validate `Status = PENDING`, `ExpiresAtUtc > NOW()`, hash matches. Mark `AcceptedAtUtc` and set `Status = ACCEPTED`.
- If the user already has a password (re-invite), the accept flow activates the account and sets `IsActive = true` rather than forcing a new password.
- If the user has no password (new user), the accept flow presents a set-password form before activating.

### 6.3 Resend Invite Behavior

Resend does **not** re-use the previous token. The flow is:

1. Admin triggers resend (A20).
2. Backend sets `Status = REVOKED` and `RevokedAtUtc = NOW()` on all current `PENDING` invites for that user.
3. Backend creates a new `UserInvitations` row with `Status = PENDING` and a fresh token and expiry.
4. New invite email is sent via Notifications service with the new token.
5. Old invite links in the user's inbox immediately become invalid.

## 7. Forgot / Reset Password Flow by Interface

The same backend endpoint handles both portals, but the token carries the `PortalOrigin` so that the reset link routes back to the correct interface with the correct branding.

### 7.1 PasswordResetTokens Table (Revised)

| Column | Type | Notes |
|---|---|---|
| `Id` | uuid | Primary key |
| `UserId` | uuid FK | |
| `TokenHash` | varchar | SHA-256 of raw token |
| `PortalOrigin` | enum | `TENANT_PORTAL` or `CONTROL_CENTER` |
| `ExpiresAtUtc` | datetime | 60-minute window |
| `UsedAtUtc` | datetime nullable | Null = not yet consumed |
| `CreatedAtUtc` | datetime | |

### 7.2 Flow — Tenant Portal

| Step | Actor | Action |
|---|---|---|
| 1 | User | Visits `https://{tenant-host}/forgot-password` |
| 2 | Frontend | Posts `{ email, portalOrigin: "TENANT_PORTAL" }` to A04 |
| 3 | Backend | Creates token record with `PortalOrigin = TENANT_PORTAL` |
| 4 | Backend | Sends email via Notifications with link: `https://{tenant-host}/reset-password?token={raw}` |
| 5 | User | Clicks link; lands on `https://{tenant-host}/reset-password` |
| 6 | Frontend | Reads `token` from query string; posts to A05 |
| 7 | Backend | Validates hash, checks expiry and `UsedAtUtc`, applies new password |
| 8 | Backend | Sets `UsedAtUtc`; redirects user to login |

**Branding:** The reset email is rendered using the tenant's branding config (logo, accent colour) pulled from the identity service tenant record.

### 7.3 Flow — Control Center

| Step | Actor | Action |
|---|---|---|
| 1 | Admin | Visits `https://{cc-host}/forgot-password` |
| 2 | Frontend | Posts `{ email, portalOrigin: "CONTROL_CENTER" }` to A04 |
| 3 | Backend | Creates token record with `PortalOrigin = CONTROL_CENTER` |
| 4 | Backend | Sends email with link: `https://{cc-host}/reset-password?token={raw}` |
| 5 | Admin | Clicks link; lands on CC reset page |
| 6 | Frontend | Posts to A05 |
| 7 | Backend | Validates and applies new password |

**Branding:** CC reset emails are platform-branded (no tenant branding). The Notifications service uses a separate "platform" email template.

### 7.4 Admin-Triggered Reset (A21) vs Self-Service Forgot (A04)

| Property | Self-Service (A04) | Admin-Triggered (A21) |
|---|---|---|
| Initiator | User | Admin |
| `PortalOrigin` | Supplied by caller | Derived from the user's account type (non-CC users → `TENANT_PORTAL`) |
| Token created by | Auth endpoint | Admin endpoint |
| Email branding | Tenant (TP) or Platform (CC) | Always Tenant (the reset is for the end user) |
| Admin sees token | Never | Never |

## 9. MVP / Phase 2 Boundary — Strict Definition

The following items are **strictly Phase 2**. No partial implementation, no hidden UI fragments, no schema enforcement in MVP code paths.

| Item | Verdict | Reason |
|---|---|---|
| Profile photo / avatar upload | Phase 2 | No object storage provisioned |
| SessionVersion enforcement in JWT | Phase 2 | Adds per-request DB read; force logout not needed for MVP |
| Force logout (A22) | Phase 2 | Depends on SessionVersion enforcement |
| MFA recommendation banner | Phase 2 | TOTP infrastructure not yet present |
| MFA enrolment | Phase 2 | Significant scope; no urgent MVP user need |
| Session / device listing | Phase 2 | Requires `UserSessions` table and tracking middleware |
| Account lock / unlock (A18, A19) | Phase 2 | Requires `IsLocked`, `FailedLoginCount`, lockout policy |
| Email change | Phase 2 | Security risk without re-verification flow |
| Password history | Phase 2 | Requires `PasswordHistory` table and lookup on every change |

| Tenant security config (CC15) | Phase 2 | Complex — tenant-level overrides affect all users |

**`AvatarUrl` and `SessionVersion` columns are added to the Users table in the MVP migration** with null/default values. They are not read, populated, or validated in any MVP code path. This avoids a second migration later.
