# E8.1 Report

> Tenant Portal Workflow UI Exposure — Shared Workflow Module + SynqLien-First Rollout.

## Scope Executed

- Created a reusable workflow UI module under `apps/web/src/components/workflow/`
  (panel + status badge + summary + start action + empty/loading/error states),
  designed for later reuse by CareConnect and SynqFund.
- Added a small browser-side data layer under `apps/web/src/lib/workflow/`
  (typed DTOs, BFF API client, service layer with selection rule) plus
  `apps/web/src/hooks/use-case-workflows.ts` (`useCaseWorkflows` /
  `useStartCaseWorkflow` / `useWorkflowDefinitions`).
- Extended the role-access model with two new actions (`workflow:view`,
  `workflow:start`) that map to existing SynqLien product roles — no new
  permission system invented.
- Integrated the panel into the SynqLien case detail page (Details tab,
  right rail) as a single new collapsible section. No other layout change.
- Added the minimum backend surface needed for the start UX to be operational:
  one new `IFlowClient` method and one new SynqLien BFF passthrough endpoint
  exposing `GET /api/liens/workflow-definitions?productKey=synqlien`. No
  changes to the hardened ownership endpoints, no new Flow features.
- Browser never talks to Flow directly — every call goes through the
  existing SynqLien `/api/lien/...` BFF proxy.

## Assumptions

- The tenant-portal SynqLien case detail surface is
  `apps/web/src/app/(platform)/lien/cases/[id]/case-detail-client.tsx`,
  whose Details tab uses `LayoutSplit` (left = sections, right = email/SMS/contacts).
  The right rail is the least-disruptive insertion point.
- The existing data-fetching pattern is plain `useState` / `useEffect` /
  `useCallback`. We did **not** introduce React Query, SWR, or any new
  state library — the new hooks follow the same vanilla pattern used by
  `casesService` so the rest of the page is unchanged.
- The browser already authenticates via the `platform_session` HttpOnly
  cookie that `apps/web/src/app/api/lien/[...path]/route.ts` rewrites to
  `Authorization: Bearer`. The new workflow module reuses this transport.
- Permissions: `workflow:view` is granted to anyone with any SynqLien role
  (matches `case:view`); `workflow:start` is granted to Sellers and Holders
  (matches `case:edit`). Tenant/platform admin override is preserved.
- Selection rule when the SynqLien API returns multiple workflow rows for
  a single case: pick the most recently `UpdatedAt`/`CreatedAt` row whose
  `Status` is not `Completed`/`Cancelled`. If none are active, fall back
  to the most recent row regardless of status. Documented at the top of
  `workflow.service.ts`.
- A workflow definition selector is required by the Flow start contract
  (`WorkflowDefinitionId` is mandatory). Rather than asking the user to
  paste a GUID, we added one tiny BFF passthrough so the modal can list
  the SynqLien-product workflow definitions. This is the only backend
  surface change in E8.1.
- The Liens BFF auth policy (`Policies.AuthenticatedUser`) is the canonical
  gate; the UI hides actions when the role-access predicate is false but
  the backend remains authoritative.

## Repository / Architecture Notes

- Tenant portal lives at `apps/web` (Next.js App Router). API calls from
  the browser go through `/api/lien/[...path]/route.ts`, which forwards
  to the gateway under `/liens/...` with the session cookie folded into
  a bearer header. The new workflow API client uses paths shaped exactly
  the same way (`/lien/api/liens/cases/{id}/workflows`).
- SynqLien backend exposes two relevant endpoint groups:
  - `Liens.Api/Endpoints/WorkflowEndpoints.cs` — `POST/GET .../cases/{id}/workflows`
    plus a `MapFlowExecutionPassthrough(...)` for atomic A1 endpoints.
  - The new `GET /api/liens/workflow-definitions` (this report) for the
    start UX.
- Flow itself is unchanged. The shared `IFlowClient` gains one read-only
  helper, `ListDefinitionsAsync(productKey, ct)`, that calls
  `GET /api/v1/workflows?productKey=...` and returns a slim DTO — Flow's
  full definition response includes stages and transitions which the UI
  doesn't need at start time.

## Shared Workflow UI Module Notes

Location: `apps/web/src/components/workflow/`

| Component                | Purpose |
| ------------------------ | ------- |
| `WorkflowPanel`          | Orchestrator. Owns load/start state, decides which sub-state to render, accepts `productSlug` + entity context so CareConnect / SynqFund can reuse it without changes. |
| `WorkflowStatusBadge`    | Compact status pill (`Active` / `Completed` / `Cancelled` / `Failed`). |
| `WorkflowSummary`        | Renders title, current step, started/last-updated dates, assignee. Collapses gracefully when fields are absent. |
| `StartWorkflowAction`    | Inline expander form. Loads available definitions, validates, submits, refreshes. Hidden when the caller cannot start. |
| `WorkflowEmptyState`     | "No workflow started" copy + start CTA when permitted. |
| `WorkflowErrorState`     | Compact non-technical error with optional Retry. |
| `WorkflowLoadingState`   | Skeleton lines, no spinner-soup. |

All copy is parameterised through props (`labels`, `productLabel`) so the
same module can render "Workflow", "Care plan", or "Funding workflow"
without code changes.

## SynqLien Integration Notes

- **Page touched:** `apps/web/src/app/(platform)/lien/cases/[id]/case-detail-client.tsx`.
- **Insertion point:** the `rightContent` of `DetailsTab` — added one new
  `CollapsibleSection title="Workflow"` ahead of the existing Email/SMS
  blocks. No other layout, header, tab, or styling change.
- **Context passed in:** the case `id` (used as `sourceEntityId`),
  `productSlug="synqlien"`, `sourceEntityType="lien_case"`, and a
  product-friendly title ("SynqLien workflow").
- **Permission source:** `useRoleAccess()` — the panel reads
  `ra.can('workflow:view')` and `ra.can('workflow:start')` and lets the

## API Wiring Notes

Browser → BFF → Liens.Api → Flow:

| UI call (browser path)                                                | Liens.Api endpoint                                             | Backed by |
| --------------------------------------------------------------------- | -------------------------------------------------------------- | --------- |
| `GET  /api/lien/api/liens/cases/{id}/workflows`                       | `GET  /api/liens/cases/{id}/workflows`                         | `IFlowClient.ListBySourceEntityAsync` |
| `POST /api/lien/api/liens/cases/{id}/workflows`                       | `POST /api/liens/cases/{id}/workflows`                         | `IFlowClient.StartWorkflowAsync` |
| `GET  /api/lien/api/liens/cases/{id}/workflows/{wfId}` (passthrough)  | `GET  /api/liens/cases/{id}/workflows/{wfId}` (passthrough)    | A1 atomic ownership endpoint |
| `GET  /api/lien/api/liens/workflow-definitions?productKey=synqlien`   | `GET  /api/liens/workflow-definitions?productKey=synqlien`     | `IFlowClient.ListDefinitionsAsync` (new) |

The browser never calls Flow directly. Auth is preserved via the existing
session cookie → bearer rewrite. After a successful start the panel
re-runs `getCaseWorkflows(caseId)` so the displayed state reflects the
freshly created instance.

## Permission / Visibility Notes

- New role-access actions: `workflow:view`, `workflow:start`.
- Mapping (matches the Lien-page conventions in `role-access.service.ts`):
  - `workflow:view` → `hasAnyLienRole` (Seller / Buyer / Holder /
    PlatformAdmin / TenantAdmin).
  - `workflow:start` → Seller or Holder (admin override applies).
- UI behaviour:
  - View-only user → panel renders summary; Start button hidden.
  - User without view → panel renders a single-line unavailable shell
    (does **not** crash the page).
  - 401/403 from the BFF → panel renders the error state with a Retry
    button; the rest of the page continues to render.
- The frontend predicate is presentation-only; the SynqLien
  `Policies.AuthenticatedUser` requirement and Flow's product-role /
  capability policies remain the authoritative gate.

## Known Issues / Gaps

- **Selection rule is heuristic.** When multiple workflows exist on one
  case the panel surfaces only the most-recent active one. A future
  phase may add a per-workflow chooser inside the panel.
- **No advance/complete in UI.** Explicit deferral per E8.1 scope.
- **No real-time refresh.** The panel re-fetches on mount and after a
  successful start; cross-session updates require a manual reload.
- **Permission introspection is role-mapping based**, not server-truth.
  401/403 from the API is handled gracefully but the UI cannot pre-show
  per-instance per-action authorization without a dedicated capabilities
  endpoint. Acceptable for E8.1 per brief.
- **Definitions list is unfiltered** beyond `productKey=synqlien`. If a
  tenant has many lien workflow definitions a future iteration should
  add scoping (e.g. by case type) inside the modal.
