## Summary

LSCC-005-02 adds automatic email retry, retry status tracking, a retry/resend distinction, a local operational audit timeline API, and improved delivery history display in the referral detail view.

The system ensures that transient SMTP failures do not permanently block provider notification delivery without any manual operator intervention.

---

## Architecture

### Retry Model

Automatic retries **update the same notification record in-place** — no new records are created. This preserves the complete attempt history on a single row.

| Attempt | Trigger | When |
|---------|---------|------|
| 1 | Initial send (`SendNewReferralNotificationAsync`) | Immediately on referral creation |
| 2 | Auto-retry (`ReferralEmailRetryWorker`) | ~5 minutes after attempt 1 failure |
| 3 | Auto-retry (`ReferralEmailRetryWorker`) | ~30 minutes after attempt 2 failure |
| 4+ | — | No further automatic retries (RetryExhausted) |

Manual resend (`POST /api/referrals/{id}/resend-email`) always creates a **new** notification record with `TriggerSource = ManualResend`. On success, it calls `ClearRetrySchedule()` on the original failed record so the retry worker does not double-send.

### Derived Status States

Persisted `Status` values: `Pending`, `Sent`, `Failed`  
Derived (computed, not persisted) states exposed in API responses:

| DerivedStatus | Condition |
|---------------|-----------|
| `Pending` | Status = Pending |
| `Sent` | Status = Sent |
| `Failed` | Status = Failed, AttemptCount < MaxAttempts, NextRetryAfterUtc = null |
| `Retrying` | Status = Failed, AttemptCount < MaxAttempts, NextRetryAfterUtc set |
| `RetryExhausted` | Status = Failed, AttemptCount >= MaxAttempts |

## Security & HIPAA Notes

- Retry logic runs entirely within the service boundary — no PII leaves the service perimeter during retry attempts
- The retry worker creates a fresh `IServiceScope` per polling cycle, preventing `DbContext` state leakage across tenants
- Audit timeline data (`GET /api/referrals/{id}/audit`) is scoped to `TenantId` and requires `AuthenticatedUser` policy — no cross-tenant data exposure
- Failure reasons are truncated at 2000 characters before persistence to prevent log injection via SMTP error messages
- The audit timeline detail field is capped at 120 characters in the server-side mapper

## Known Limitations

- `ProviderAvailabilityServiceTests` (5 tests) continue to fail — this is a pre-existing issue unrelated to LSCC-005-02
- The retry worker polls on a fixed 60 s interval; if the service is restarted mid-retry-window, the next retry will fire at the next poll after `NextRetryAfterUtc` passes (no missed-retry recovery needed beyond that)
- `GET /api/referrals/{id}/audit` returns local operational events only (status history + notifications). Cross-service audit events from the platform audit service (LSCC-005-01) are not included — the two timelines are shown separately in the UI
