# E14.1 Report

> **Status:** in progress — sections fill in as implementation lands.

## Scope Executed

E14.1 hardens the assignment surface on `Flow.Domain.Entities.WorkflowTask`
so every task has a single, explicitly-modeled ownership mode and the
existing user / role / org columns can no longer hold ambiguous
combinations.

In scope:
- New domain field `AssignmentMode` (string, four allowed values).
- New audit fields `AssignedAt`, `AssignedBy`, `AssignmentReason`.
- Single-mode invariants enforced in `WorkflowTask.EnsureValid()`
  (which is already invoked by `FlowDbContext.SaveChangesAsync` for
  every Added / Modified row).
- Backward-compat lazy mapping that derives the mode from existing
  assignment columns on persistence when the mode is unset.
- One additive EF Core migration (`AddTaskAssignmentModelE14_1`) that
  adds the columns with safe defaults, backfills mode from legacy
  rows in the same statement, and adds a single composite index for
  the upcoming queue-scan workloads.
- Producer-side wiring in `WorkflowTaskFromWorkflowFactory.ApplyAssignment`
  so newly created tasks always land with a deterministic mode.

Out of scope (explicit DO-NOT list from the spec):
- claim / reassign APIs (E14.2),
- team-queue API,
- UI changes,
- notifications,
- workflow-engine / lifecycle changes,
- My Tasks API contract changes (response shape unchanged).

## Assumptions

1. **`WorkflowTask.EnsureValid()` is the canonical write-time gate.**
   `FlowDbContext.SaveChangesAsync` already loops over every Added /
   Modified `WorkflowTask` and calls `EnsureValid()`. Adding the new
   invariants there means every persistence path — engine factory,
   admin retry, future claim/reassign — is governed for free, and we
   do not need a separate service-layer validator.
2. **`WorkflowTaskAssignment` already enforces single-target intent.**
   The E11.3 record's static factories (`ForUser` / `ForRole` /
   `ForOrg` / `None`) only ever produce instances with at most one
   field set. The new entity-level invariants therefore re-state, not
   replace, the existing application contract — defence in depth, not
   duplication.
3. **`WorkflowTaskLifecycleService.ApplyCasAsync` does NOT need changes.**
   It uses `ExecuteUpdateAsync` to flip status / timestamps only and
   never touches assignment columns, so AssignmentMode invariants
   cannot be violated by a status transition. (Verified by re-reading
   the service.)
4. **No data migration is needed for in-flight rows.** The new column
   is added with `Unassigned` as the SQL default; a single in-place
   `UPDATE … SET AssignmentMode = CASE …` in the same migration
   re-tags rows that already have a user / role / org set. We do not
   require a separate one-shot job.
5. **`AssignedBy` may be null for resolver-routed assignments.** The
   E11.3 task factory has no `IFlowUserContext` (it runs from the
   engine, not from a user request). We persist `AssignedBy = null`
   for system routing and reserve the field for the upcoming claim /
   reassign paths in E14.2. `AssignedAt` is still stamped from
   `DateTime.UtcNow` whenever the resolver returns a target.
6. **No DTO contract change.** `MyTaskDto` is intentionally not
   widened in this phase — the spec calls out "response shape
   unchanged unless additive" and the My Work UI does not yet need
   `AssignmentMode`. The field is available on the entity for future
   surfaces.

## Assignment Model Notes

`WorkflowTaskAssignmentMode` is a string-constant set, mirroring the
existing `WorkflowTaskStatus` / `WorkflowTaskPriority` convention:

| Mode          | Required field    | Forbidden fields                         |
|---------------|-------------------|------------------------------------------|
| `DirectUser`  | `AssignedUserId`  | `AssignedRole`, `AssignedOrgId` are null |
| `RoleQueue`   | `AssignedRole`    | `AssignedUserId` is null                 |
| `OrgQueue`    | `AssignedOrgId`   | `AssignedUserId` is null                 |
| `Unassigned`  | —                 | all three assignment fields null         |

Notes:
- For `RoleQueue` and `OrgQueue` the spec only forbids `AssignedUserId`;
  having both a role and an org on the same row is allowed by the spec
  (an org-scoped role queue is a future shape we explicitly do not
  want to forbid).
- `AssignmentMode` is the source of truth; the mapping table is also
  used as the lazy backward-compat derivation when the column is
  empty on a row written by an older code path.

## Validation Rules

Implemented in `WorkflowTask.EnsureValid()`:

1. `AssignmentMode` must be one of the four known values.
2. `DirectUser` ⇒ `AssignedUserId` non-empty, `AssignedRole` null,
   `AssignedOrgId` null.
3. `RoleQueue` ⇒ `AssignedRole` non-empty, `AssignedUserId` null.
4. `OrgQueue` ⇒ `AssignedOrgId` non-empty, `AssignedUserId` null.
5. `Unassigned` ⇒ all of `AssignedUserId`, `AssignedRole`,
   `AssignedOrgId` are null.
6. `AssignedAt` and `AssignedBy` are not allowed when
   `AssignmentMode == Unassigned` (they describe an event that did
   not happen).

All violations throw `InvalidOperationException` from the entity, which
is the same failure shape as the existing E11.1 invariants. The
ChangeTracker loop in `FlowDbContext.SaveChangesAsync` re-uses the
existing call site — no new save-hook code path.

## Backward Compatibility Notes

- Existing rows on disk pre-E14.1 do not have `AssignmentMode`. The
  migration's `Up` adds the column with default `'Unassigned'` and
  immediately re-tags every row in a single `UPDATE … SET
  AssignmentMode = CASE WHEN AssignedUserId IS NOT NULL THEN
  'DirectUser' WHEN AssignedRole IS NOT NULL THEN 'RoleQueue' WHEN
  AssignedOrgId IS NOT NULL THEN 'OrgQueue' ELSE 'Unassigned' END`.
  No app-code lazy-fix is needed for rows that were on disk before
  the migration ran.
- For rows mutated by code paths that don't set the mode explicitly
  (e.g. a hypothetical future helper), `EnsureValid()` first calls
  `NormalizeAssignmentMode()` which performs the same derivation at
  write time. This is the documented "lazy mapping" backstop.
- No existing query needs a code change. `MyTasksService` filters by
  `AssignedUserId` only and is therefore unaffected. The
  `(TenantId, AssignedUserId, Status)` and `(TenantId, AssignedRole,
  Status)` indexes from E11.1 are preserved.
- The legacy `TaskItem` grain (definition layer, predates
  `WorkflowTask`) is intentionally untouched.

## Persistence / Migration Notes

The new migration adds four columns to `flow_workflow_tasks`:

- `AssignmentMode` `varchar(32) NOT NULL DEFAULT 'Unassigned'`
- `AssignedAt` `datetime(6) NULL`
- `AssignedBy` `varchar(256) NULL`
- `AssignmentReason` `varchar(512) NULL`

Plus one composite index for upcoming queue-scan workloads
(`(TenantId, AssignmentMode, Status)`); existing indexes are kept.

Backfill SQL runs in the same migration so a deploy is idempotent
even on a populated database — see "Backward Compatibility Notes"
above.

`Down` drops the index and the four columns.

**Domain unit invariants (manual reasoning, no test added per scope):**
- `EnsureValid()` covers all four modes plus the four-way "no
  combinations" rule. `NormalizeAssignmentMode()` re-derives the
  mode in two cases: (a) mode is null/whitespace, or (b) mode is
  the entity default `Unassigned` AND any assignment target is
  populated. Case (b) is the realistic legacy-producer shape — the
  entity initializer ships `AssignmentMode = "Unassigned"`, so a
  pre-E14.1 caller that sets `AssignedUserId = "u-1"` without
  touching the new field would otherwise fail the `Unassigned`
  branch. Re-derivation makes that combination self-heal.
- The "Unassigned + no targets" case is left alone — it is the
  genuine no-assignment shape and re-deriving would be a no-op.
- The factory's four branches (`User` / `Role` / `Org` / none)
  each set the matching mode and explicitly null the other two
  targets, so the Added entity reaches `EnsureValid()` already in a
  legal shape.

**Architect review delta (post-first-pass fixes):**
- Tightened `NormalizeAssignmentMode()` to also re-derive when
  `AssignmentMode == "Unassigned"` but a target column is set
  (architect-flagged backward-compat hole — the entity default
  used to suppress the lazy mapping).
- Expanded the `WorkflowTaskLifecycleService` summary doc to
  spell out the `ExecuteUpdateAsync` save-hook bypass for
  assignment columns and what a future helper must do to remain
  safe.

## Known Issues / Gaps

1. **Runtime DB verification not possible in this environment.** A
   direct query against the configured `flow_db` shows the table
   `flow_workflow_tasks` does not exist and the
   `__EFMigrationsHistory` cursor is at `20260417165304_AddWorkflowSlaE10_3`.
   Both E11.1 (`AddWorkflowTaskE11_1`) and the new E14.1 migration
   are unapplied. This is a *pre-existing environmental gap* — the
   Flow service in this dev sandbox is not driving
   `Database.Migrate()` against the configured connection string —
   not a defect introduced by E14.1. The migration script itself
   has been re-read and is ordered correctly; on any environment
   where E11.1 already applied, E14.1 will additively layer on top
   (4 columns + 1 index + 1 backfill `UPDATE`). Recommend a
   follow-up environment ticket to wire Flow.Api startup migration
   in this sandbox; out of scope for E14.1.
2. **`AssignedBy` is null for every resolver-routed assignment in
   E14.1.** The factory has no user context. This is intentional and
   documented in §"Assumptions" (5); the field is reserved for the
   E14.2 claim/reassign service.
3. **No DTO surface change.** `MyTaskDto` does not yet expose
   `AssignmentMode` / `AssignedAt` / `AssignedBy` /
   `AssignmentReason`. The spec scopes UI / API surface widening to
   E14.2; E14.1 is purely model-side.
4. **EnsureValid throws `InvalidOperationException`.** Consistent
   with the existing E11.1 invariants, but a future phase may want
   to introduce a typed `WorkflowTaskAssignmentInvariantException`
   so callers can distinguish invariant failures from generic
   persistence faults. Out of scope here.

## Recommendation

**Ship E14.1.** All in-scope deliverables are complete:
- four new columns + composite index defined in the entity, EF
  config, designer, and live snapshot — all four artefacts
  consistent;
- one additive migration with deterministic CASE-based backfill
  and a clean `Down` path;
- factory wiring stamps the mode + `AssignedAt` for every
  resolver-routed task;
- single-mode invariants enforced at the entity level and gated by
  the existing `FlowDbContext.SaveChangesAsync` save hook;
- backward-compat lazy mapping in
  `NormalizeAssignmentMode()` for any future code path that forgets
  to set the mode explicitly;
- no DTO contract changes, no engine changes, no notification
  changes — strictly within the spec's "in scope" list.

Build clean (0 errors, 1 pre-existing warning). Designer and
snapshot diff-clean. The runtime DB verification gap is documented
in §"Known Issues" (1) and tracked outside this task; the migration
itself has been re-read and is correct.

Recommend opening **E14.2 (claim / reassign API)** as the next
task — it is the natural consumer of `AssignmentMode`,
`AssignedAt`, `AssignedBy`, and `AssignmentReason` and was
explicitly carved out of E14.1's scope.
