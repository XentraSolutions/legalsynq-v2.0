# E12.1 — Notification Contract & Templates

## 1. Scope

Standardise the platform's notification payload contract and introduce a
reusable notification template model so future workflow, task, SLA, and
admin notifications can be emitted consistently across services without
each feature inventing its own payload shape.

This is a **foundational, parallel-safe** phase: the existing notification
adapter path (Flow `OutboxDispatcher` → `HttpNotificationAdapter` →
notifications service `SubmitNotificationDto`) keeps working unchanged.
Everything added here is additive shared code and documentation.

## 2. Assumptions

- The existing notifications microservice (`apps/services/notifications`)
  remains the single physical recipient for outbound notifications. It
  owns provider routing, branding, suppression, contact policy, usage
  metering, and per-tenant template overrides.
- The shared contracts assembly (`shared/contracts/Contracts`) is the
  correct home for cross-service contract types. It already targets
  `net8.0`, has zero project references, and is referenced by the
  notifications service. New types live under
  `Contracts.Notifications`.
- Flow's existing `INotificationAdapter` / `NotificationMessage` shape
  (Flow.Application/Adapters/NotificationAdapter) is preserved; producers
  may keep using it or switch to `NotificationEnvelope` later.
- The notifications service template entity model (`Template`,
  `TemplateVersion`, `TemplateRenderingService`) remains the persistence
  /override layer; the new `NotificationTemplateRegistry` documents the
  canonical platform template *catalog* and seed defaults — it does not
  replace the per-tenant DB rows.
- Channel set covers `email` + `in_app` today, with `sms`, `webhook`,
  `push` reserved by the contract (no provider wiring). Reserved
  channels are accepted but no-op'd by the notifications service.
- Recipient resolution beyond direct user-id / email is **not** wired
  yet. Role / org modes are accepted by the contract and recorded in
  metadata for the future fan-out engine to consume.
- Branding / locale is intentionally out of scope here; the existing
  `BrandingResolutionService` continues to own per-tenant branding.

## 3. Canonical Notification Contract

Defined in `shared/contracts/Contracts/Notifications/NotificationEnvelope.cs`.

| Field | Required | Purpose |
|---|---|---|
| `TemplateKey` | yes | Stable key looked up in the template registry. |
| `TenantId` | optional* | Owning tenant; only `null` for platform-wide notices. |
| `ProductKey` | optional | Source product (`flow`, `tasks`, `liens`…). |
| `EntityType` / `EntityId` | optional | Business entity being referenced. |
| `Recipient` | yes | `NotificationRecipient` (see §5). |
| `Subject` / `Body` | optional | Pre-rendered overrides; preferred path is template + variables. |
| `BodyVariables` | optional | Token map fed to the renderer. |
| `Severity` | defaulted | `info` / `warning` / `critical`. |
| `Category` | defaulted | `workflow` / `sla` / `task` / `admin` / `system`. |
| `CorrelationId` | optional | Cross-service trace id (e.g. workflow instance id). |
| `OutboxId` | optional | Originating outbox id; doubles as idempotency key. |
| `ChannelHints` | optional | Producer's preferred channel subset. |
| `Metadata` | optional | Free-form audit / debugging metadata. |

Design rules baked into the contract:

- Records + primitives only — JSON-safe.
- Channel-neutral: nothing in the envelope (or template) assumes email.
- Additive evolution — every new field appended with a default so
  downstream producers compile unchanged.
- Carries enough metadata for downstream dedupe (`OutboxId`) and audit
  correlation (`CorrelationId`, `ProductKey`, `EntityType`/`EntityId`).

## 4. Template Model & Registry

`NotificationTemplate` (channel applicability list + subject/body
templates + token contract + severity/category + enabled flag) is the
reusable definition. `NotificationTokenDefinition` declares each token
the body / subject expects, and which are required.

`NotificationTemplateRegistry` is an in-process, read-mostly catalog
seeded with `PlatformDefaults()`. Seeded keys (deterministic naming —
see §7):

- `workflow.completed`
- `workflow.sla.dueSoon`, `workflow.sla.overdue`, `workflow.sla.escalated`
- `workflow.admin.retry`, `workflow.admin.forceComplete`, `workflow.admin.cancel`
- `task.assigned`, `task.completed` (registered, `Enabled = false` —
  emission lands in later phases).

The registry is the canonical platform catalog. The notifications
service's existing `Template`/`TemplateVersion` rows (per-tenant
overrides) layer on top at render time and are unchanged.

## 5. Recipient Model

`NotificationRecipient` (record) carries:

| Mode | Wired now? | Notes |
|---|---|---|
| `UserId` | yes | Direct platform user lookup. |
| `Email` | yes | Direct address fallback for transactional sends. |
| `Role`  | deferred | Accepted; resolution to user list deferred to E12.x fan-out engine. |
| `Org`   | deferred | Accepted; tenant-wide broadcast deferred. |

Helpers (`ForUser`, `ForEmail`, `ForRole`, `ForOrg`) keep producer code
short and avoid mode-mismatch bugs.
`IsImmediatelyResolvable()` documents the deferred modes.

## 6. Channel Model

`NotificationChannels` exposes constants and capability lists:

- **Supported (wired today):** `email`, `in_app`.
- **Reserved (accepted by contract, no provider yet):** `sms`,
  `webhook`, `push`.

Producers pass channel hints; the notifications service intersects
hints with the template's `Channels` and silently skips any channel
that is reserved but unwired. This keeps future channel rollout
additive.

## 7. Event-to-Template Mapping

Naming rules (apply to every key including future additions):

- Lower-case dotted segments: `{category}.{event}[.{qualifier}]`.
- First segment is the originating category — `workflow`, `task`,
  `sla`, `admin`.
- SLA transitions nested under their owning category:
  `workflow.sla.dueSoon` (not `sla.workflow.dueSoon`).
- Admin actions nested under the affected category:
  `workflow.admin.retry`.
- No tenant- or product-specific phrasing in the key; per-product
  wording belongs in tenant template overrides.

Mapping for existing producers:

| Producer event | Template key |
|---|---|
| `WorkflowComplete` (Flow `OutboxDispatcher`) | `workflow.completed` |
| `WorkflowSlaDueSoon`   | `workflow.sla.dueSoon` |
| `WorkflowSlaOverdue`   | `workflow.sla.overdue` |
| `WorkflowSlaEscalated` | `workflow.sla.escalated` |
| `AdminRetry`           | `workflow.admin.retry` |
| `AdminForceComplete`   | `workflow.admin.forceComplete` |
| `AdminCancel`          | `workflow.admin.cancel` |
| (future) Task assigned | `task.assigned` |
| (future) Task completed| `task.completed` |

Producers reference `NotificationTemplateKeys.*` constants rather than
literals so renames become compile-time breaks.

## 8. Token Rendering

`StrictTokenRenderer` reuses the same `{{token}}` Mustache grammar as
the existing `TemplateRenderingService` so producers never face two
template dialects. Strict-mode behaviour:

- **Unknown template key** → `UnknownTemplateException` (refuses to
  render rather than producing a "[missing template]" string).
- **Missing required token** → `MissingRequiredTokenException` listing
  every missing token (refuses to render rather than emitting raw
  `{{token}}` to the recipient).
- **Optional token absent** → renders as empty string deterministically.

The existing tolerant renderer in
`Notifications.Infrastructure.Services.TemplateRenderingService` is
unchanged so legacy code paths keep working; new code paths can opt
into strict rendering by going through `StrictTokenRenderer.Render`.

## 9. Adapter Compatibility / Translation Layer

`NotificationContractTranslator.ToLegacySubmit(envelope)` produces a
neutral `LegacySubmitDescriptor` that any caller can map in one line
into either of the two existing adapter shapes:

1. Flow's `NotificationMessage` (used by `HttpNotificationAdapter`).
2. The notifications service's `SubmitNotificationDto`.

Mapping rules (also documented inline on the translator):

- `TemplateKey` and `BodyVariables` forwarded verbatim.
- `ChannelHints` collapses to a single primary channel (first hint
  wins, default `in_app`).
- Recipient flattens to `userId` / `email`; role/org modes are passed
  through metadata for the future fan-out engine.
- `OutboxId` becomes the idempotency key when present.
- `Severity`, `Category`, `CorrelationId`, `ProductKey`, `EntityType`,
  `EntityId` and `ChannelHints` are surfaced via the metadata map so
  legacy DTOs (which don't have first-class fields for them) retain
  the data without DTO changes.

This translator does **not** depend on either service project — the
shared contracts assembly stays free of service references.

`SubmitNotificationDto` was not modified. If/when a future phase wants
to surface severity/category as first-class DTO fields, it can be added
additively without disturbing existing producers.

## 12. Known Issues / Gaps

- **Recipient fan-out** — `Role` and `Org` recipient modes are
  accepted by the contract but not yet resolved. Producers can begin
  emitting envelopes that target a role; they will currently round-trip
  through metadata and reach the legacy DTO as a single in-app channel
  drop until E12.x adds the resolver.
- **First-class severity/category in `SubmitNotificationDto`** — kept
  in metadata for now to stay additive. Adding nullable
  `Severity`/`Category` fields to the DTO is a low-risk follow-up.
- **In-app channel wiring** — `in_app` is in the supported list at the
  contract level but the notifications service does not yet store a
  per-user inbox feed. `NotificationContractTranslator` defaults the
  primary channel to `in_app` only when the producer supplied no hint;
  in practice today's producers all pass `email` / `system`.
- **Tenant-scoped overrides on `NotificationTemplateRegistry`** — the
  registry exposes `Upsert` for tenant overrides but the notifications
  service has not been updated to push its DB-stored overrides into the
  registry at startup. Out of scope for E12.1; tracked for the phase
  that introduces the canonical-renderer execution path.
- **Locale / branding tokens in registry defaults** — the seeded
  defaults do not declare `brand.*` tokens; the existing
  `RenderBranded` path on the legacy renderer continues to inject
  branding tokens for tenants that opt in. The strict renderer treats
  brand tokens as optional (no required declaration) so it does not
  reject envelopes from non-branded tenants.

## 13. Readiness Recommendation for E12.2

Ready to proceed. The contract surface is stable, the template registry
covers every event today's outbox dispatcher emits, and the translator
keeps the existing send path working. E12.2 can now:

- Switch one producer at a time to construct `NotificationEnvelope`
  and route through the translator without coordinated big-bang
  changes.
- Begin wiring the in-app channel against the canonical envelope.
- Add the role/org recipient resolver behind the existing recipient
  shape without touching producers again.
