# LSCC-005-01: Referral Flow Hardening & Operational Visibility

**Status:** Completed  
**Date:** 2026-04-01  
**Scope:** CareConnect microservice — referral email delivery, token revocation, resend, replay/duplicate hardening, audit visibility, UI status indicators, improved invalid-token page.

---

## 1. Motivation

LSCC-005 shipped the referral email notification flow (4-part HMAC token, SMTP send, notification DB record). Post-launch operational review identified the following gaps:

| Gap | Impact |
|-----|--------|
| No delivery attempt counter | Cannot distinguish "never tried" from "tried and failed" |
| 3-part token had no version field | Token revocation required DB-side expiry; no atomic kill-switch |
| No resend endpoint | Staff had to delete & recreate referrals on SMTP failure |
| No duplicate/replay hardening | Double-click accept or replayed HTTP request would double-process |
| Token tampering not version-aware | An attacker modifying the version field before HMAC re-check could bypass revocation |
| No audit events for security-relevant token operations | Revoked/replayed token attempts invisible to audit trail |
| Referral detail page showed no email status | Staff had no visibility into provider outreach state |
| Invalid token page was sparse | Provider confusion on expired/revoked links |

## 2. Changes Implemented

### 2.1 Domain Layer (`CareConnect.Domain`)

**`CareConnectNotification`** — email delivery tracking:
- Added `AttemptCount int` (default 0) — incremented on every `MarkSent()` / `MarkFailed()` call
- Added `LastAttemptAtUtc DateTime?` — set on every attempt
- `MarkSent()` now increments `AttemptCount`, sets `SentAtUtc` + `LastAttemptAtUtc`
- `MarkFailed(reason)` now increments `AttemptCount`, sets `FailedAtUtc`, `FailureReason`, `LastAttemptAtUtc`
- Both methods leave `Status` correct regardless of prior state (retry-safe)

**`Referral`** — token versioning:
- Added `TokenVersion int` (default 1, EF column default)
- Added `IncrementTokenVersion()` — atomically increments; no external mutation
- All previously issued tokens are implicitly revoked by the version change

**`NotificationType`** — new constant:
- `ReferralEmailResent = "ReferralEmailResent"` — distinguishes initial sends from manual resends in notification history

**Migration:** `20260401110000_ReferralHardening`
- Adds `AttemptCount` (default 0), `LastAttemptAtUtc` to `CareConnectNotifications`
- Adds `TokenVersion` (default 1) to `Referrals`

### 2.2 Application Layer (`CareConnect.Application`)

**New DTO: `ViewTokenValidationResult(Guid ReferralId, int TokenVersion)`**
- Returned by `ValidateViewToken` — replaces the old `Guid?` return type
- Callers must separately verify `result.TokenVersion == referral.TokenVersion` to detect revoked tokens

**New DTO: `ReferralNotificationResponse`**
- Surfaces `Id`, `NotificationType`, `RecipientType`, `RecipientAddress`, `Status`, `AttemptCount`, `FailureReason`, `SentAtUtc`, `FailedAtUtc`, `LastAttemptAtUtc`, `CreatedAtUtc`

**`IReferralEmailService`** (updated):
- `GenerateViewToken(Guid referralId, int tokenVersion)` — version is embedded in HMAC
- `ValidateViewToken(string token)` → `ViewTokenValidationResult?`
- `ResendNewReferralNotificationAsync(Referral, Provider, ct)` — creates `ReferralEmailResent` notification

**`ReferralEmailService`** — 4-part token format:
```
Base64url({ referralId }:{ tokenVersion }:{ expiryUnix }:{ hmacHex })
```
HMAC-SHA256 covers `"{referralId}:{tokenVersion}:{expiry}"` — version is cryptographically bound.  
Old 3-part tokens (`parts.Length != 4`) are automatically rejected — no silent fallback.

**`IReferralService`** (updated) — new methods:
- `ResendEmailAsync(tenantId, referralId, ct)` — sends new notification, audits
- `RevokeTokenAsync(tenantId, referralId, ct)` — increments `TokenVersion`, audits
- `GetNotificationsAsync(tenantId, referralId, ct)` — returns `List<ReferralNotificationResponse>`

**`ReferralService`** — hardening logic:
- `ResolveViewTokenAsync`: adds version check; emits `careconnect.referral.token.invalid` audit event on mismatch
- `AcceptByTokenAsync`: adds version check (rejects revoked tokens) + replay guard (status != New audit event + `Conflict` response on double-accept)
- New `EmitInvalidTokenAudit` helper with `EventCategory.Security`

**`INotificationRepository`** (updated):
- `GetLatestByReferralAsync(tenantId, referralId, notificationType?, ct)` 
- `GetAllByReferralAsync(tenantId, referralId, ct)`

### 2.3 Infrastructure Layer (`CareConnect.Infrastructure`)

**`NotificationRepository`** — implements the two new methods:
- `GetLatestByReferralAsync`: filters by `RelatedEntityType=Referral`, orders by `CreatedAtUtc DESC`, takes 1
- `GetAllByReferralAsync`: same filter, all records, newest-first

### 2.4 API Layer (`CareConnect.Api`)

Three new endpoints on `/api/referrals`:

| Method | Path | Auth | Capability |
|--------|------|------|------------|
| `GET`  | `/{id}/notifications` | `AuthenticatedUser` | None (referral read gate applies upstream) |
| `POST` | `/{id}/resend-email`  | `AuthenticatedUser` | `ReferralCreate` |
| `POST` | `/{id}/revoke-token`  | `AuthenticatedUser` | `ReferralCreate` |

Error responses: 404 (not found), 409 Conflict (wrong status for resend), 200 OK (success).

**`ReferralResponse`** — new fields:
- `TokenVersion int` (default 1)
- `ProviderEmailStatus string?` — "Sent", "Failed", "Pending", or null
- `ProviderEmailAttempts int`
- `ProviderEmailFailureReason string?`

`GetByIdAsync` now loads the latest notification record to populate these fields.

### 2.5 Frontend (`apps/web`)

**`types/careconnect.ts`**:
- `ReferralNotification` interface added
- `ReferralDetail` extended with `tokenVersion?`, `providerEmailStatus?`, `providerEmailAttempts?`, `providerEmailFailureReason?`
- `ReferralSummary` extended with org context fields (`referringOrganizationId?`, `receivingOrganizationId?`, etc.)
- `ReferralDetail` changed from `type =` to `interface extends ReferralSummary`

**`lib/careconnect-api.ts`** — three new client methods:
- `referrals.getNotifications(id)` → `ReferralNotification[]`
- `referrals.resendEmail(id)` → `ReferralDetail`
- `referrals.revokeToken(id)` → `ReferralDetail`

**`components/careconnect/referral-delivery-card.tsx`** (new):
- Shows email delivery status badge (Sent / Failed / Pending / Not sent)
- Shows attempt count and token version if > 1
- Shows failure reason when status is Failed
- "Resend Email" button (only shown when `status === 'New'`) — calls `resendEmail`, then `router.refresh()`
- "Revoke Link" button — confirms via dialog, calls `revokeToken`, then `router.refresh()`
- Expandable "View history" drawer — lazy-loads full notification history from `getNotifications`
- Referrer-only (rendered on the referral detail page only for `isReferrer`)

**Referral detail page** (`/careconnect/referrals/[id]`):
- `ReferralDeliveryCard` added between `ReferralDetailPanel` and `ReferralStatusActions`, shown for referrers

**Invalid token page** (`/referrals/accept/invalid`):
- Redesigned with reason-aware messaging: `missing-token`, `revoked`, `expired-or-invalid`
- Colour-coded header band (orange for revoked, red for expired/missing)
- Contextual icon (lock for revoked, warning triangle for others)
- "What to do next" three-step guidance panel
- Public accept page (`/referrals/accept/[referralId]`) also updated to detect "revoked" in the 401 response body

## 3. Security Properties

| Property | Implementation |
|----------|----------------|
| **Token revocation** | `IncrementTokenVersion()` + version check in `ValidateViewToken` caller |
| **Version tampering** | Version is part of the HMAC payload — any change breaks the signature |
| **Old 3-part tokens** | Rejected by `parts.Length != 4` guard — no silent downgrade |
| **Replay / double-accept** | `status != New` guard in `AcceptByTokenAsync` + `409 Conflict` response + security audit event |
| **Invalid token audit trail** | `careconnect.referral.token.invalid` (Security category) emitted on: malformed, expired, revoked |
| **Token revoke audit** | `careconnect.referral.token.revoked` (Security category) emitted |
| **Replay audit** | `careconnect.referral.accept.replay` (Security category) emitted |
| **Resend audit** | `careconnect.referral.email.resent` (Business category) emitted |

## 5. Migration Notes

The migration `20260401110000_ReferralHardening` is backward-safe:
- `AttemptCount` defaults to `0` — existing notification rows are unaffected
- `LastAttemptAtUtc` is nullable — existing rows default to `NULL`
- `TokenVersion` defaults to `1` — existing referral rows remain on version 1, and existing tokens (which embed no version) will be rejected because they are 3-part format; no silent re-grant

All old 3-part HMAC tokens issued before this deployment will be **automatically rejected** on first use. Any referrals with pending providers should receive a resend via the new "Resend Email" action on the referral detail page.
