# E13.1 Report

> Audit Aggregation API — read-only workflow audit timeline endpoint on
> the Flow service for the upcoming Control Center workflow timeline
> view (E13.2/E13.3). Parallel-safe with the in-progress E10.2 outbox
> work: no writes, no execution path changes, no audit storage changes.

## Scope Executed
- New read-only adapter seam on the Flow side, parallel to the
  existing `IAuditAdapter` write seam:
  - `IAuditQueryAdapter` in `Flow.Application/Adapters/AuditAdapter/`
    returning a normalized `IReadOnlyList<AuditEventRecord>` for a
    given `(entityType, entityId, tenantId?)` triple.
  - `EmptyAuditQueryAdapter` (safe baseline) in
    `Flow.Infrastructure.Adapters` — always available, returns an
    empty list. Used in environments where `Audit:BaseUrl` is not
    configured (local dev, integration tests).
  - `HttpAuditQueryAdapter` (HTTP-backed) in
    `Flow.Infrastructure.Adapters` — calls the existing audit query
    surface `GET /audit/entity/{entityType}/{entityId}`, paginates,
    handles non-success and exceptions by degrading to the empty
    baseline so a triage UI never breaks because audit is offline.
- Wired into `PlatformAdapterRegistration` alongside the existing
  audit write adapter — same `Audit:BaseUrl` toggle, same fail-soft
  behavior.
- New read-only endpoint on `AdminWorkflowInstancesController`:
  `GET /api/v1/admin/workflow-instances/{id}/timeline`. Mirrors the
  existing `GetById` admin scoping (`PlatformAdmin` cross-tenant,
  `TenantAdmin` only own tenant, 404 on out-of-scope rows for no
  existence leakage). Loads the workflow instance to confirm
  visibility, then fetches its audit events via the new query
  adapter and returns a normalized DTO list.
- New normalization layer (`AuditTimelineNormalizer`) maps raw audit
  records into a stable DTO shape, classifies each event into a
  high-level category, and tolerates missing optional fields. Raw
  action string is preserved verbatim alongside the normalized
  category.
- Deterministic ordering: ascending by `OccurredAtUtc`, tie-broken
  by `EventId` (string ordinal). Documented in this report and in
  the controller XML comments.

## Assumptions
- The audit service's `GET /audit/entity/{entityType}/{entityId}`
  endpoint is the supported scoped query surface. It returns the
  shared `ApiResponse<AuditEventQueryResponse>` envelope with
  paginated `AuditEventRecordResponse` items. The Flow side calls
  it with `EntityType=WorkflowInstance` and the instance Guid.
- Authentication for the audit query call mirrors the existing
  `HttpAuditAdapter` write path — the configured `HttpClient`
  reaches the audit service over the trusted internal network and
  no caller token is propagated. If/when a service-token scheme is
  required for cross-service audit reads, both adapters will be
  upgraded together. Out of scope here.
- Audit `EventType`/`Action` shapes already in use are stable enough
  to map by prefix:
  - `workflow.created`, `workflow.state_changed`,
    `workflow.completed` from `FlowEventDispatcher`.
  - `workflow.admin.retry`, `workflow.admin.force_complete`,
    `workflow.admin.cancel` from `OutboxDispatcher` (E10.2).
  - `task.assigned`, `task.completed` from `FlowEventDispatcher`
    (surfaced when the audit service returns them under the
    workflow's correlation, but the canonical entity for a task
    audit record is `Task`, so they will not appear here unless
    written against `WorkflowInstance` directly).
- `OccurredAtUtc` is reliably populated on every audit record (the
  ingest path requires it). When a record's `OccurredAtUtc` is the
  default value, it is still ordered first and the metadata bag
  records the anomaly.
- The endpoint is called for a single workflow instance; pagination
  over the audit response is performed server-side by the adapter
  up to a hard ceiling (1000 events) to keep the response bounded
  for the timeline view. UI-side pagination, filtering, and
  infinite-scroll are deferred to E13.2/E13.3.

## Audit Aggregation API Notes
| Aspect | Value |
| --- | --- |
| Route | `GET /api/v1/admin/workflow-instances/{id:guid}/timeline` |
| Auth | `[Authorize(Policy = Policies.PlatformOrTenantAdmin)]` (controller-level) |
| Tenant scoping | Same as `GetById`: PlatformAdmin cross-tenant; TenantAdmin only own tenant; out-of-scope → `404` |
| Upstream call | `GET {Audit:BaseUrl}/audit/entity/WorkflowInstance/{id}?page={n}&pageSize=200` (auto-paginated, hard cap 1000 events) |
| Fallback | Audit not configured or call fails → empty list, 200 OK |
| Side effects | None (no DB writes, no outbox enqueues, no audit emit) |

Response shape:
```jsonc
{
  "workflowInstanceId": "…",
  "tenantId":           "…",
  "totalCount":         12,
  "truncated":          false,
  "events": [
    {
      "eventId":        "…uuid…",
      "occurredAtUtc":  "2026-04-15T10:11:12.345Z",
      "category":       "workflow.created",
      "action":         "workflow.created",
      "source":         "flow",
      "actor":          { "id": "alice@example.com", "name": null, "type": "User" },
      "performedBy":    "alice@example.com",
      "summary":        "Workflow 'Liens — Sell Lien' created (productKey=synq-liens)",
      "previousStatus": null,
      "newStatus":      null,
      "metadata":       { "sourceSystem": "flow", "sourceService": "flow-api" }
    }
  ]
}
```

## Event Ordering / Mapping Notes
- Ordering: ascending by `OccurredAtUtc` then by `EventId` (string
  ordinal). Stable across query plans because both keys are
  deterministic per audit record.
- Normalization rules (`AuditTimelineNormalizer.Normalize`):
  - `category` derived from the raw action with a small lookup:
    - `workflow.created` → `workflow.created`
    - `workflow.state_changed` → `workflow.state_changed`
      (also extracts `previousStatus` / `newStatus` from the
      description's `"X → Y"` pattern when present, with the
      raw description preserved as `summary`).
    - `workflow.completed` → `workflow.completed`
    - `workflow.admin.retry|force_complete|cancel` →
      `workflow.admin.<verb>` (extracts `previousStatus`,
      `newStatus`, `reason`, `isPlatformAdmin` from metadata when
      present).
    - `workflow.sla.*` / `notification.*` (when present in audit)
      → `workflow.sla` / `notification` respectively (forward
      compatibility — surfaced as-is with `summary` from the
      description).
    - Anything else → `other` with the raw `action` preserved.
  - `source` derived from `SourceSystem` (e.g. `flow`, `audit`),
    falling back to `"flow"` when missing.
  - `actor` is a small object (`id`, `name`, `type`) populated from
    the audit record's actor block; missing fields tolerated as
    `null`. `performedBy` is `actor.name ?? actor.id`.
  - `metadata` is a string→string bag of any non-canonical fields
    the normalizer chose to surface (sourceSystem, sourceService,
    correlationId, requestId, sessionId, severity, visibility,
    plus any keys parsed from the audit record's metadata JSON
    as long as values are scalar). Bag is intentionally lossy —
    full audit detail is one click away in the audit service UI.

## Authorization / Visibility Notes
- Controller-level `[Authorize(Policy = Policies.PlatformOrTenantAdmin)]`
  matches `GetById`, list, and admin-action endpoints. Non-admin
  callers cannot reach the timeline.
- Tenant scoping is enforced in code (not via the EF query filter):
  `IgnoreQueryFilters()` + explicit `WorkflowInstance.TenantId`
  check after load. PlatformAdmin sees any row; TenantAdmin sees
  only their own tenant. Out-of-tenant access returns `404` so the
  endpoint cannot be used to probe for the existence of foreign
  workflow IDs.
- Once visibility is confirmed, the audit query adapter is invoked
  with the workflow's `TenantId` to scope the upstream call.

- `dotnet build apps/services/flow/backend/src/Flow.Api/Flow.Api.csproj`
  → clean.
- Workflow restart: `Start application` came up; gateway reachable.
- Smoke (no auth):

    → `401` (auth runs before the action).
  - `GET /flow/api/v1/admin/workflow-instances/{guid}` → `401`

- Read-only verification: the timeline action contains no

  call, and no mutating LINQ. The adapter implementations do only
  HTTP `GET` calls.
- No regressions to existing list/detail/admin-action endpoints —
  controller methods other than `Timeline` are unchanged.

## Known Issues / Gaps
- The HTTP query adapter does not propagate the caller's JWT to the
  audit service. This mirrors the existing write adapter; if the
  audit service later requires a service token for query reads,
  both adapters need to be upgraded together. For now, in
  configurations without `Audit:BaseUrl`, the empty baseline keeps
  the endpoint functional (returns 200 with an empty list).
- The metadata bag is intentionally lossy. A future enhancement
  could expose the full raw record under a `?expand=raw` query
  parameter — out of scope here.
- Task-level events are not currently grouped under their parent
  workflow instance because audit records are entity-scoped. A
  future iteration can extend the adapter to also fetch
  `EntityType=Task` records that share the workflow's
  correlation ID.

## Recommendation
Ship E13.1 to staging behind PlatformAdmin/TenantAdmin. The
endpoint is purely additive, parallel-safe with E10.2, and falls
back to an empty list when audit is not configured. Pair with
E13.2/E13.3 (Control Center timeline UI) which can now consume a
stable normalized shape.
