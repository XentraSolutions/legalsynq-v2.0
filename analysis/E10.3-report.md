# E10.3 Report

> **Status:** in progress — written incrementally as implementation lands.
> Final sections are filled in at the end of T004.

## Scope Executed

- Extend the existing E10.3 SLA / timer engine (which already covers
  `WorkflowInstance`) **down to `WorkflowTask`**, the user-facing
  work-item grain produced by E11.x and assigned by E14.x.
- Compute deterministic per-task `DueAt`, classify into
  OnTrack / AtRisk (a.k.a. DueSoon) / Overdue, persist the result, and
  surface it in the operator portal Work area (My Tasks, Role Queue,
  Org Queue, task detail drawer).
- Add a sibling background worker (`WorkflowTaskSlaEvaluator`) to the
  already-running `WorkflowSlaEvaluator`; both share the same code
  shape and idempotency posture.

## Assumptions

1. **Task-only SLA in this phase.** Workflow-level SLA (`WorkflowInstance.DueAt`,
   `SlaStatus`, etc.) already shipped in the prior E10.3 slice. This
   phase does NOT touch it; we only add the parallel surface for tasks.
2. **AtRisk == DueSoon.** The spec uses the term *AtRisk*; the
   pre-existing `WorkflowSlaStatus` constants already define `DueSoon`
   with identical semantics (within a configurable window before
   `DueAt`). We reuse the constant unchanged so workflow-level and
   task-level surfaces speak the same vocabulary; the UI labels it
   "At Risk" for operator clarity.
3. **DueAt is the original commitment.** Once stamped at task creation,
   `DueAt` is **not** recomputed on claim or reassign. This is the
   "customer-facing deadline" interpretation — moving a task between a
   user, a role queue, and an org queue does not change *when* the work
   is owed. Documented under *Due-Date / Calculation Notes*.
4. **No notifications, no auto-actions, no business calendar.** The
   evaluator only reads, classifies, and writes a state column. No
   outbox emission for task SLA transitions in this phase (workflow
   SLA already emits; replicating that for tasks is deferred to the
   escalation phase to keep this slice low-noise).
5. **Single replica.** Same posture as `WorkflowSlaEvaluator`. The
   evaluator is idempotent under repeated execution but does not use
   `SKIP LOCKED` to coordinate multiple replicas. Documented as a
   known gap.
6. **Priority shapes duration.** Default per-priority durations come
   from configuration (`WorkflowTaskSla:Durations:*Minutes`) and are
   read once at task creation. Changing config does not retroactively
   re-stamp existing tasks (deterministic + audit-friendly).

## SLA Data Model Notes

Additive columns on `flow_workflow_tasks` (all nullable except
`SlaStatus` which defaults to `OnTrack`):

| Column                | Type            | Notes                                                  |
| --------------------- | --------------- | ------------------------------------------------------ |
| `DueAt`               | `datetime(6)?`  | Stamped at creation; never recomputed.                 |
| `SlaStatus`           | `varchar(16)`   | OnTrack / DueSoon / Overdue (default OnTrack).         |
| `SlaBreachedAt`       | `datetime(6)?`  | First-observation breach timestamp (set once).         |
| `LastSlaEvaluatedAt`  | `datetime(6)?`  | Stamped on every evaluator visit (window rotation).    |
| `SlaPolicyKey`        | `varchar(64)?`  | Optional policy id; reserved for future SLA designer.  |

Indexes added:
- `(TenantId, Status, SlaStatus)` — admin / API "show me overdue /
  at-risk" tenant-scoped scans.
- `(TenantId, Status, DueAt)` — same scope, ordered for "what's due
  next" UI lists.
- `(Status, DueAt)` (no tenant prefix) — *evaluator hot path*. The
  `WorkflowTaskSlaEvaluator` runs cross-tenant with
  `IgnoreQueryFilters()`; a TenantId-leading index would force the
  planner to scan or merge across tenant partitions, so a non-tenant
  index keeps the range scan tight as the table grows. Added in
  response to architect review.

Migration packaged as `20260418024120_AddWorkflowTaskSlaE10_3`,
regenerated via `dotnet ef migrations add` so the `.Designer.cs` file
and `FlowDbContextModelSnapshot` stay in lockstep with the DbContext
— no migration-chain drift.

## Due-Date / Calculation Notes

Single deterministic rule, applied **once at task creation** in
`WorkflowTaskFromWorkflowFactory`:

```
DueAt = CreatedAt + DurationMinutesFor(Priority)
```

Per-priority defaults (configurable via `WorkflowTaskSla:Durations`):

| Priority | Default minutes | Default human form |
| -------- | --------------- | ------------------ |
| Urgent   |   240           |  4h                |
| High     |   720           | 12h                |
| Normal   |  1440           | 24h                |
| Low      |  4320           | 72h                |

Reassignment / claim semantics — **DueAt is invariant under all
assignment transitions** (DirectUser ↔ RoleQueue ↔ OrgQueue ↔
Unassigned). Rationale: the deadline is a commitment to the work
itself, not to whoever currently holds it; resetting on reassignment
would let queues game their SLA by passing tasks around. The evaluator
re-classifies status on its normal cadence — no special hook needed in
the claim / reassign endpoints.

Edge cases:
- **Tasks created before this phase.** `DueAt` is null; the evaluator
  skips them; `SlaStatus` stays `OnTrack` (the schema default). This
  keeps existing rows visible but unflagged.
- **Completed / Cancelled tasks.** Excluded from the evaluator batch
  (`Status NOT IN (Completed, Cancelled)`). Their persisted SLA state
  is the last-observed value at the moment they finished.
- **Already-overdue at first evaluation.** Evaluator stamps
  `SlaBreachedAt = now` (not retro-dated) on first observation; this
  phase does not back-fill from `DueAt` because we have no audit trail
  of how long the row has been overdue.

## SLA Worker / Timer Notes

`WorkflowTaskSlaEvaluator : BackgroundService` (sibling to
`WorkflowSlaEvaluator`):

- Polls every `PollingIntervalSeconds` (default **30s**).
- Per tick: opens a fresh DI scope, pulls up to `BatchSize`
  (default **200**) active tasks with `DueAt` set, ordered by oldest
  `LastSlaEvaluatedAt` first then by `DueAt` (so the workload is fair
  under continuous churn).
- Computes `(SlaStatus, SlaBreachedAt)` from
  `(now, DueAt, currentBreachedAt, DueSoonThresholdMinutes)`.
- Writes only when `SlaStatus` actually changes; always stamps
  `LastSlaEvaluatedAt = now` on every visited row.
- `IgnoreQueryFilters()` because this worker has no tenant context.
- No outbox emission this phase — task-level audit / notifications
  deferred to E10.4-equivalent.

### Environmental observation (not a defect of this slice)

The new task evaluator surfaces a pre-existing migration gap on the
local `flow_db`: the base `flow_workflow_tasks` table — created by
`20260417182606_AddWorkflowTaskE11_1` — has not been applied to the DB
this Replit environment points at. The Flow service has no
`db.Database.MigrateAsync()` on startup, and the user-controlled local
DB is several E11.x migrations behind. This is **not** introduced by
E10.3 and is identical to the situation any of E11.x / E14.x would hit
if they were running write paths against this DB. The evaluator's
catch-log-sleep-retry posture is the correct response and confirms the
worker's resilience contract works as designed. To clear the noise on
this DB, an operator runs `dotnet ef database update` against the Flow
solution (or merges this branch into an environment whose `flow_db`
already has E11.1+ applied).

## Known Issues / Gaps

- Single-replica only. Multi-replica deployment would require a
  `SKIP LOCKED` claim phase like `OutboxProcessor`.
- No outbox emission for task SLA transitions (deferred to escalation phase).
- `SlaBreachedAt` is "first-observed", not "actual breach moment".
  Acceptable while polling is at 30s; revisit if tightened.
- No back-fill of `DueAt` for pre-existing tasks. Operationally fine
  because they read as `OnTrack` and the evaluator skips them.
- `SlaPolicyKey` column reserved for the future SLA designer; not
  populated by anything in this phase.
- No business-hours / holiday calendar engine (explicit deferral).

## Recommendation

**Ship.** The task-slice SLA engine is additive, idempotent, and
mirrors the architecture of the already-shipped workflow-level
evaluator. All three layers (domain + persistence, background worker
+ DTO, frontend surfaces) build clean and the worker boots and ticks
in the running container. The only outstanding environmental friction
is a pre-existing E11.1 migration gap on this Replit's `flow_db`,
which is unrelated to this slice and resolved by running EF migrations
against the Flow solution. Defer outbox-driven escalation /
notification for task SLA transitions to the next phase, as planned.
