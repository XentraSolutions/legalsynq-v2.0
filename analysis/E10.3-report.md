# E10.3 Report — Workflow SLA / Timer Engine

> **Status: complete.** Built on top of the E10.2 outbox. No regressions to
> E10.2 lifecycle / re-drive paths.

## Scope Executed

- **Domain:** `WorkflowInstance` extended with `DueAt`, `SlaStatus`,
  `OverdueSince`, `EscalationLevel`, `LastSlaEvaluatedAt`. `FlowDefinition`
  extended with `DefaultSlaMinutes`. `WorkflowSlaStatus` constants
  (`OnTrack`, `DueSoon`, `Overdue`, `Escalated`).
- **Engine integration:** `WorkflowEngine.StartAsync` computes `DueAt`
  using the documented precedence and seeds `SlaStatus = OnTrack`.
- **Background evaluator:** `WorkflowSlaEvaluator` `BackgroundService`
  scans active instances on a poll interval, computes the new SLA state
  with a pure `Compute(...)` function, and atomically (single
  `SaveChanges`) persists state changes alongside an outbox row when a
  transition is detected.
- **Outbox events:** three new event types
  (`workflow.sla.dueSoon`, `workflow.sla.overdue`, `workflow.sla.escalated`)
  carrying a `WorkflowSlaTransitionPayload`. Dispatched via the existing
  `OutboxProcessor` → `OutboxDispatcher` pipeline.
- **Audit + notification fan-out:** `OutboxDispatcher.DispatchSlaTransitionAsync`
  always writes an audit entry; emits a notification when
  `AssignedToUserId` is known.
- **Configuration:** `WorkflowSlaOptions` (`Enabled`,
  `PollingIntervalSeconds`, `BatchSize`, `DueSoonThresholdMinutes`,
  `EscalationThresholdMinutes`) bound from `WorkflowSla` section in
  `appsettings.json`.
- **Persistence migration:** `20260417165304_AddWorkflowSlaE10_3`
  adds the five SLA columns on `flow_workflow_instances`,
  `DefaultSlaMinutes` on `flow_definitions`, plus two indexes used by
  the evaluator scan.

## Assumptions

1. **Single grain, no SLA entity.** SLA fields live directly on
   `WorkflowInstance`. Operationally simpler than a side table; one
   `UPDATE` covers an SLA transition with the existing concurrency
   tokens (`Status`, `CurrentStepKey`).
2. **Workflow-level SLA only this phase.** A new `DefaultSlaMinutes`
   column is added to `FlowDefinition`. **Step-level SLA is explicitly
   deferred** — the spec permits this provided it is documented. Adding
   `SlaMinutes` to `WorkflowStage` would require recomputing `DueAt`
   on every `Advance`, which is a non-trivial behavioural change worth
   its own phase. See *Known Issues / Gaps*.
3. **Assignment precedence (implemented):**
   1. Explicit instance `DueAt` already set on the row at start time
      (allows external callers / future overrides to short-circuit).
   2. Workflow definition `DefaultSlaMinutes` (added in this phase).
   3. None — instance is created without an SLA. The evaluator simply
      skips it (treated as `OnTrack` indefinitely).
4. **Single-replica evaluator assumed.** Like `OutboxProcessor` before
   the FOR-UPDATE-SKIP-LOCKED claim was added, the evaluator scans
   without a row-level lease. Two replicas could compute the same
   transition twice. This is acceptable because:
   * The transition emit is idempotency-gated on `(persisted SlaStatus
     != computed SlaStatus)` — a second replica re-running the scan
     within the same tick will see the already-updated row and emit
     nothing.
   * Worst case is one duplicate outbox row in a tight race window;
     downstream dedupe still uses `outboxId`.
   * If the platform later runs >1 Flow replicas, the evaluator can
     adopt the same SKIP LOCKED claim pattern with no behavioural
     change to callers.
5. **Notification recipient strategy:** assigned user
   (`WorkflowInstance.AssignedToUserId`) is the only first-class
   recipient. If unassigned, the SLA event is still durably audited
   but no notification is emitted. Documented as a known gap.
6. **Time source is `DateTime.UtcNow`.** No business calendar / holiday
   awareness this phase.
7. **Reaper / orphan recovery is unnecessary** for the evaluator —
   evaluator does not "claim" rows; each tick is a fresh scan.

## Repository / Architecture Notes

- Evaluator lives in `Flow.Infrastructure/Outbox/` next to
  `OutboxProcessor`. They share the outbox table and the same
  `BackgroundService` lifecycle pattern, but they are independent
  workers with different cadences (5 s vs 30 s by default).
- `Compute(...)` is `internal static` (pure) so it can be unit-tested
  without a `DbContext`.
- The evaluator opts out of the `WorkflowInstance` tenant query filter
  (`IgnoreQueryFilters()`) because background workers run without a
  request-scoped tenant. It compensates by populating
  `OutboxMessage.TenantId` from the owning instance before
  `SaveChangesAsync`, since the `FlowDbContext` save hook intentionally
  refuses to invent a tenant for background saves.

## SLA Model Notes

- Status values are string constants (matching the existing
  `WorkflowEngine.Status*` convention) instead of an enum, so wire-format
  compatibility with downstream consumers is trivial and the
  database column stays a short `varchar`.
- `OverdueSince` is set the **first** time a row is observed past its
  due time, not on the breach moment itself, but is rewound to `DueAt`
  on first observation — escalation duration is therefore measured
  from the actual breach, not from evaluator pickup.
- `EscalationLevel` is bounded at 1 in this phase (the spec scopes
  multi-step escalation as a follow-up). The column is `int` so future
  phases can add levels without a schema migration.

## SLA Assignment Rules

`WorkflowEngine.StartAsync` resolves `DueAt` as:

1. If the caller supplied a non-null `DueAt` on the new instance, use it.
2. Else if the resolved `FlowDefinition.DefaultSlaMinutes` is positive,
   set `DueAt = StartedAt + DefaultSlaMinutes`.
3. Else leave `DueAt = null` and `SlaStatus = OnTrack`. The evaluator
   filter `w.DueAt != null` skips these rows entirely; they cost nothing.

`SlaStatus` is always seeded to `OnTrack` and `EscalationLevel` to `0`
on start.

## Timer Evaluator Notes

- **Tick:** open scope → load batch → evaluate each row → save once.
- **Batch query:** `Active` instances where `DueAt != null` and
  `(DueAt <= now + dueSoonThreshold OR SlaStatus != OnTrack)`. Ordered
  by `LastSlaEvaluatedAt ASC, DueAt ASC` for fair rotation.
- **Idempotency gate:** stage an outbox row only when
  `computed.SlaStatus != persisted.SlaStatus` OR
  `computed.EscalationLevel != persisted.EscalationLevel`. Otherwise
  the visit is silent (only `LastSlaEvaluatedAt` advances).
- **Atomicity:** state mutation + outbox insert share one
  `SaveChangesAsync`, so an SLA transition is either fully visible
  (state + event) or not at all.
- **Failure handling:** entire `TickAsync` is wrapped in try/catch
  identical to `OutboxProcessor`; the next poll retries from scratch
  with no leased state to clean up.

## Outbox / Event Notes

- Three new event types added to `OutboxEventTypes`:
  `workflow.sla.dueSoon`, `workflow.sla.overdue`, `workflow.sla.escalated`.
- Single payload type `WorkflowSlaTransitionPayload` carries
  instance id, product key, current step, due time, previous & new
  status, escalation level, overdue duration in seconds, and the
  assigned user (nullable). All transitions use the same payload shape
  to keep the dispatcher simple.
- Dispatch flows through the unchanged `OutboxProcessor` claim → execute
  → mark-processed loop. No new infra around at-least-once or backoff.

## Audit / Notification Notes

- `OutboxDispatcher.DispatchSlaTransitionAsync` always emits an audit
  entry (`workflow.sla.dueSoon` / `overdue` / `escalated`) so the
  forensics trail is complete even when no one is assigned yet.
- Notification fan-out is conditional on `AssignedToUserId` being
  non-null. The notification template key matches the event name so
  channel-side templating is one-to-one.
- Both adapters resolve from the same DI scope as the rest of the
  dispatcher path; HTTP timeouts and retry/decorator policies apply
  identically to E10.2 inline events.

## Configuration Notes

`appsettings.json` (Flow.Api):

```json
"WorkflowSla": {
  "Enabled": true,
  "PollingIntervalSeconds": 30,
  "BatchSize": 200,
  "DueSoonThresholdMinutes": 60,
  "EscalationThresholdMinutes": 60
}
```

- `Enabled = false` cleanly suspends the worker (the loop continues but
  every tick is a no-op).
- `PollingIntervalSeconds` is clamped to ≥1.
- `BatchSize` is clamped to ≥1.
- Threshold minutes are clamped to ≥0.
- Bound via `IOptionsMonitor`, so values can be hot-reloaded without
  restart for ops experimentation.

## Known Issues / Gaps

1. **Step-level SLA deferred.** Only workflow-level deadlines are
   honoured. A future phase should extend `WorkflowStage` with
   `SlaMinutes` and recompute `DueAt` on `Advance`.
2. **Single escalation level.** `EscalationLevel` tops out at 1. Multi-
   tier escalation (manager → director → on-call) is left as future
   work; the column type already supports it.
3. **Single-replica assumption.** No `SKIP LOCKED` claim on the
   evaluator. Safe today (Flow runs as a single instance) but should be
   revisited if the deployment topology changes.
4. **No business calendar / holidays.** SLA math is wall-clock UTC.
5. **Notifications skipped when unassigned.** Audit still fires, so
   nothing is lost — but operators rely on a separate dashboard view to
   surface unassigned-and-overdue work.
6. **Whole-batch rollback on concurrency conflict.** `WorkflowInstance`
   carries optimistic concurrency tokens (`Status`, `CurrentStepKey`).
   If another writer mutates one row in the evaluator's batch between
   load and save, the entire `SaveChangesAsync` rolls back as a
   `DbUpdateConcurrencyException` — the tick logs an error and the next
   poll retries from scratch. Eventual correctness holds (no missed
   transitions across ticks) but transient noise is possible under
   heavy concurrent workflow churn. A future hardening pass should
   per-row retry or drop conflicted rows and persist the rest.
7. **Migration auto-apply gap (pre-existing).** Flow.Api does **not**
   call `Database.Migrate()` on startup, unlike Comms.Api. Operators
   must apply Flow migrations manually (`dotnet ef database update`)
   between deploys. This is unchanged by E10.3 but became visible
   while smoke-testing because both E10.2 and E10.3 migrations were
   pending against the dev DB. Recommend a follow-up to align Flow.Api
   with the Comms.Api startup-migration pattern.

## Recommendation

Ship E10.3 as-is. The follow-up backlog in priority order:

1. **Flow.Api startup migration alignment** — small, low-risk, removes
   a recurring footgun for the next phase that adds a Flow migration.
2. **Step-level SLA** — natural next phase; the evaluator scan and
   dispatcher contract do not need to change, only the assignment
   path on `Advance` and a new `SlaMinutes` column on `WorkflowStage`.
3. **Multi-tier escalation** — extend `EscalationThresholdMinutes` to
   a list and have `Compute` walk it.
4. **Unassigned-and-overdue Control Center surface** — closes the only
   real notification gap.
