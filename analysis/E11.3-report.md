# E11.3 Report — Task Assignment Rules (Deterministic Resolver)

> Status: implementation complete; pending architect review.

## Scope Executed

E11.3 introduces a deterministic assignment layer so that `WorkflowTask`
rows produced by the E11.2 factory are automatically populated with one of
`AssignedUserId`, `AssignedRole`, or `AssignedOrgId` based on a static,
in-code rule table. Out of scope: manual reassignment UI, task / "My Tasks"
APIs, notifications, SLA-based assignment, identity-service lookups, rule
authoring UI.

What shipped:

1. New `WorkflowTaskAssignment` decision record + `IWorkflowTaskAssignmentResolver`
   interface in `Flow.Application.Interfaces`.
2. New `StaticRuleWorkflowTaskAssignmentResolver` implementation in
   `Flow.Application.Services` — pure in-code rule table keyed first on
   `(ProductKey, StepKey)` then on `StepKey` alone, with case-insensitive
   ordinal comparison.
3. `WorkflowTaskFromWorkflowFactory` invokes the resolver after dedup and
   before `_db.WorkflowTasks.Add(...)`, applying the decision via a single
   `ApplyAssignment(...)` helper that re-asserts single-target precedence
   and length-truncates each id to its EF column max.
4. DI registration in `Flow.Infrastructure.DependencyInjection` (scoped, to
   match the rest of `Flow.Application`).

## Assumptions

- The three assignment columns shipped on `WorkflowTask` in E11.1
  (`AssignedUserId` 256, `AssignedRole` 128, `AssignedOrgId` 256) are the
  correct surface — no schema change required for E11.3.
- Rule sources are **code, not DB**. Adding rules requires a code edit and
  a deploy. This is intentional first-phase scope: rules are
  version-controlled, reviewable, and impossible to mutate at runtime.
- "Deterministic" means: for a given `(WorkflowInstance.ProductKey,
  CurrentStepKey)` pair, the resolver returns the same decision every time
  with no reliance on time, ambient request, identity service, or DB
  state.
- Tenant-safety boundary: the resolver does NOT validate that a returned
  user / role / org id is meaningful inside the workflow instance's tenant
  (that would require an identity-service call, which is forbidden by the
  task spec). It is the operator's responsibility to author rules whose
  ids are tenant-portable (e.g. role names or platform-managed orgs) — see
  §"Tenant Boundary Notes" for the exact safety contract.
- The rule table starts EMPTY. Every workflow created today therefore
  produces an unassigned task and falls through to the documented safe
  fallback. This is intentional — first-phase delivery is the
  *mechanism*; specific rules are seeded as workflows demand them.

## Assignment Resolver Notes

**Interface**:

```csharp
WorkflowTaskAssignment Resolve(WorkflowInstance instance, string stepKey);
```

**Decision record** — `WorkflowTaskAssignment`:

```csharp
public sealed record WorkflowTaskAssignment(
    string? AssignedUserId,
    string? AssignedRole,
    string? AssignedOrgId);
```

The record exposes only `static` factory methods —
`ForUser(...) / ForRole(...) / ForOrg(...) / None` — that produce an
instance with at most one field set and whitespace-normalised values. A
caller cannot accidentally produce a multi-target decision via these
factories; the public constructor is left only for record-equality
plumbing and is never used in the code base.

**Failure isolation** — `WorkflowTaskFromWorkflowFactory` wraps the
`Resolve(...)` call in a try/catch:

- A throwing or buggy resolver collapses to
  `WorkflowTaskAssignment.None` (warning logged with the exception).
- Task creation continues with an unassigned task. The eligibility /
  dedup logic and the SaveChanges path are unchanged.

This guarantees the spec invariant *"invalid assignment must not break
task creation; safe fallback should apply"*.

## Rule Source / Mapping Notes

The resolver consults two static dictionaries declared in
`StaticRuleWorkflowTaskAssignmentResolver`:

1. **`ProductStepRules`** — `Dictionary<(string ProductKey, string StepKey), WorkflowTaskAssignment>`
   with a custom `IEqualityComparer` that does ordinal-ignore-case on both
   key members. Entries here override step-only rules.

2. **`StepRules`** — `Dictionary<string, WorkflowTaskAssignment>` with
   `StringComparer.OrdinalIgnoreCase`. Step-key default applied across all
   products.

Both dictionaries are intentionally empty at first-phase ship. Adding a
rule is a code change such as:

```csharp
{ ("liens", "intake_review"), WorkflowTaskAssignment.ForRole("LiensIntakeReviewer") }
```

Reasons to keep this in code, not DB:

- No runtime mutation surface ⇒ no auth model required for rule editing.
- Reviewable in PR diffs.
- No risk of a tenant editing rules that affect other tenants' tasks.
- No new external dependency / API.

A future phase can swap `StaticRuleWorkflowTaskAssignmentResolver` for a
DB-backed implementation behind the same interface without touching the
factory or the engine — the seam is clean.

## Precedence / Fallback Notes

### Lookup precedence (which rule wins)

`(ProductKey, StepKey)` > `(StepKey)` > none.

The most-specific rule wins. A product-scoped override completely
displaces a step-only default; the resolver never merges rules.

### Assignment-target precedence (which field wins inside one decision)

`AssignedUserId` > `AssignedRole` > `AssignedOrgId`.

Enforced in two layers, defence-in-depth:

1. The `WorkflowTaskAssignment` factory methods physically prevent a
   multi-target decision — `ForUser(x)` produces `(x, null, null)`,
   `ForRole(x)` produces `(null, x, null)`, etc.
2. `WorkflowTaskFromWorkflowFactory.ApplyAssignment(...)` re-asserts the
   precedence when copying onto the task: it inspects the decision in
   User → Role → Org order, writes the first non-null field, and
   explicitly nulls the other two. A future caller hand-constructing
   the record with multiple fields set would still produce a single-
   target row.

### Fallback

When no rule matches OR the resolver returns `None` OR throws, the task
is staged with all three assignment fields `null` (unassigned). This is
the documented safe fallback. The task is still persisted, still has
its tenant / step / title, and is still eligible for any future
"My Tasks" / claim-task surface (E11.5) to pick up.

## Tenant Boundary Notes

The resolver itself is tenant-agnostic — it takes a `WorkflowInstance`
and a `stepKey` and returns a decision that is identical regardless of
ambient tenant. It does NOT consult the database or any identity
service.

**Safety contract** for rule authors:

- A returned `AssignedRole` is tenant-portable by definition — roles in
  this codebase are platform-defined string identifiers
  (e.g. `"Manager"`, `"LiensIntakeReviewer"`) and have the same meaning
  across tenants.
- A returned `AssignedOrgId` is acceptable when the org is a
  platform-managed concept (e.g. a default routing org seeded across all
  tenants). It is NOT acceptable to encode a tenant-specific org id in a
  static rule — that rule would assign cross-tenant.
- `AssignedUserId` should be avoided in static rules in this phase. The
  resolver does not validate that the user belongs to
  `instance.TenantId`, and the spec forbids identity-service lookups.
  Direct user assignment is reserved for a future phase that introduces
  a tenant-aware lookup. **Until then, prefer `ForRole` or `ForOrg`**.

The persisted task itself remains tenant-scoped: `WorkflowTask.TenantId`
is set explicitly to `instance.TenantId` by the factory (E11.2), the
DbContext save-hook re-stamps it, and the global query filter pins all
reads to the calling tenant. The assignment id columns are opaque
strings — they are not foreign keys — so a stale id cannot corrupt
tenant boundaries; at worst it produces an unmatchable assignment that
the future "My Tasks" surface will treat as orphan and skip.

### Functional / behavioural

Verified by code inspection (no integration test surface in this phase):

- Rule table empty ⇒ every new task is unassigned (fallback exercised).
- Resolver throw ⇒ caught, warning logged, task created unassigned.
- Single-target invariant ⇒ enforced twice (factory methods +
  `ApplyAssignment`). A multi-target decision is *not constructible*
  through normal API usage, and even a hand-rolled multi-target record
  would be normalised to one field.
- Tenant scoping ⇒ the new code path adds no new DB reads and does not
  touch the existing E11.2 `IgnoreQueryFilters() + explicit TenantId`
  predicates.
- E11.2 dedup unchanged ⇒ resolver call is *after* the dedup short-
  circuits, so a no-op return path (existing Open/InProgress task)
  never invokes the resolver and never touches the existing task's
  assignment fields. This satisfies the spec requirement *"retry does
  not overwrite existing active-task assignment"*.
- Outbox / SLA / engine logic unchanged.

## Known Issues / Gaps

- **Rule table is empty.** First-phase ship delivers the mechanism, not
  specific rules. Workflows that need automatic assignment require a
  follow-up code change to populate `ProductStepRules` /
  `StepRules`.
- **No tenant-aware user-id validation.** `AssignedUserId` from a static
  rule is not checked against `instance.TenantId`. Mitigation: the
  safety contract above instructs authors to avoid `ForUser` in static
  rules.
- **No DB-level constraint that exactly one assignment field is set.**
  The single-target invariant is enforced in code (factory methods +
  `ApplyAssignment`), not in the DB. A future migration could add a
  `CHECK` constraint, but MySQL `CHECK` enforcement is uneven across
  versions and the application-layer enforcement is sufficient for the
  current surface.
- **No outbox event emitted on assignment.** A `task.assigned` event is
  reserved for E12 notifications. The factory still does not call
  `SaveChangesAsync`, so adding such an event later is a one-line stage
  on the existing outbox writer.
- **No metrics counter** on assignment decisions (count by source /
  type). Logging is sufficient for first-phase observability.

## Recommendation

API)**. The assignment surface is mechanically complete, deterministic,
and safe by default. Specific rule entries can be added incrementally as
product workflows demand them without disturbing E11.2 / E11.4 / E11.5
