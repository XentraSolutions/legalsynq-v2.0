# E9.2 Report

> Control Center Workflow Detail Drawer — Read-Only Workflow Inspection.

## Scope Executed

- New read-only Flow admin endpoint
  `GET /api/v1/admin/workflow-instances/{id}` returning a richer
  `AdminWorkflowInstanceDetail` (list-row fields + current step name +
  last engine error + assignee + correlation key).
- New CC type `WorkflowInstanceDetail` and server-API method
  `controlCenterServerApi.workflows.getById(id)`.
- New `WorkflowDetailDrawer` client component with sections
  **Summary / Source Context / Lifecycle / Diagnostics**, plus a
  conservative deep link to the source product record where the route
  shape is known (SynqLien case, SynqFund application, CareConnect
  referral).
- The `/workflows` page now reads `?selected=<id>` and server-renders
  the drawer with the hydrated detail. Closing the drawer clears
  `selected` while preserving every other filter.
- Each table row gained a small **Open →** action; the row is also
  highlighted while the drawer is open.

## Assumptions

- The CC `/workflows` surface continues to be **PlatformAdmin-only** at
  the page boundary (`requirePlatformAdmin()`). The Flow detail
  endpoint additionally allows **TenantAdmin** (server-side scoped to
  their own tenant) so the same endpoint can power a tenant-scoped CC
  surface in a future phase without re-authoring auth.
- The detail endpoint is the only authoritative data source for the
  drawer; the page does not fall back to "open from row data" because
  the drawer needs the engine-only fields (current step name, last
  error message) the list mapping does not return.
- Tenant portal deep links are intentionally **relative paths opened in
  a new tab**. The tenant portal lives on a different origin in
  production and CC has no env var for that origin today, so the link
  is best-effort: it works when an operator is already on the tenant
  portal domain or proxies to it. This is acceptable for triage.
- Only product/source-entity combinations with a verified route shape
  (`SYNQ_LIENS`/`lien_case`, `SYNQ_FUND`/`fund_application` or
  `application`, `SYNQ_CARECONNECT`/`referral`) get a deep link.
  Everything else still shows the source-entity type and id but no
  link, and degrades gracefully.

## Repository / Architecture Notes

- **Backend**: `apps/services/flow/backend/src/Flow.Api/Controllers/V1/AdminWorkflowInstancesController.cs`
  is the single source for both the E9.1 list and the new E9.2
  `GetById`. It re-uses the same `_tenantProvider` + `Roles.PlatformAdmin`
  auth pattern and keeps the same defence-in-depth shape:
  `IgnoreQueryFilters()` on every query and explicit `TenantId == w.TenantId`
  predicates on the `FlowDefinitions`, `WorkflowStages`, and
  `ProductWorkflowMappings` joins.
- **Mapping selection** uses the same deterministic ordering established
  by the E9.1 architect feedback (Active first, then most recent
  `UpdatedAt ?? CreatedAt`).
- **CC API client** (`apps/control-center/src/lib/control-center-api.ts`)
  reuses the existing `apiClient.get` + `CACHE_TAGS.workflows` (10 s) +
  `isNotFound` plumbing pattern that the tenants/users sections already
  use, so the new `getById` looks identical to its peers.
- **Drawer component** (`apps/control-center/src/components/workflows/workflow-detail-drawer.tsx`)
  is a client component (`'use client'`) for ESC handling and `useRouter`
  navigation, but it does no fetching of its own — the page passes the
  server-fetched detail in as a prop. This keeps the
  list/table/drawer/fetcher concerns separated as the brief requires
  and leaves the drawer reusable for future exception/stuck-workflow
  views.
- **URL-state pattern**: the drawer is opened/closed strictly via the
  `?selected=<id>` query param. This is identical to the page-state
  pattern already used for paging (`?page=N`) and filters
  (`?productKey=`, `?status=`, `?tenantId=`, `?search=`), so the drawer
  composes naturally with bookmarking and back/forward navigation. No
  client state library was introduced.

## Workflow Detail Drawer Notes

- Lives at `apps/control-center/src/components/workflows/workflow-detail-drawer.tsx`.
- Slide-in from the right, full-height, 28–32 rem wide on ≥sm.
  Backdrop click and ESC both close it via `router.replace(closeHref)`
  with `scroll: false` so the underlying list does not jump.
- Sections rendered:
  - **Summary** — workflow title, instance id (de-emphasised), status
    badge, product, current step (name + key), tenant.
  - **Source Context** — source entity type, id (mono-spaced), optional
    deep link, correlation key.
  - **Lifecycle** — started, updated, completed, assigned user.
  - **Diagnostics** — only renders if `lastErrorMessage` is present
    *or* the workflow ended in `Failed` / `Cancelled`. Error messages
    are wrapped in a `<pre>` for legibility.
- States handled:
  - `selectedId` set + detail returned → full content.
  - `selectedId` set + detail `null` (404 / forbidden / scoped-out) →
    compact amber "no longer available / no visibility" banner.
  - `selectedId` set + page-side fetch threw → red error banner inside
    the drawer; the list is unaffected.
  - `selectedId` not set → drawer renders nothing.
- Footer carries an explicit "Read-only inspection. Execution actions
  remain on the product surface." reminder so a future maintainer does
  not turn it into an execution surface.

## API / BFF Notes

- Browser → `controlCenterServerApi` (Server Component) →
  `apiClient.get('/flow/api/v1/admin/workflow-instances/{id}', 10, ['cc:workflows'])`
  → API gateway (port 5010) → Flow service. No browser-to-Flow direct
  coupling was introduced.
- 404 from Flow is mapped to `null` by `getById` (uses the existing
  `isNotFound` helper), so the drawer renders the "not available" state
  without throwing into the page boundary.
- Cache tag `cc:workflows` is shared with the list endpoint, so any
  future `revalidateTag('cc:workflows')` will refresh both surfaces in
  a single sweep.

## Permission / Visibility Notes

- **Page guard**: `requirePlatformAdmin()` (unchanged from E9.1).
- **Endpoint guard**: `[Authorize(Policy = Policies.PlatformOrTenantAdmin)]`
  — `PlatformAdmin` may inspect any instance; `TenantAdmin` may inspect
  only instances in their own tenant. A `TenantAdmin` who tries to
  fetch an out-of-tenant id receives **404** (not 403) so the existence
  of the row is not leaked.
- The endpoint `IgnoreQueryFilters()` then re-applies tenant scoping in
  code — same pattern as the list — which means a stray cross-tenant
  row still cannot leak through joined definition / mapping / stage
  metadata.
- An ordinary product user can never reach the drawer because both the
  page guard and the endpoint policy reject them.

## Validation Results

- **Backend build**: succeeded (`dotnet build` for the Flow solution
  ran clean during the workflow restart, 0 errors).
- **CC build / fast-refresh**: rebuilt cleanly (Fast Refresh `done in
  1229ms`, no TypeScript errors after the rebuild).
- **Workflow restart**: `Start application` came back up; gateway
  `/flow/health` → 200.
- **Endpoint smoke**:
  - `GET /flow/api/v1/admin/workflow-instances/{guid}` without a token
    → **401** (auth gate works).
  - `GET /workflows?selected=<id>` (CC, unauthenticated) → **307**
    (login redirect, page guard intact).
- **No regression** on the E9.1 list page: it renders unchanged, and
  the new `Open →` column is the only visible delta.

## Known Issues / Gaps

- Tenant portal deep links are relative-path / new-tab only because no
  cross-app origin is configured for CC. When CC eventually gets a
  `NEXT_PUBLIC_TENANT_PORTAL_ORIGIN` (or equivalent), upgrade
  `deriveSourceRecordHref` to emit absolute URLs.
- Source-entity → route mapping covers SynqLien / SynqFund /
  CareConnect today. SynqBill, SynqRx, SynqPayout entries fall through
  to "no link" — they need a route once the corresponding product
  surfaces stabilise.
- The Diagnostics panel currently surfaces `lastErrorMessage` and the
  terminal status only; it does not yet show step history, retry
  counts, or correlated audit events. Those belong to the future E9.3
  exception/stuck-workflow surface.
- `currentStepName` resolution is best-effort — if the workflow
  definition was renamed or its stage row was deleted, the drawer
  falls back to `currentStepKey`.

## Recommendation

E9.2 is ready for **E9.3 (exception / stuck-workflow view)**.

The detail drawer is now the natural mounting point for "Why is this
workflow stuck?" follow-up: the same `?selected=<id>` URL contract,
the same `controlCenterServerApi.workflows.getById` data path, and the
same drawer component can host an additional Diagnostics-tab worth of
content (last events, retry log, age) without re-architecting any of
the list/page/drawer separation we just established.

No execution surface, no broad redesign, no direct browser-to-Flow
coupling, and the E9.1 list remains stable and read-only.
