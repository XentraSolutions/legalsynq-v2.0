## 2. Changes Delivered

### 2.1 Global Toast System

**New files:**
- `apps/web/src/lib/toast-context.tsx`
- `apps/web/src/components/toast-container.tsx`

The toast system is a standard React context implementation. `ToastProvider` manages a list of `Toast` objects (`id`, `message`, `variant`). Each toast schedules its own `setTimeout` dismissal at 4 seconds; early dismissal via the × button clears the timer to avoid a state update on an already-removed item. Timer refs are stored in a `useRef<Map>` so they survive re-renders without triggering effects.

`useToast()` — call site hook; exposes only `show(message, variant?)`.
`useToastState()` — internal hook used by `ToastContainer`; exposes `toasts` and `dismiss`.

`ToastContainer` renders fixed bottom-right with three visual variants:

| Variant | Color | Icon |
|---|---|---|
| `success` | Green | `ri-checkbox-circle-fill` |
| `error` | Red | `ri-error-warning-fill` |
| `info` | Blue | `ri-information-fill` |

Slide-in animation added to `tailwind.config.ts` as a custom keyframe (`translateY(8px) → 0, opacity 0 → 1, 0.2 s ease-out`).

**Platform layout** (`apps/web/src/app/(platform)/layout.tsx`) was updated to wrap `AppShell` with `<ToastProvider>` and render `<ToastContainer />` as a sibling inside the provider. This makes `useToast()` available to every CareConnect client component without any additional wiring per page.

### 2.2 Referral Timeline

**New file:** `apps/web/src/components/careconnect/referral-timeline.tsx`

Client component. On mount, calls `careConnectApi.referrals.getHistory(referralId)`. Renders an `<ol>` timeline sorted newest-first. Each item shows:

- Colour-coded dot (keyed to `newStatus` via a `STATUS_DOT` map)
- `oldStatus → newStatus` transition label; creation entries (no `oldStatus`) show only `newStatus`
- Formatted UTC timestamp (locale-aware, 12-hour)
- Optional `notes` field (pre-wrap)

States handled: loading skeleton (2 placeholder rows), empty (`"No activity yet."`), and error (`"Could not load activity history."`).

**API method added** to `careconnect-api.ts`:
```
referrals.getHistory(id: string) → ApiResponse<ReferralHistoryItem[]>
  GET /careconnect/api/referrals/{id}/history
```

**Type added** to `types/careconnect.ts`:
```typescript
interface ReferralHistoryItem {
  id, referralId, oldStatus, newStatus,
  changedByUserId?, changedAtUtc, notes?
}
```

**Wired into** `apps/web/src/app/(platform)/careconnect/referrals/[id]/page.tsx` as an "Activity" card below the action buttons.

### 2.3 Appointment Actions (Confirm / NoShow / Reschedule)

**New file:** `apps/web/src/components/careconnect/appointment-actions.tsx`

Client component. Determines which actions to show based on `isReceiver`, `isReferrer`, and `appointment.status`:

| Role | Status | Button shown |
|---|---|---|
| Receiver | `Scheduled` | **Confirm Appointment** |
| Receiver | `Confirmed` | **Mark No-Show** |
| Receiver or Referrer | `Scheduled` or `Confirmed` | **Reschedule** |

Terminal statuses (`Cancelled`, `Completed`, `NoShow`) render nothing.

**Confirm / NoShow** call `PUT /api/appointments/{id}` with `{ status }` via `careConnectApi.appointments.update(id, body)`. On success, a toast is shown and the page is refreshed via `router.refresh()`. On 401 the user is redirected to login; on 403 an inline error is shown. On other API errors the error message is surfaced inline and a failure toast is shown.

**Reschedule modal:**

1. Opens as a fixed full-screen overlay with a `backdrop-blur-sm` backdrop.
2. On open, fetches provider availability for the next 14 days via `careConnectApi.providers.getAvailability(providerId, { from, to })`.
3. Renders an `AvailabilityList` with `selectedSlotId` prop for single-select highlighting.
4. Shows a confirmation banner with the new slot's formatted datetime once selected.
5. Optional notes textarea.
6. Calls `POST /api/appointments/{id}/reschedule` with `{ newAppointmentSlotId, notes? }` via `careConnectApi.appointments.reschedule(id, body)`.
7. On 409 (slot just taken): clears selection, refetches slots, shows inline slot error — user can pick again without closing the modal.

**New API methods** added to `careconnect-api.ts`:
```
appointments.update(id, { status, notes? })
  PUT /careconnect/api/appointments/{id}

appointments.reschedule(id, { newAppointmentSlotId, notes? })
  POST /careconnect/api/appointments/{id}/reschedule
```

**Wired into** `apps/web/src/app/(platform)/careconnect/appointments/[id]/page.tsx` above `AppointmentCancelButton`.

### 2.5 Dashboard Stat Counts

`apps/web/src/app/(platform)/careconnect/dashboard/page.tsx` was rewritten to replace the `—` stat cells with real counts.

**Referrer view** — five parallel `Promise.allSettled` calls:

| Fetch | Used for |
|---|---|
| `referrals.search({ pageSize: 5 })` | Active referral list (filtered client-side to non-terminal) |
| `referrals.search({ status: 'Completed', pageSize: 1 })` | `totalCount` → "Completed" stat |
| `referrals.search({ status: 'Declined', pageSize: 1 })` | `totalCount` → "Declined" stat |
| `appointments.search({ status: 'Scheduled', pageSize: 20 })` | Merged for upcoming set |
| `appointments.search({ status: 'Confirmed', pageSize: 20 })` | Merged for upcoming set |

Scheduled + Confirmed appointments are deduplicated by ID into a `Map`, then filtered to `isWithinDays(scheduledAtUtc, 7)` and sorted by time. "Upcoming (7 days)" stat = merged count before slicing.

**Receiver view** — five parallel calls:

| Fetch | Used for |
|---|---|
| `referrals.search({ status: 'New', pageSize: 5 })` | Pending referral list |
| `referrals.search({ status: 'Accepted', pageSize: 1 })` | `totalCount` → "Accepted" stat |
| `referrals.search({ status: 'Completed', pageSize: 1 })` | `totalCount` → "Completed" stat |
| `appointments.search({ status: 'Scheduled', pageSize: 50 })` | Merged for today's schedule |
| `appointments.search({ status: 'Confirmed', pageSize: 50 })` | Merged for today's schedule |

"Today's Appts" = appointments where `isToday(scheduledAtUtc)` is true.

All fetches use `Promise.allSettled` — any individual failure silently falls back to `0` / empty list rather than breaking the entire page render.

## 5. Outstanding Items

The following are not part of this task and remain open for future work:

- **Cross-tenant provider discovery mock gap** — `ProviderAvailabilityServiceTests` (5 tests) mock `IProviderRepository` with `GetByIdAsync` but the service now calls `GetByIdCrossAsync` after the LSCC-003 cross-tenant fix. The tests need their mock setup updated to return data from `GetByIdCrossAsync`. No production code change required.
- **Referral reschedule** — there is no `POST /api/referrals/{id}/reschedule` concept; rescheduling is appointment-level. If the product requirement evolves to allow changing the requested provider on an existing referral, a new workflow rule and endpoint would be needed.
- **Appointment status history endpoint** — `GET /api/appointments/{id}/history` does not exist on the backend. If an appointment activity timeline is ever needed (mirroring `ReferralTimeline`), the endpoint and domain event recording would need to be added first.
