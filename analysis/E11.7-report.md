# E11.7 Report

**Status:** implemented, build clean, smoke-verified.

## Scope Executed

Bound `WorkflowTask` completion to `WorkflowInstance` progression. The
existing `POST /api/v1/workflow-tasks/{id}/complete` endpoint now does
both, atomically, while the workflow engine remains the sole execution
authority. Lifecycle, query, outbox, SLA, admin, and task-creation
paths are untouched.

## Assumptions

* **Single owning instance.** A `WorkflowTask` always points at one
  `WorkflowInstance` (`WorkflowInstanceId` is non-nullable on
  `WorkflowTask`). The factory in E11.2 enforces this; the orchestrator
  trusts the column.
* **Step correspondence.** A task's `StepKey` is mirrored from
  `WorkflowInstance.CurrentStepKey` at creation time (E11.2). Tasks for
  superseded steps therefore exist if the workflow has since advanced
  via another path; those are exactly the "stale" cases this phase
  protects against.
* **No fan-out resolution at this phase.** Definitions where the
  current step has multiple active outbound transitions still require
  an explicit `toStepKey` (engine raises `ambiguous_transition` ⇒ 409).
  Adding a "complete with explicit next step" surface is deferred — the
  task-completion verb is not authoritative for routing.
* **InProgress is the only valid source.** The lifecycle service
  already disallows `Open → Completed`; the orchestrator pre-validates
  the same so that "complete a never-started task" never opens a
  transaction.
* **Engine is the only mutator.** This phase does not write to
  `WorkflowInstance` outside `IWorkflowEngine.AdvanceAsync`.

## Repository / Architecture Notes

Two services compose:

* `IWorkflowTaskLifecycleService.CompleteTaskAsync` (E11.4) — atomic
  CAS `InProgress → Completed` via `ExecuteUpdateAsync`. Honours
  tenant query filter; raises `WorkflowTaskConcurrencyException` on
  CAS loss; raises `InvalidStateTransitionException` on wrong source.
* `IWorkflowEngine.AdvanceAsync` (MERGE-P5 / E10.x) — checks workflow
  is `Active`, that `CurrentStepKey == expectedCurrentStepKey`, and
  that exactly one outbound transition exists (or that the named
  `toStepKey` matches one). Stages the next-step task via
  `WorkflowTaskFromWorkflowFactory` (E11.2) and enqueues the
  `workflow.advance` / `workflow.complete` outbox event in the same
  `SaveChangesAsync` write.

Both services share the per-request `IFlowDbContext`, so wrapping them
in one `BeginTransactionAsync` makes both writes — the lifecycle CAS
update and the engine's stage / instance / outbox writes — commit or
roll back together.

The orchestration runs inside the database's
`IExecutionStrategy.ExecuteAsync`, which is required when manual
transactions are combined with EF's retry-on-transient-failure
strategy. Without it the second writer can break the retry contract.

## Binding Logic Notes

The governing rule is intentionally narrow:

> A task transition `InProgress → Completed` advances the owning
> workflow **iff** the task's `StepKey` equals the workflow's current
> step at the moment the engine acquires the row. Any other shape —
> stale step, terminal workflow, ambiguous outbound — fails the entire
> request and rolls back the task transition.

**Step / state matrix:**

| Task source state | Workflow state    | Step match? | Outcome                                          |
|-------------------|-------------------|-------------|--------------------------------------------------|
| `InProgress`      | `Active`          | yes, unique outbound | `200` — task `Completed`, workflow advanced (or `Completed` if next step is terminal) |
| `InProgress`      | `Active`          | no (`task.StepKey != instance.CurrentStepKey`) | `409` `stale_current_step` — nothing committed |
| `InProgress`      | `Completed` / `Cancelled` / not started | n/a | `409` `instance_not_active` / `instance_not_started` — nothing committed |
| `InProgress`      | `Active`, multiple outbound | yes | `409` `ambiguous_transition` — nothing committed (deferred fan-out support) |
| `Open`            | any               | n/a         | `422` `Open → Completed` rejected pre-transaction |
| `Completed` / `Cancelled` | any       | n/a         | `422` source-state rejection pre-transaction        |
| missing / cross-tenant | any          | n/a         | `404` (uniform with cross-tenant id probing)        |

Nothing else: no heuristics, no implicit fan-out resolution, no silent
no-ops.

## API / Lifecycle Changes

* **Reused endpoint, broader behaviour.** `POST /api/v1/workflow-tasks/{id}/complete`
  now delegates to `IWorkflowTaskCompletionService.CompleteAndProgressAsync`
  rather than `IWorkflowTaskLifecycleService.CompleteTaskAsync` directly.
  External request shape (path, no body) is unchanged.
* **Response shape upgraded** from `WorkflowTaskTransitionResult` to
  `WorkflowTaskCompletionResult` — **strictly additive**. The new
  record carries every field of the old one (`TaskId`,
  `PreviousStatus`, `NewStatus`, `TransitionedAtUtc`) and adds
  `WorkflowInstanceId`, `FromStepKey`, `ToStepKey`, `WorkflowStatus`,
  `WorkflowAdvanced`. Existing clients deserialise unchanged; UI
  callers can now refresh workflow state in one round-trip.
* `start` and `cancel` endpoints are unchanged — those still go
  straight to the lifecycle service. Cancellation is intentionally NOT
  bound to workflow progression in this phase; the spec scopes the
  binding to completion only.
* **Middleware:** `InvalidWorkflowTransitionException ⇒ 409` added so
  the new endpoint surfaces engine refusals consistently. Existing
  workflow-instance controllers were already catching this exception
  locally; the middleware mapping now covers any code path that lets
  it bubble.

## Concurrency / Consistency Notes

* **Atomicity.** Both writes — task CAS and engine save — run inside
  one `IDbContextTransaction`, itself inside the database's execution
  strategy. The persisted outcome is one of:
  * task `Completed` AND workflow advanced (engine Save committed); or
  * neither change visible (any throw rolls back `tx`).

  We never persist "task Completed but workflow on the previous step".
* **Duplicate-progression protection.** Two layered guards:
  1. The lifecycle's `WHERE Id=@id AND Status='InProgress'`
     `ExecuteUpdateAsync` returns 1 for the winner and 0 for every
     other concurrent caller (loser ⇒ `WorkflowTaskConcurrencyException`
     ⇒ 409). The engine advance only runs after CAS success.
  2. The engine's own `expectedCurrentStepKey` check + its
     `SaveWithConcurrencyAsync` guard against two completions of two
     different tasks both racing to advance the same instance — the
     loser sees `stale_current_step` and the transaction rolls back
     (its task stays `InProgress`).
* **Stale-task semantics.** A task whose step no longer matches the
  workflow's current step fails the whole call with
  `InvalidWorkflowTransitionException("stale_current_step")` ⇒ 409.
  The task is **not** silently completed and left dangling — by
  rolling back the lifecycle CAS, an operator inspecting the task
  later still sees it `InProgress`, which makes the diagnosis obvious
  and prevents drift between task state and workflow state.
* **Cross-tenant safety.** The pre-load uses `WorkflowTask` through
  the global query filter; an attempt to complete another tenant's
  task surfaces as `404` (identical to "missing"). The engine's own
  load is also tenant-scoped.
* **Pre-validation outside the transaction.** Existence /
  wrong-source-state / blank-StepKey rejections fire before
  `BeginTransactionAsync`, so the most common error responses do not
  hold a connection.
* **What we do not do (intentional):** no `RowVersion` column added
  (the existing CAS + engine concurrency already cover the races); no
  workflow-instance row lock taken outside the engine's own write; no
  retries — the execution strategy retries transient connection
  failures, application-level conflicts surface to the caller.

**Functional matrix walkthrough** (by code path, not against a live
populated DB — the dev workflow does not seed authenticated tenants):

* Correct active task → CAS succeeds, engine advances, outbox enqueued
  in the same `SaveChangesAsync`, transaction commits.
* Stale task (workflow already advanced via another route) → engine
  raises `stale_current_step`, transaction rolls back, task remains
  `InProgress`.
* Workflow already terminal → engine raises `instance_not_active`,
  transaction rolls back.
* Duplicate concurrent complete on the same task → only one caller's
  CAS returns 1; the other caller sees
  `WorkflowTaskConcurrencyException` ⇒ 409.
* Two different tasks racing the same workflow → one caller wins both
  the CAS and the engine save; the other wins its CAS but the engine
  raises `stale_current_step` and the transaction rolls back —
  including its task CAS.
* `Open` task → `InvalidStateTransitionException` ⇒ 422 fires before
  the transaction opens.

**Non-interference**

* Outbox: engine still calls `_outbox.Enqueue` inside the same
  `SaveChangesAsync` — unchanged path.
* SLA: no SLA-evaluator code touched. The `SlaStatus`/`OverdueSince`
  fields are read-only here.
* Admin actions: `WorkflowInstancesController` /
  `ProductWorkflowExecutionController` retained their local
  `catch (InvalidWorkflowTransitionException)` blocks; behaviour
  identical to pre-E11.7. The new middleware mapping is reached only
  when the exception is uncaught (i.e. only the new endpoint).
* Task creation (E11.2 factory) still runs inside the engine's
  `AdvanceAsync` after the next stage is selected — unchanged.

## Known Issues / Gaps

* **Definitions with multiple outbound transitions** at the current
  step are explicitly unsupported through this verb (engine raises
  `ambiguous_transition`). Operators relying on UI fan-out must use
  the `WorkflowInstance` `advance` endpoint with a `toStepKey`
  payload. A future "complete-with-route" variant could be added.
* **Single-task progression.** If a step legitimately requires N
  tasks to complete before advancement (a "join"), the current binding
  advances on the first one. The current schema does not encode this
  requirement, so it is out of scope; should be handled by the
  workflow-definition layer when introduced.
* **No notification fan-out** is wired here. The engine's outbox event
  is emitted as before; consumers (notifications service, control
  centre) pick it up via the existing pipeline.
* **No new automated tests** were added in this phase. The downstream
  services are individually covered, and the orchestrator is a thin
  composition; integration coverage of the end-to-end "complete →
  advance" path is recommended as a follow-up.

## Recommendation

**Ready for the My Work UI to consume.**

The endpoint contract is stable and the failure modes are well-typed
(404 / 409 / 422). The UI can:

* call `POST /workflow-tasks/{id}/complete` and use the response
  `WorkflowStatus` / `ToStepKey` to refresh the workflow card without
  a follow-up GET;
* on `409` with `Code == "stale_current_step"`, prompt the user to
  refresh My Tasks (a peer task already advanced the workflow);
* on `409` with `Code == "instance_not_active"`, mark the task row
  inert;
* on `409` `WorkflowTaskConcurrencyException`, treat as "another tab
  already finished this" and re-fetch the row.

The "join multiple tasks", "fan-out routing from UI", and "automated
test coverage of the binding" items are reasonable follow-ups but do
not block UI work.
