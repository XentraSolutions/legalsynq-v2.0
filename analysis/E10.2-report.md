# E10.2 Report

> Workflow Outbox + Async Processing — durable side effects and reliable
> re-drive. Workflow state changes and downstream side effects are
> recorded in the same transaction; a background worker processes them
> with retry / backoff / dead-letter. Flow remains the single execution
> authority.

## Scope Executed

- New `OutboxMessage` aggregate persisted in `flow_db.flow_outbox_messages`
  (Pomelo/MySQL 8). EF entity config + Pomelo migration
  `20260417152404_AddOutboxMessagesE10_2`. Migration applied to RDS;
  table verified live.
- `IOutboxWriter` seam (Application) + `OutboxWriter` (Infrastructure)
  used by `WorkflowEngine` (start/advance/complete/cancel/fail) and
  `AdminWorkflowInstancesController` (force-complete / cancel / retry)
  to enqueue side effects in the **same** EF SaveChanges as the state
  mutation.
- `OutboxProcessor` `BackgroundService`:
  - Per-tick reaper for stale `Processing` rows past lease window
    (default 5 min) → re-queue if attempts left, dead-letter otherwise.
  - Atomic claim via raw ADO.NET `SELECT … FOR UPDATE SKIP LOCKED` +
    in-tx `UPDATE Status='Processing', AttemptCount+=1`. Bypasses EF's
    `MySqlRetryingExecutionStrategy` "user-initiated transaction" guard.
  - Per-row dispatch in fresh inner DI scope.
  - Outcome: success → `Succeeded` / `ProcessedAt`; failure →
    exponential backoff (`base * mult^(attempt-1)`) up to `MaxAttempts`,
    then `DeadLettered`. Outcome saves use `CancellationToken.None` so
    a shutdown mid-failure cannot strand the row.
- `OutboxDispatcher` fan-out by `EventType`:
  - `workflow.start | advance | complete | cancel | fail` → audit row
    via `IAuditAdapter` (always); `WorkflowComplete` additionally fires
    a `system` notification via `INotificationAdapter`.
  - `workflow.admin.forceComplete | cancel` → audit only.
  - `workflow.admin.retry` → audit + idempotent re-drive nudge that
    re-reads the workflow instance with `IgnoreQueryFilters` /
    `AsNoTracking` and emits a structured log line (engine NOT
    re-invoked — Flow stays the sole execution authority).
- `OutboxOptions` bound to `Outbox:` config section. Defaults:
  Enabled=true, PollingIntervalSeconds=5, BatchSize=25, MaxAttempts=5,
  BaseBackoffSeconds=30, BackoffMultiplier=2.0, ProcessingLeaseSeconds=300.
- `FlowDbContext.SaveChangesAsync` made background-safe: tenant
  resolution is best-effort (catches `InvalidOperationException` from
  `ClaimsTenantProvider` when there is no HTTP claim); the existing
  hard assertion still fires for `Added` entities lacking a `TenantId`.

## Assumptions

- MySQL 8.0+ — `FOR UPDATE SKIP LOCKED` is required for safe multi-worker
  claim. Verified against RDS Aurora MySQL 8.
- The audit adapter (`IAuditAdapter`) and notification adapter
  (`INotificationAdapter`) carry their own resilience policies (timeouts,
  retries, fallbacks) — the dispatcher does not add a second layer.
- Downstream collectors dedupe on `outboxId` carried in event metadata
  (`metadata["outboxId"]`); we deliver at-least-once.
- Re-drive (`workflow.admin.retry`) intentionally never re-invokes
  `WorkflowEngine.Advance`. Re-execution is the user/product command
  path's responsibility; the handler is a verification + audit nudge.
- Background scope updates to existing `OutboxMessage` rows can rely on
  the `TenantId` already stored on the row, so the per-save tenant
  assertion is allowed to be bypassed in those scopes.

## Repository / Architecture Notes

- Single solution: `apps/services/flow/backend`. Three projects:
  `Flow.Domain`, `Flow.Application`, `Flow.Infrastructure`, `Flow.Api`.
- `WorkflowEngine` (Application) is the single execution authority.
- `MySqlRetryingExecutionStrategy` is wired globally on `FlowDbContext`,
  which means user-initiated EF transactions are forbidden. This drove
  the decision to use raw ADO.NET for the claim transaction.
- `Flow.Infrastructure.csproj` was missing
  `Microsoft.Extensions.Hosting.Abstractions` + `…Options` for
  `BackgroundService` + `IOptionsMonitor`; both added.

## Outbox Model Notes

- Schema: `Id char(36) PK`, `EventType varchar(64) NOT NULL`,
  `WorkflowInstanceId char(36) NULL`, `PayloadJson longtext NOT NULL`,
  `Status varchar(16) NOT NULL DEFAULT 'Pending'`,
  `AttemptCount int NOT NULL DEFAULT 0`,
  `NextAttemptAt datetime(6) NOT NULL`,
  `LastError varchar(2048) NULL`, `ProcessedAt datetime(6) NULL`,
  `TenantId varchar(128) NOT NULL`, plus `CreatedAt/By`, `UpdatedAt/By`.
- `Status` stored as string (enum `OutboxStatus`:
  `Pending | Processing | Succeeded | DeadLettered`).
- Indices: `(Status, NextAttemptAt)` for the polling query; `TenantId`
  for ops queries; `WorkflowInstanceId` for tracing.
- No tenant query filter on `OutboxMessage` — the worker scope has no
  tenant claim and queries every tenant's rows by design. Authz is
  asserted at write-time (writer requires a tenant context) and at
  dispatch-time (each handler enforces tenant boundaries via the
  payload's `TenantId`).

## Mutation Integration Notes

- `WorkflowEngine.StartAsync / AdvanceAsync / CompleteAsync /
  CancelAsync / FailAsync` each call `_outbox.EnqueueAsync(...)` after
  applying the state mutation, **before** `SaveChangesAsync`. The
  enqueue + state row commit in the same transaction.
- `AdminWorkflowInstancesController.ForceComplete / Cancel / Retry`
  similarly enqueue an admin-action outbox row in the mutation
  transaction.
- The unused `_audit` field on `AdminWorkflowInstancesController` is no
  longer wired into any code path now that audit fan-out is the
  dispatcher's job — left in place this iteration to keep the diff
  scoped; flagged for removal in next sweep.

## Worker / Processor Notes

- `OutboxProcessor : BackgroundService`. Loop: reap → claim → dispatch.
- Each tick opens its own DI scope; per-row dispatch opens an inner
  scope so a poisonous DbContext from one row cannot bleed into the
  next.
- Reaper runs as two single-statement updates (no user transaction
  required) — one to dead-letter rows past attempts, one to re-queue
  the rest. Recovers any row stranded in `Processing` past lease.
- Claim uses raw ADO.NET `BeginTransaction` against
  `db.Database.GetDbConnection()` — small in-tx window (one SELECT, one
  UPDATE), commits before any dispatch.

## Retry / Backoff / Dead-Letter Notes

- Backoff schedule with defaults (base=30s, mult=2): ~30s, 60s, 120s,
  240s. After 5th attempt fails, row → `DeadLettered`.
- `AttemptCount` is bumped at claim time, so the first attempt records
  count=1; the dead-letter check is `AttemptCount >= MaxAttempts`.
- Outcome saves use `CancellationToken.None` to bound the strand
  window during shutdown; the reaper still backstops anything missed.
- `LastError` truncated to 2048 chars to fit the column.

## Audit / Notification / Re-drive Notes

- All dispatched audit/notification calls carry `outboxId` in metadata
  for downstream dedupe.
- Lifecycle audit rows describe the transition (e.g. `from → to`) and
  preserve `productKey`, `fromStatus`, `toStatus`, `reason`,
  `performedBy`.
- Notifications are emitted only on `WorkflowComplete` to keep noise
  proportional; other lifecycle events are audit-only.
- Admin re-drive checks the workflow instance's current status with
  `IgnoreQueryFilters().AsNoTracking()` (no tenant claim available) and
  logs success or "state moved on" — never throws on mismatch, never
  re-executes engine logic.

## Known Issues / Gaps

- The `AdminWorkflowInstancesController._audit` field is unused and
  should be removed in a follow-up cleanup pass.
- Per-handler idempotency inside fan-out is collector-driven (downstream
  dedupes on `outboxId`); a partial fan-out failure can re-emit
  earlier-step side effects on retry.
- Lease-based reaper is conservative (5 min default). Operators should
  tune `Outbox:ProcessingLeaseSeconds` if downstream HTTP timeouts
  exceed it.
- Outbox writer is a value-only seam — it does not own the
  enclosing transaction; correctness depends on callers SaveChanges-ing
  the workflow mutation and the outbox row together (current call sites
  do).

## Recommendation

Ship E10.2. The pipeline is durable end-to-end (mutation + enqueue
co-committed; reaper + retry/backoff/dead-letter; idempotent re-drive),
verified live against RDS, and architect-flagged severities are
addressed (stale-Processing reaper, non-cancelable outcome save,
background-safe SaveChanges).
