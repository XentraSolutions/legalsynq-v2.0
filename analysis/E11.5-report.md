# E11.5 — My Tasks API

**Status:** implemented, build clean, smoke-tested.
**Scope:** read-only "My Tasks" query surface over the new `WorkflowTask`
grain (E11.1). Lifecycle mutations remain the exclusive responsibility
of E11.4's `IWorkflowTaskLifecycleService` and are not re-exposed in
this engagement.

---

## Endpoint

| Verb | Route                                              | Auth                          | Result                              |
|------|----------------------------------------------------|-------------------------------|-------------------------------------|
| GET  | `/api/v1/tasks/me`                                 | `Policies.AuthenticatedUser`  | `200 PagedResponse<MyTaskDto>`      |
| POST | `/api/v1/workflow-tasks/{id}/start`                | `Policies.AuthenticatedUser`  | `200 WorkflowTaskTransitionResult`  |
| POST | `/api/v1/workflow-tasks/{id}/complete`             | `Policies.AuthenticatedUser`  | `200 WorkflowTaskTransitionResult`  |
| POST | `/api/v1/workflow-tasks/{id}/cancel`               | `Policies.AuthenticatedUser`  | `200 WorkflowTaskTransitionResult`  |

**Lifecycle error mapping** (already wired by `ExceptionHandlingMiddleware`):
`NotFoundException ⇒ 404`, `InvalidStateTransitionException ⇒ 422`
(semantic — current state forbids the transition),
`WorkflowTaskConcurrencyException ⇒ 409` (CAS race; retryable),
`ValidationException ⇒ 400`.

The lifecycle controller (`WorkflowTasksController`) is intentionally
separate from `MyTasksController`: lifecycle is a per-task mutation
surface, "My Tasks" is a per-user query surface. Lifecycle calls are
NOT restricted to tasks visible in `/me` — any tenant-visible task may
be transitioned by a permitted caller.

### Query parameters

| Name       | Type             | Default | Notes                                                                                              |
|------------|------------------|---------|----------------------------------------------------------------------------------------------------|
| `status`   | `string[]?`      | `null`  | Repeatable. Case-insensitive. Accepts `Open`, `InProgress`, `Completed`, `Cancelled`. Unknown ⇒ 400. |
| `page`     | `int`            | `1`     | Values < 1 normalised to 1.                                                                        |
| `pageSize` | `int`            | `25`    | Clamped to `MyTasksDefaults.MaxPageSize` (100).                                                    |

### Response shape

`PagedResponse<MyTaskDto>` (existing pagination envelope). `MyTaskDto`
is intentionally narrow: lifecycle timestamps + the four assignment-relevant
fields (`Status`, `Priority`, `Title`, `Description`) + `WorkflowInstanceId`
and a small workflow-context pair (`WorkflowName`, `ProductKey`).
Engine fields (`MetadataJson`, `CorrelationKey`, role / org assignments,
audit columns beyond the four lifecycle timestamps) are deliberately
**not** projected so the contract does not couple the UI to engine
internals.

## Identity / Tenant model

* **User** — pulled from `IFlowUserContext.UserId` (which adapts
  `ICurrentRequestContext` populated by the JWT pipeline). There is
  **no** `userId` query / route parameter; user impersonation is
  impossible by API shape, not just policy.
* **Tenant** — enforced automatically by the `WorkflowTask` global
  query filter on `FlowDbContext`. A tenant `A` token cannot return
  rows from tenant `B` even if `AssignedUserId` happens to match — the
  tenant predicate is applied first.
* **"My"** — `WorkflowTask.AssignedUserId == currentUserId`.
  Role- and org-based assignments are **not** included in this phase
  because resolving the caller's effective roles / orgs requires an
  identity-service round-trip that is out of scope. Unassigned tasks
  are excluded.

## Query / Filtering Notes

* **Single SQL.** Count + page slice run through the same predicate;
  the join to `WorkflowInstance` → `FlowDefinition` is expressed via
  navigation properties so EF emits ONE statement with LEFT JOINs —
  no N+1.
* **Index match.** `(TenantId, AssignedUserId, Status)` composite
  shipped in E11.1 is the perfect cover: tenant + user predicates use
  the leftmost prefix; the optional `status` filter narrows the scan.
* **Ordering** (deterministic, paging-stable):
  1. **Active first** — `Open` / `InProgress` precede `Completed` /
     `Cancelled`. Encoded as `IsActive DESC` (boolean ordering).
  2. **Recency** — `UpdatedAt DESC`, with `CreatedAt` fallback because
     `UpdatedAt` is null on never-modified rows.
  3. **Stable tiebreaker** — `Id`, so `Skip/Take` is reproducible
     across requests with identical timestamps.
* **Status normalisation.** Multi-value, blank-stripped, dedup'd
  (`OrdinalIgnoreCase`), unknown values rejected with `400` (mapped via
  the existing `ValidationException` middleware) and re-canonicalised
  to the on-disk casing before the SQL `IN` comparison.
* **Page-shape safety.** `page < 1 ⇒ 1`, `pageSize < 1 ⇒ default`,
  `pageSize > 100 ⇒ 100`. Both rules duplicated in DTO and service so
  direct service-layer callers stay safe.

## Out of Scope / Follow-ups

* **Role / org "my tasks"** — needs identity-service resolution of the
  caller's effective roles + orgs. Defer.
* **Cursor pagination** — current envelope is `Skip/Take`. Acceptable
  for the expected page volume; can be migrated without breaking the
  envelope by adding an optional `cursor` parameter.
* **`WorkflowTaskStatus` enum vs string** — model binding is string-based
  to match the on-disk representation. Switching to an enum would
  require a custom binder; not justified yet.
