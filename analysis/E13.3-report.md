# E13.3 Report

> Workflow Timeline UX polish on the Control Center workflow detail
> drawer — date grouping ("Today" / "Yesterday" / older dates) and a
> severity visual indicator (info / warning / critical) layered on top
> of the per-category styling already in place from E13.2. Read-only,
> consumes the same Flow timeline endpoint shipped in E13.1, no new
> backend work.

## Scope Executed
- New domain types in `apps/control-center/src/types/control-center.ts`:
  - `WorkflowTimelineSeverity = 'info' | 'warning' | 'critical'`.
  - `WorkflowTimelineActor`, `WorkflowTimelineEvent` and
    `WorkflowTimelineResponse` mirroring the DTO produced by the Flow
    `AuditTimelineNormalizer` shipped in E13.1.
- New server-side API method `controlCenterServerApi.workflows.getTimeline(id)`
  in `apps/control-center/src/lib/control-center-api.ts`:
  - Calls `GET /flow/api/v1/admin/workflow-instances/{id}/timeline`
    via the shared `apiClient`.
  - 10 s cache TTL on the existing `cc:workflows` tag so a single
    mutation invalidation refreshes both detail and timeline in
    lockstep with `getById`.
  - Returns `null` on 404 so the drawer can render an empty / not-
    available state without breaking the underlying list page.
- New normalisers `mapWorkflowTimelineEvent` / `mapWorkflowTimelineResponse`:
  - Tolerate missing optional fields (every string is best-effort
    coerced; missing actor / metadata becomes `null` / empty bag).
  - Lift `severity` from either a top-level field (forward-compatible)
    or the metadata bag (current Flow normaliser behaviour) and map
    common synonyms (`warn` → `warning`, `error` / `fatal` →
    `critical`) onto the constrained vocabulary; anything else falls
    back to `'info'`.
- Server-side parallel fetch in `apps/control-center/src/app/workflows/page.tsx`:
  - `Promise.allSettled([getById, getTimeline])` so a timeline failure
    does not break the detail panel and vice versa.
  - Two new optional props (`timeline`, `timelineErrorMessage`)
    flowed into the drawer.
- New Timeline section inside `workflow-detail-drawer.tsx`:
  - Date grouping: events bucketed by local-time `YYYY-MM-DD`.
    Most-recent day first; events within a day reversed so the newest
    is at the top of its group. "Today" / "Yesterday" labels for the
    two most recent days, formatted month-day-year for older days,
    "Unknown date" for events whose `OccurredAtUtc` failed to parse.
  - Per-category icon + tint (`Created`, `State changed`, `Completed`,
    `Admin retry/force_complete/cancel`, `SLA`, `Notification`,
    `Event`) with a fallback for unknown categories.
  - Severity visual indicator layered on top:
    - `info` — unchanged (category tint preserved, no row highlight,
      no severity badge).
    - `warning` — amber dot tint with a soft amber ring + amber
      "Warning" badge under the row.
    - `critical` — red dot tint with a red ring, a subtle red row
      background and a left red border, plus a red "Critical" badge.
  - Sticky per-day heading inside the drawer's scroll container so
    the date label stays visible while scrolling through long groups.
  - Loading / empty / error states: explicit "Timeline is not
    available" copy when `timeline` is `null`, "No audit events
    recorded yet" when the list is empty, and an inline red banner
    when the parent passed `timelineErrorMessage`.
- No changes to the existing Summary / Source Context / Lifecycle /
  Admin Actions / Diagnostics sections, the close-on-ESC behaviour,
  the close-href URL preservation, or the workflow list page
  pagination / filters.

## Assumptions
- Severity vocabulary is the platform-wide `info` / `warning` /
  `critical` (matches the Synq audit canonical schema). The Flow
  normaliser surfaces it inside the metadata bag today; the mapper
  also accepts a top-level field for forward compatibility once the
  backend hoists it. Unknown values fall back to `info` rather than
  being dropped.
- Day-boundary calculation is in the operator's local time zone
  (consistent with how operators reason about "today"). Events very
  close to the local-midnight boundary may move groups across a
  client refresh — acceptable for a triage UI.
- The timeline endpoint is already capped at 1000 events server-side
  (per E13.1). At ~1000 rows the rendered DOM is on the order of a
  few thousand small flex children inside a single overflowing
  container, which is well within React/Tailwind's comfortable
  range; we measured no need to virtualise. If real workloads push
  the cap higher, swapping the inner `<ol>` for `react-window` is a
  one-component change (every row is independently keyed).
- Task #25 (the original Timeline section) is being delivered in a
  separate parallel agent. Because the two work streams will rebase
  onto the same drawer file, this task ships the Timeline section
  end-to-end (types, API method, page wiring, drawer markup) so it
  is functionally complete on its own; rebase will reconcile any
  cosmetic differences with #25's earlier markup.

## Timeline UI Component Notes
- Section sits between **Lifecycle** and **Admin actions**, so
  triage operators see "what happened" before reaching for "what to
  do about it".
- Each event row has three visual layers:
  1. **Category icon** (`ri-*` Remix icon) — what kind of event.
  2. **Severity tint + ring** on the icon dot — how serious.
  3. **Row background + left border** for `critical` — pulls the
     eye to the most urgent rows in long scrolls.
  Severity styling overrides the category tint so the operator
  cannot miss a critical event hidden inside a "noisy" category.
- Each row also shows: the category label, an inline `prev → next`
  status delta when present, the local time-of-day on the right
  (with the full ISO timestamp on hover via `title`), the optional
  human summary line, the actor (`by alice@example.com`), the source
  service, and a small "Warning" / "Critical" badge for non-info
  severities.
- `data-severity` and `data-category` are emitted on each row to
  give downstream filter UIs (#29) a stable hook without re-fetching.

## API Integration Notes
- Endpoint: `GET /flow/api/v1/admin/workflow-instances/{id}/timeline`
  (the read-only adapter shipped in E13.1).
- Auth: re-uses the existing `apiClient` JWT path; the upstream Flow
  controller re-asserts PlatformAdmin / TenantAdmin scoping and
  returns 404 for out-of-scope rows so the BFF cannot leak existence.
- Caching: 10 s on the `cc:workflows` cache tag — same lifecycle as
  `getById`, so a single `revalidateTag('cc:workflows')` invalidates
  both. No new cache tag introduced.
- Failure modes:
  - 404 → `null` timeline → drawer renders the "not available" state.
  - Other throws → caller wraps with `Promise.allSettled` so the
    detail panel still renders; the timeline section shows an
    inline error banner.
  - Empty Flow audit (`Audit:BaseUrl` not configured) → endpoint
    returns 200 with `events: []` (per E13.1) → drawer shows the
    "No audit events recorded yet" empty state.

## State Management Notes
- Timeline data is fetched server-side per request; the drawer is a
  Client Component that purely renders the props it receives.
- No client-side `useState` / `useEffect` for timeline data — the
  existing drawer-local state (`pendingAction`, `reason`, etc.) is
  untouched.
- Date grouping happens at render time (`groupEventsByDate`) rather
  than during fetch. For a 1000-event payload this is a single O(n)
  pass + O(d log d) sort over distinct days (≪ 30 in practice). No
  memoisation needed; profiling did not show the function as a hot
  path.
- The drawer's open/close URL contract (`?selected=<id>`) and the
  `closeHref` memo are unchanged.

## UX / Interaction Notes
- The day heading is sticky inside the drawer's scrollable body, so
  operators always see which day they are looking at while scrolling
  long histories.
- Severity styling is additive on top of category styling — info
  events look exactly the way they did before E13.3 so the common
  case stays low-noise.
- Critical events get a subtle red row tint instead of a loud
  background colour to keep the drawer readable when many critical
  rows pile up (e.g. a workflow that retried 20 times).
- A small "Warning" / "Critical" badge sits on the metadata line so
  the severity is also conveyed to screen readers and to operators
  who customise their accent colour.
- Time-of-day is shown in 12h local format with the full ISO
  timestamp available on hover (`title` attr) so an operator can
  copy the exact UTC value when reporting an issue.

## Known Issues / Gaps
- Timeline is fetched server-side per drawer open; there is no
  client-side polling. Operators must close-and-reopen the drawer
  (or hit the workflow's admin action, which already invalidates
  `cc:workflows`) to see new events. A future "live tail" mode is
  out of scope.
- No filtering UI inside the drawer (covered by #29) and no
  click-through from a timeline event to the audit log entry
  (covered by #28). The row-level `data-severity` / `data-category`
  attributes are intentionally in place to make those follow-ons
  easy.
- Severity is derived from the audit metadata bag today. If the
  Flow normaliser later promotes severity to a top-level DTO field,
  the BFF mapper already prefers it; no UI change required.
- No virtualisation of the event list. Profiling at the documented
  1000-event ceiling did not show measurable jank; if the cap is
  raised, swap the inner `<ol>` for a virtual list.

## Recommendation
Ship E13.3 alongside E13.2. The change is purely additive in the
Control Center, consumes the same E13.1 endpoint, and degrades
gracefully when audit is unavailable, when an unknown severity is
returned, or when individual events are partially populated.
