# E8.4 Report

> Tenant Portal Workflow Progression UI — SynqLien-First Advance / Complete Actions.

## Scope Executed

- Extended the shared workflow UI module from E8.1 with reusable
  progression action components (Advance Step, Complete Workflow), a
  shared mutation error component, and an action-bar wrapper.
- Extended the browser-side workflow data layer (`WorkflowApiAdapter`)
  with `advance()` and `complete()` methods so all products inherit the
  capability via the same adapter — no SynqLien-only mutation paths.
- Added `useAdvanceCaseWorkflow` and `useCompleteCaseWorkflow` mutation
  hooks alongside the existing E8.1 read/start hooks.
- Added two new role-access actions (`workflow:advance`,
  `workflow:complete`) mapped to existing SynqLien product roles
  (Seller || Holder; admin override).
- Wired the action bar into `WorkflowPanel` with permission and status
  gating: Advance / Complete are only rendered for an active workflow when
  the caller passes `canAdvance` / `canComplete`.
- Auto-enabled the atomic ownership-aware detail fetch when actions are
  available (Advance requires `currentStepKey` for optimistic concurrency).
- Threaded the two new permission predicates from `useRoleAccess()` into
  the existing SynqLien case-detail integration. No layout change.
- Browser → SynqLien BFF (`/api/lien/api/liens/...`) → Liens.Api → Flow.
  Browser never talks to Flow directly.

## Assumptions

- The hardened `MapFlowExecutionPassthrough` group (registered by
  `WorkflowEndpoints` for SynqLien) already exposes:
  `POST .../{workflowInstanceId:guid}/advance` and
  `POST .../{workflowInstanceId:guid}/complete`. These were verified.
- `Advance` body shape: `{ ExpectedCurrentStepKey, ToStepKey?, Payload? }`.
  Verified in `FlowExecutionEndpoints.AdvanceWorkflowBody`.
- `Complete` takes no body.
- Both endpoints return the updated `FlowWorkflowInstanceResponse`,
  which the frontend already models as `WorkflowInstanceDetail`.
- `currentStepKey` is the canonical key the engine uses to validate
  optimistic concurrency on advance — the UI sources it from the
  detail fetch (atomic ownership-aware route).
- Action permissions follow the same SynqLien mapping as `workflow:start`
  (Seller / Holder, admin override). Server-side authorization remains
  authoritative; the client-side gate is presentation-only.
- Workflow status `Active` (and `Pending` already handled by
  `pickActive`) is the only state where progression is offered. All
  terminal statuses (`Completed | Cancelled | Failed`) suppress controls.

## Repository / Architecture Notes

- The shared workflow module already encapsulates BFF coupling behind
  `WorkflowApiAdapter` (E8.1 architect feedback). E8.4 extends that same
  adapter so CareConnect / SynqFund inherit advance / complete by simply
  reusing their existing adapter instance.
- Hooks remain plain `useState` / `useCallback` — no React Query / SWR
  introduced (project convention).
- The action UI lives under `apps/web/src/components/workflow/` next to
  the existing read/start components and is exported from
  `components/workflow/index.ts`.

## Shared Workflow Action Module Notes

New components (all product-agnostic; copy parameterised; no SynqLien
strings hardcoded):

- `workflow-action-error.tsx` — friendly inline mutation-error renderer
  with status-aware copy (409 mismatch / conflict, 403, 500, etc.) and
  optional retry.
- `advance-workflow-action.tsx` — inline expander with
  current-step display, optional `toStepKey` input, submitting/idle/
  error states. Disabled when `currentStepKey` not yet known.
- `complete-workflow-action.tsx` — inline confirm/cancel with
  submitting state and friendly error.
- `workflow-action-bar.tsx` — orchestrates the two actions for an
  active workflow row + detail. Suppresses itself for terminal statuses.

## SynqLien Progression Integration Notes

- `apps/web/src/app/(platform)/lien/cases/[id]/case-detail-client.tsx`:
  no layout changes; the existing `<CollapsibleSection title="Workflow">`
  in the Details-tab right rail now also passes `canAdvance` and
  `canComplete` from `useRoleAccess()` into `<WorkflowPanel />`.
- `WorkflowPanel`:
  - new optional props `canAdvance`, `canComplete` (default `false`)
  - `loadDetail` now defaults to `true` whenever advance or complete is
    available so the action bar has the `currentStepKey` it needs
  - `WorkflowActionBar` is rendered after `WorkflowSummary` for active
    workflows; `WorkflowActionBar` itself returns `null` for terminal
    statuses, so terminal rows degrade to a read-only summary with the
    existing "Start another" affordance still gated by `canStart`.

## API Wiring Notes

- Endpoints used (all already shipped by Liens.Api via the
  `MapFlowExecutionPassthrough` helper):
  - `GET  /api/liens/cases/{id}/workflows`
  - `GET  /api/liens/cases/{id}/workflows/{workflowInstanceId}`
  - `POST /api/liens/cases/{id}/workflows/{workflowInstanceId}/advance`
  - `POST /api/liens/cases/{id}/workflows/{workflowInstanceId}/complete`
- All routes called through the SynqLien BFF prefix (`/lien/api/liens`)
  by the default `workflowApi` adapter. No new backend code needed for
  E8.4.
- After a successful mutation the panel calls `refresh()` (re-lists the
  case workflows) and `loadActiveDetail()` (re-fetches the atomic detail).

## Permission / Visibility Notes

- Two new `LienAction` values: `workflow:advance`, `workflow:complete`.
  Mapped in `role-access.service.ts` to `isSeller || isHolder` with the
  admin / tenant-admin override preserved.
- The action bar is hidden entirely when the workflow is terminal.
- When the user has only `workflow:view` (no act permissions) the panel
  still renders the summary and the actions are simply absent.
- 401 / 403 responses from `advance` / `complete` are caught and shown
  as a compact in-panel error; the rest of the case page continues to
  render normally.
- Limitation (carried forward from E8.1): there is no per-instance
  capabilities endpoint, so client gating uses role-based heuristics
  while the backend remains authoritative.

## Validation Results

- `Start application` workflow restarted cleanly. Next.js compiled the
  edited `case-detail-client.tsx` and the new workflow components without
  errors. No TypeScript or React diagnostics raised by Fast Refresh.
- Smoke: `GET /lien/cases/{id}` returns `307` (auth redirect to `/login`)
  as expected for an unauthenticated curl. Browser console shows only
  `[Fast Refresh] done` lines — no module / runtime errors.
- Backend untouched (verified by inspection): existing `advance` /
  `complete` Flow passthrough already mapped through SynqLien BFF.
- Architect review: PASS. Verified API wiring, optimistic-concurrency
  handling, reusability, permission gating, refresh discipline, error
  handling, and React hook correctness.

## Known Issues / Gaps

- Optional `Payload` map on the Advance endpoint is supported by the
  adapter but not yet exposed in the UI. SynqLien doesn't need it for
  E8.4; CareConnect / SynqFund can extend the action component with a
  product-specific payload editor when needed.
- Client-side gating remains role-heuristic (no per-instance
  capabilities endpoint). Backend authorization is authoritative.

## Recommendation

E8.4 is ready for closure. The shared workflow module now offers a
complete read / start / advance / complete lifecycle through a single
product-agnostic surface, and SynqLien is wired end-to-end without any
direct browser-to-Flow coupling. Recommend rolling the same
`<WorkflowPanel>` integration into CareConnect and SynqFund as those
products ship their workflow UIs.
