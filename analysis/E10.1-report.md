# E10.1 Report

> Control Center Admin Actions — Controlled Workflow Intervention Layer.
> PlatformAdmin can Retry / Force Complete / Cancel eligible workflows
> from the Control Center detail drawer. Every action requires a
> non-empty reason and is recorded in the Audit service. Flow remains
> the single execution authority.

## Scope Executed
- Backend: 3 new admin POST endpoints on Flow's
  `AdminWorkflowInstancesController` (`retry`, `force-complete`,
  `cancel`) sharing one `HandleAdminActionAsync` pipeline:
  reason validation → tenant scoping → load → state-matrix delegation →
  save → audit emit → structured response.
- New DTOs (`AdminWorkflowActionRequest`, `AdminWorkflowActionResult`)
  versioned alongside the existing read DTOs.
- Audit emitted via existing `IAuditAdapter` (fire-and-forget safe;
  additionally try/catch + warn so audit-pipeline outage cannot fail
  the action).
- Errors as ProblemDetails: `reason_required` (400),
  `not_allowed_in_state` (409), `concurrent_state_change` (409).
- Control Center types (`WorkflowAdminAction`,
  `WorkflowAdminActionResult`) and a thin server-side API client
  method (`controlCenterServerApi.workflows.adminAction`).
- New CC BFF route `POST /api/admin/workflow-instances/[id]/[action]`
  guarded by `requirePlatformAdmin`, validates action verb + reason,
  proxies to Flow with the user's JWT, busts `CACHE_TAGS.workflows`
  on success, and translates upstream errors back to the drawer with
  a best-effort status code.
- Drawer `WorkflowDetailDrawer` extended with an "Admin actions"
  section gated on `canAdmin` prop. Three buttons (Retry / Force
  complete / Cancel) with eligibility computed client-side from
  `detail.status` (+ `lastErrorMessage` for retry); ineligible buttons
  are disabled with hover hint. Active button opens an inline
  confirmation panel with a required reason textarea (capped at 1000
  chars, mirrors server cap), submit calls the BFF and on success
  triggers `router.refresh()`. Errors render in a compact red banner
  inside the panel.
- `canAdmin` plumbed through `/workflows/page.tsx` and
  `/workflows/exceptions/page.tsx` (both pages already require
  PlatformAdmin server-side, so the prop is set unconditionally there).

## Assumptions
- "Retry" semantics: re-enter engine-managed execution from the
  current `CurrentStageId` / `CurrentStepKey`. Implemented as
  `Status="Active"`, clear `LastErrorMessage`, bump `UpdatedAt`. The
  Flow engine is responsible for picking it up on its next tick. We
  intentionally do not rewind history or re-create steps; that is a
  larger workflow-engine concern and is out of scope for E10.1.
- Reason is stored only on the audit record (and echoed in the
  response). No new column added to `WorkflowInstance`.
- Tenant visibility for the BFF is PlatformAdmin-only at the CC
  surface for this phase. The Flow endpoint additionally permits
  `PlatformOrTenantAdmin` (with cross-tenant rows hidden as 404, not
  403) so future TenantAdmin Control Center surfaces can light up the
  same endpoints without backend changes.
- Audit `Action` strings: `workflow.admin.retry`,
  `workflow.admin.force_complete`, `workflow.admin.cancel`. Underscore
  spelling for `force_complete` matches existing Flow constant style;
  the URL path uses the kebab `force-complete` form per HTTP convention.
- `performedBy` resolves from the JWT subject claims in priority
  order: `preferred_username` → `email` → `ClaimTypes.Email` →
  `ClaimTypes.NameIdentifier` → `sub` → `Identity.Name`, falling back
  to `unknown_admin` so an audit row is never produced with an empty
  actor.

## Admin Endpoint Notes
| Action | URL | Allowed states | Mutation |
| --- | --- | --- | --- |
| Retry | `POST .../{id}/retry` | `Failed` ∪ (`Active`/`Pending` with `LastErrorMessage`) | `Status=Active`, clear error, bump UpdatedAt |
| Force complete | `POST .../{id}/force-complete` | `Active` ∪ `Pending` | `Status=Completed`, set CompletedAt+UpdatedAt |
| Cancel | `POST .../{id}/cancel` | `Active` ∪ `Pending` | `Status=Cancelled`, set CompletedAt+UpdatedAt |

Common: reason required (trim, ≤1000 chars), tenant scoping (404 on
cross-tenant for non-PlatformAdmin), `DbUpdateConcurrencyException` →
409 `concurrent_state_change`, audit emitted post-save with metadata
(`source`, `productKey`, `previousStatus`, `newStatus`, `reason`,
`isPlatformAdmin`).

Response body:
```jsonc
{
  "workflowInstanceId": "…",
  "action":             "retry",
  "previousStatus":     "Failed",
  "newStatus":          "Active",
  "performedBy":        "alice@example.com",
  "timestamp":          "2026-04-17T14:33:00.123Z",
  "reason":             "Investigated upstream timeout, safe to resume."
}
```

## Control Center UI Notes
- Drawer renders the Admin Actions section only when the parent page
  passes `canAdmin`. Both Workflow pages currently set this
  unconditionally because they already gate on PlatformAdmin
  server-side; future TenantAdmin pages can omit the prop without UI
  changes.
- Eligibility helper `isActionEligible` mirrors the server matrix; if
  the client/server divergence ever occurs, the request still
  completes round-trip and the drawer shows the
  `not_allowed_in_state` error in a banner without breaking state.
- Confirmation panel:
  - Required reason textarea, 1000-char cap (matches server
    `MaxReasonLength`).
  - Submit button disabled when reason is empty or while submitting.
  - On 4xx, the body's `error|detail|title` is shown in a red banner;
    panel stays open so operator can adjust and retry.
  - On 2xx, `router.refresh()` re-runs the page's server data fetch,
    so the drawer reflects the new `status` immediately.

## Audit / Logging Notes
- Audit event shape (existing `AuditEvent` record):
  - `Action`: `workflow.admin.{action}` (e.g. `workflow.admin.cancel`).
  - `EntityType`: `WorkflowInstance`, `EntityId`: instance Guid.
  - `TenantId`: from the workflow row (not the operator), so audit
    rows are queryable per-tenant even when a PlatformAdmin acts.
  - `UserId`: best-effort actor as described under Assumptions.
  - `Description`: `Admin action {a}: {previousStatus} → {newStatus}`.
  - `Metadata`: source / productKey / previousStatus / newStatus /
    reason / isPlatformAdmin.
- Audit write is wrapped in try/catch with a structured warn log; the
  admin action is also recorded via `LogInformation` regardless so a
  forensic trail exists in app logs even in the (already
  documented-as-unlikely) case of audit pipeline failure.
- CC BFF emits `revalidateTag(CACHE_TAGS.workflows)` on success so
  list/exception views see fresh data on next render.

## Authorization / Visibility Notes
- Flow endpoints reuse the existing `[Authorize(Policy =
  AuthorizationPolicies.PlatformOrTenantAdmin)]` controller-level
  attribute. PlatformAdmin acts cross-tenant; TenantAdmin acts only on
  their own tenant's rows; cross-tenant rows are hidden as 404 (parity
  with E9.1/E9.2 GetById hardening).
- CC BFF route enforces `requirePlatformAdmin` only — TenantAdmin
  Control Center surfaces are intentionally not added in this phase.
- The Flow controller is the single enforcement point for the state
  matrix and the audit emit; the CC client is the cosmetic layer.

## Validation Results
- Backend build: `dotnet build src/Flow.Api/Flow.Api.csproj` → clean.
- Workflow restart: `Start application` came up; gateway reachable on
  `:5010`, Control Center on `:5004`.
- Smoke (no auth):
  - `POST /flow/api/v1/admin/workflow-instances/{id}/retry` → `401`
  - `POST .../cancel` → `401`
  - `POST .../bogus` → `401` (auth runs before route resolution; an
    authenticated request to a bogus action would 404)
  - `POST /api/admin/workflow-instances/{id}/retry` (CC, no session)
    → `307 → /login?reason=unauthenticated` (expected)
- Empty-reason and state-matrix paths could not be exercised
  end-to-end without an authenticated session, but the request
  validator + ProblemDetails wiring are unit-trivial: verified by
  reading the implementation against the contract.

## Known Issues / Gaps
- "Retry" relies on the existing engine to pick up an `Active`
  workflow with no further nudging. If the engine instead expects an
  explicit "kick" message in some product flows, that gap will surface
  as the workflow staying Active without progress. Mitigation: this is
  observable in the existing E9 monitoring/exceptions views.
- The CC BFF now translates upstream Flow errors via the structured
  `ApiError.status` field plus a small allow-list of curated
  ProblemDetails titles (`reason_required`, `not_allowed_in_state`,
  `concurrent_state_change`). Raw upstream message text is never
  echoed back to the operator — every status maps to a curated code +
  operator-safe message; upstream 5xx is normalised to 502 with a
  generic message. The Flow audit + ASP.NET log remain the source of
  truth for forensic detail.
- TenantAdmin Control Center surfaces for these actions are not part
  of this phase; backend already supports them (see Assumptions).

## Architect Review
- Initial verdict: PASS on backend state matrix, tenant scoping,
  audit safety, concurrency handling, and drawer behavior.
- One Major raised: brittle BFF error translation via message
  string-matching. Resolved in a follow-up edit by switching to the
  structured `ApiError.status` field with a curated
  status → code/message table and a small allow-list for forwarding
  Flow's known ProblemDetails titles. Raw upstream text is no longer
  echoed to the operator; upstream 5xx is normalised to 502.

## Recommendation
Ship E10.1 to staging behind PlatformAdmin. The Flow controller is
the single enforcement point and the audit trail is comprehensive.
Pair with a short SOP describing when each action is appropriate and
what to write in the reason field — the audit value compounds when
operators write meaningful reasons.
