# E14.2 Report

LegalSynq Flow — **Workflow Task Claim / Reassign — Governed Assignment Transition APIs**.

Builds on E14.1 (`AssignmentMode` + `AssignedAt`/`AssignedBy`/`AssignmentReason` + single-mode invariants in `WorkflowTask.EnsureValid()`). E14.2 makes those fields actionable through two governed HTTP endpoints (`claim`, `reassign`) backed by a single dedicated transition service.

## Scope Executed

- New `WorkflowTaskAssignmentService` — the only path through which user-driven assignment mutations may flow.
- Two new endpoints on the existing `WorkflowTasksController`:
  - `POST /api/v1/workflow-tasks/{id}/claim`
  - `POST /api/v1/workflow-tasks/{id}/reassign`
- Two new structured exception types (`AssignmentRuleException`, `AssignmentForbiddenException`) with a stable `Code` field, mapped to **422** and **403** by `ExceptionHandlingMiddleware`.
- Stable error-code catalog `WorkflowTaskAssignmentErrorCodes`.
- Audit emission on every successful claim and reassign via the existing `IAuditAdapter`.
- DI wiring for the new service.

Out of scope per spec and explicitly **not** built: queue UI, bulk operations, notifications, SLA/workload balancing, history surfaces, advanced dashboards, workflow-engine redesign, broad My Tasks UI changes.

## Assumptions

1. **Route base.** Spec suggests `/api/v1/tasks/{id}/{claim,reassign}` but the existing per-task lifecycle surface is `/api/v1/workflow-tasks/{id}/...`. I mounted the new endpoints on `WorkflowTasksController` to keep all per-task mutations on one controller (start/complete/cancel/claim/reassign). `/api/v1/tasks` in this codebase is the legacy `TaskItem` controller, which is a different grain.
2. **Claimable statuses.** Only `Open`. `InProgress` is excluded — once a worker has actively started a task, claim is the wrong primitive (use reassign). `Completed` / `Cancelled` are terminal.
3. **Reassignable statuses.** `Open` and `InProgress`. Terminal states are rejected. Reassigning an InProgress task is allowed because supervisors must be able to redirect work that someone else has started.
4. **`Unassigned` as a claim source is rejected.** Spec gives latitude here. Genuinely unassigned tasks are an unusual shape (the resolver normally produces a queue or direct assignment) — letting *any* authenticated user grab them would be a privilege loophole. Admins can use reassign-to-self (or reassign-to-another) instead. Documented in the service.
5. **`Unassigned` as a reassign target is allowed only for admins.** Useful for "park this task" workflows; same authorization gate as the rest of reassign.
6. **Self-claim of own DirectUser task.** Returns `task_already_assigned` (422). Idempotency is intentionally not modelled; the result is observable and the caller can handle it.
7. **Reason field.** Required for reassign (non-whitespace, ≤ 500 chars). Optional for claim — claim has implicit semantics ("I picked this up from the queue") so a reason adds noise. Persisted when supplied.
8. **Authorization model.**
   - Claim: any authenticated user with a tenant context. Eligibility for the queue is checked by the service against `ICurrentRequestContext.Roles` (∪ `ProductRoles`) for `RoleQueue` and `OrgId` for `OrgQueue`.
   - Reassign: gated at the controller layer by `[Authorize(Policy = Policies.PlatformOrTenantAdmin)]`. Backend remains authoritative — the service does **not** trust the controller and re-asserts the check via `ICurrentRequestContext.IsPlatformAdmin || Roles.Contains("TenantAdmin")`.
9. **Tenant non-leakage.** Both endpoints rely on the global `WorkflowTask` query filter; cross-tenant ids surface as `NotFoundException` → 404, identical to a missing task.
10. **Concurrency token.** `WorkflowTask` has no `RowVersion`. We use the same CAS pattern as `WorkflowTaskLifecycleService`: `ExecuteUpdateAsync` with `WHERE Id = @id AND Status = @prevStatus AND AssignmentMode = @prevMode AND AssignedUserId = @prevUserId AND AssignedRole = @prevRole AND AssignedOrgId = @prevOrgId`. A losing race surfaces as `WorkflowTaskConcurrencyException` → 409.
11. **E14.1 invariant re-application.** Per the E14.1 caveat in `WorkflowTaskLifecycleService`, any helper that uses `ExecuteUpdateAsync` to mutate assignment columns must re-apply the single-mode rule inline because the `EnsureValid` save-hook is bypassed. The new service constructs the target tuple deterministically from validated input, then runs `WorkflowTaskAssignmentMode.IsKnown(...)` plus a switch that mirrors `EnsureValid` exactly before issuing the update.
12. **Audit failure handling.** Audit is best-effort; a thrown audit exception is logged and swallowed so the user-visible operation is not undone. This matches `IAuditAdapter`'s "fire-and-forget safe" doc-contract.

## Assignment Transition Service Notes

`WorkflowTaskAssignmentService` exposes two methods:

- `ClaimAsync(Guid taskId, string? reason, CancellationToken)` → `WorkflowTaskAssignmentResult`
- `ReassignAsync(Guid taskId, ReassignRequest req, CancellationToken)` → `WorkflowTaskAssignmentResult`

Both follow the same shape:

1. **Read** the current row (tenant-filtered, `AsNoTracking`) — pull `WorkflowInstanceId`, `Status`, `AssignmentMode`, all three target columns. Missing → `NotFoundException` (→ 404, identical for cross-tenant).
2. **Validate state** — terminal status rejected (`task_state_invalid`).
3. **Validate the requested transition** — claim has its own rules (must be from queue/role/org; not from `Unassigned`; not from someone else's `DirectUser`); reassign requires reason + target tuple matching target mode exactly.
4. **Authorize** — claim checks queue eligibility against `ICurrentRequestContext`; reassign re-asserts admin role.
5. **Construct target tuple** — `(newMode, newUserId, newRole, newOrgId)` derived deterministically from validated input.
6. **Re-apply E14.1 single-mode rule** — local switch identical to `EnsureValid`; throws `AssignmentRuleException` on any mismatch (defence-in-depth — the validated input should already be correct).
7. **CAS update** — `ExecuteUpdateAsync` with the previous five columns in the WHERE clause + `Id`. Sets `AssignmentMode`, all three targets, `AssignedAt`, `AssignedBy`, `AssignmentReason`, `UpdatedAt`, `UpdatedBy`. Zero rows affected → `WorkflowTaskConcurrencyException` (→ 409).
8. **Audit** — fire `workflow.task.claim` or `workflow.task.reassign` via `IAuditAdapter`; failures logged + swallowed.
9. **Return** a `WorkflowTaskAssignmentResult` so the controller can echo new state without a follow-up GET.

## Claim API Notes

`POST /api/v1/workflow-tasks/{id}/claim`

**Authorization:** `Policies.AuthenticatedUser` (inherited from controller).

**Request body** (optional):
```json
{ "reason": "string?  (≤ 500 chars)" }
```

**Behaviour:** the authenticated caller becomes `DirectUser` assignee. Eligibility:

| Source assignment mode | Allowed? | Rule |
| --- | --- | --- |
| `RoleQueue`  | ✅ | Caller must hold `AssignedRole` in `Roles` ∪ `ProductRoles`. |
| `OrgQueue`   | ✅ | Caller's `OrgId` must equal `AssignedOrgId`. |
| `DirectUser` (already mine) | ❌ | `task_already_assigned` (422). |
| `DirectUser` (someone else) | ❌ | `task_already_assigned` (422). |
| `Unassigned` | ❌ | `task_not_claimable` (422). Use reassign. |

**Status guard:** only `Open` is claimable. `InProgress`/`Completed`/`Cancelled` → `task_state_invalid` (422).

**Tenant guard:** wrong tenant → `NotFoundException` → 404 (no leakage).

**Stamping:** `AssignmentMode = DirectUser`, `AssignedUserId = caller`, `AssignedRole/OrgId = null`, `AssignedAt = now`, `AssignedBy = caller`, `AssignmentReason = body.reason ?? "claimed from queue"`.

## Reassign API Notes

`POST /api/v1/workflow-tasks/{id}/reassign`

**Authorization:** `Policies.PlatformOrTenantAdmin` (overrides controller's `AuthenticatedUser`). Re-checked in the service.

**Request body** (required):
```json
{
  "targetMode":     "DirectUser | RoleQueue | OrgQueue | Unassigned",
  "assignedUserId": "string?",
  "assignedRole":   "string?",
  "assignedOrgId":  "string?",
  "reason":         "string (required, non-whitespace, ≤ 500 chars)"
}
```

**Validation:**
- `targetMode` required, must be one of the four known modes (`assignment_mode_invalid`).
- `reason` required (`missing_assignment_reason`).
- Target fields must match `targetMode` exactly (`assignment_target_invalid`):
  - `DirectUser`: `assignedUserId` required; others must be null.
  - `RoleQueue`: `assignedRole` required; `assignedUserId` must be null.
  - `OrgQueue`: `assignedOrgId` required; `assignedUserId` must be null.
  - `Unassigned`: all three target fields must be null.

**Status guard:** `Open` or `InProgress`. Terminal → `task_state_invalid` (422).

**Stamping:**
- `Unassigned` target: `AssignedAt = null`, `AssignedBy = null`, `AssignmentReason = null` (per E14.1 invariant).
- All other targets: `AssignedAt = now`, `AssignedBy = caller`, `AssignmentReason = body.reason`.

## Authorization / Eligibility Notes

Two distinct gates as required:

| Action | Controller policy | Service-layer re-assertion |
| --- | --- | --- |
| Claim | `AuthenticatedUser` | Queue eligibility (role/org match) computed against `ICurrentRequestContext`. |
| Reassign | `PlatformOrTenantAdmin` | `IsPlatformAdmin || Roles.Contains("TenantAdmin")`. |

Both gates close to **403** (`AssignmentForbiddenException`) for ineligible callers — never to 404, except where the row genuinely is not visible to the tenant (then it is correctly 404). For claim, ineligibility is *not* a 401 because the caller is correctly authenticated; it is a *queue-membership* refusal.

## Audit / Metadata Notes

Every successful claim and reassign emits one `AuditEvent`:

| Action | EntityType | Description |
| --- | --- | --- |
| `workflow.task.claim` | `WorkflowTask` | `"Task claimed from <prevMode>"`. |
| `workflow.task.reassign` | `WorkflowTask` | `"Task reassigned <prevMode> → <newMode>"`. |

**Metadata fields** (all stringified for the dictionary):
- `taskId`, `workflowInstanceId`
- `prevMode`, `prevAssignedUserId`, `prevAssignedRole`, `prevAssignedOrgId`
- `newMode`, `newAssignedUserId`, `newAssignedRole`, `newAssignedOrgId`
- `reason` (when supplied)
- `performedBy` (= `_user.UserId`)

`OccurredAtUtc = DateTime.UtcNow` at the moment after the CAS succeeds.

`AssignedAt` / `AssignedBy` / `AssignmentReason` are stamped on the row in the same `ExecuteUpdateAsync` call as the target columns, so the persisted state is always internally consistent — no two-step write that could leave audit fields blank if the second write failed.

## DTO / API Notes

- **Request DTOs (new, additive):** `ClaimTaskRequest`, `ReassignTaskRequest`. Defined alongside the controller.
- **Response DTO (new):** `WorkflowTaskAssignmentResult` — small record exposing `TaskId`, `WorkflowInstanceId`, `AssignmentMode`, `AssignedUserId`, `AssignedRole`, `AssignedOrgId`, `AssignedAt`, `AssignedBy`, `AssignmentReason`, `Status`, `OccurredAtUtc`.
- **No change** to existing `MyTaskDto`, `WorkflowTaskTransitionResult`, or `WorkflowTaskCompletionResult`. My Tasks consumers are unaffected. Widening `MyTaskDto` to expose the new assignment metadata is deferred (the data is already returned by the new endpoints; UI can fetch on demand).
- **Error response** (422 / 409 / 403 / 404): existing `ErrorResponse` shape (`{error, errors?}`). The new structured exceptions surface their `Code` value inside the `error` message string for client-side disambiguation, matching `InvalidWorkflowTransitionException`'s precedent.

## Architect Review Delta (post-first-pass fixes)

The first architect pass flagged two issues; both are addressed in the current revision:

1. **Fail-open on malformed source queue rows.** Original code's `EnsureCallerHoldsRole` / `EnsureCallerInOrg` early-returned when the queue's required role / org was blank, which would have let any authenticated user claim a `RoleQueue`/`OrgQueue` task with a missing target string (only reachable via legacy / migration-drift data, but still a governance hole). **Fix:** both helpers now throw `AssignmentRuleException(assignment_target_invalid)` ⇒ 422 when the queue target is missing. The data shape is treated as "cannot evaluate eligibility" rather than "no eligibility required".
2. **Reason length silently truncated.** Original `NormalizeReason` truncated to 500 chars instead of rejecting. That would have mis-recorded what the caller actually said in the audit row. **Fix:** `NormalizeReason` now throws `AssignmentRuleException(assignment_target_invalid)` ⇒ 422 with the offending length. No truncation.

Both fixes preserve all other behaviour. Build remains clean (0 errors).

## Known Issues / Gaps

1. **Pre-existing dev-environment migration drift.** `__EFMigrationsHistory` for the Flow database in this sandbox doesn't include E11.1 or E14.1, so `flow_workflow_tasks` may not exist locally. Migration files themselves are correct. Tracked from E14.1; no E14.2 code change can fix this without owning the dev bootstrap.
2. **No `RowVersion` column on `WorkflowTask`.** Concurrency is provided by a 5-column CAS WHERE clause (`Status`, `AssignmentMode`, `AssignedUserId`, `AssignedRole`, `AssignedOrgId`). Sufficient for the assignment transition surface — it is impossible for a competing claim/reassign to land between our read and write without changing at least one of those columns. If E14.x grows further mutations on assignment metadata that don't change one of those columns, a true `RowVersion` may become attractive.
3. **No bulk endpoints** (`/claim/bulk`, `/reassign/bulk`). Out of scope per spec; would require chunked CAS + per-row audit and a 207 multi-status result shape.
4. **No notifications** on reassign (e.g. "you've been assigned a task"). Out of scope per spec.
5. **`MyTaskDto` not widened** with the new assignment metadata. The new endpoints return the full new state in their response, so UI can patch its local cache without a follow-up list fetch. Deferred until queue UI work picks up the data demand.
6. **No unit tests added.** Spec did not require them. The CAS + invariant-mirroring switch is small enough to follow by inspection; recommend a focused test pass once Flow's test infrastructure is back in working order.
7. **Queue-eligibility role check is exact-string-match.** `ICurrentRequestContext.Roles` and `ProductRoles` are searched with `OrdinalIgnoreCase`. No role hierarchy / capability resolution is done — if the platform later introduces a "Supervisor implies CareConnectReceiver" mapping, this service will need updating.

## Recommendation

**Ready for queue UI / SLA phase.** E14.2 closes the gap between the E14.1 model and operator-driven workflow management:

- The single-mode invariant is preserved (E14.1 save-hook still fires for every `SaveChanges` path; the new CAS path mirrors the same switch inline).
- Tenant isolation is preserved (global query filter; cross-tenant returns 404).
- Concurrency is safe (5-column CAS; race losers see 409).
- Audit is wired (`workflow.task.claim` / `workflow.task.reassign` with full prev/new metadata).
- Authorization is split into a worker gate (claim) and an admin gate (reassign), and re-asserted in the service.

Suggested next phases:
- **E14.3 (queue UI):** list role/org queues with `AssignmentMode == RoleQueue|OrgQueue`, expose claim button, show per-task `AssignedAt`/`AssignedBy`.
- **E14.4 (SLA & workload):** overdue indicators driven by `AssignedAt`, fairness-aware dispatch suggestions.
- **E14.5 (history):** read model over `IAuditAdapter` for a per-task assignment timeline.
