# E9.1 Report

> Control Center Workflow Operations List — Cross-Product Read-Only Visibility.

## Scope Executed

- Added a new admin-scope listing endpoint to the Flow service:
  `GET /api/v1/admin/workflow-instances`.
- Added a paged, filterable, read-only **Workflows** page to the Control
  Center (port 5004) at `/workflows`, surfaced under the OPERATIONS nav
  section.
- Reused the existing tenant-list page pattern (server component +
  `requirePlatformAdmin()` + `controlCenterServerApi.<x>.list()` +
  table component) to keep the new page consistent with the rest of the
  Control Center.

## Assumptions

- The Control Center surface is platform-operations-only today, so the
  page is gated by `requirePlatformAdmin()` even though the underlying
  Flow endpoint also accepts TenantAdmin (server-side scoped).
- The Flow MySQL provider is case-insensitive at the default collation,
  so `EF.Functions.Like(field, "%term%")` is sufficient for the search
  filter — no per-row `LOWER()` cast required.
- "Source entity" context (entity type / id) is best-effort: it is read
  from the optional `ProductWorkflowMapping.WorkflowInstanceId` join,
  which is populated for new mappings but may be null on legacy rows.
- `WorkflowInstance.TenantId` is stored as a lowercase Guid string
  (verified against `ClaimsTenantProvider`), so the inbound `tenantId`
  filter is normalised to lowercase server-side before comparison.

## Repository / Architecture Notes

- Control Center is a standalone Next.js 15 app at `apps/control-center/`,
  not part of `apps/web`. It has its own server-side API client
  (`lib/control-center-api.ts`), nav (`lib/nav.ts`), routes
  (`lib/routes.ts`), auth guards (`lib/auth-guards.ts`), and types
  (`types/control-center.ts`). Patterns from `apps/web` were ported, not
  imported.
- All Control Center → Flow traffic flows through the platform gateway
  using the `/flow/{**catch-all}` route → `flow-cluster@5012`. The page
  never calls Flow directly from the browser.
- `FlowDbContext` applies a per-tenant query filter via `ITenantProvider`
  on every workflow-bearing entity. The new admin endpoint bypasses that
  filter via `IgnoreQueryFilters()` and re-applies tenant scoping in code
  so a TenantAdmin caller can never accidentally see another tenant's
  rows even if they pass a `tenantId` query parameter.

## Control Center Workflow List Notes

- New page: `apps/control-center/src/app/workflows/page.tsx`.
  - Server Component; awaits `searchParams` (Next 15 contract).
  - Reads `requirePlatformAdmin()` for access gate, surfaces the current
    `getTenantContext()` scope as an amber chip when the operator is
    pinned to a single tenant.
- New table component:
  `apps/control-center/src/components/workflows/workflow-operations-table.tsx`.
  - Columns: Workflow (name + short id), Product, Status (coloured
    badge), Current Step, Source Entity, Tenant, Assigned, Started,
    Updated.
  - Pagination preserves the active filters via a `baseQuery` prop.
  - Empty state and stable formatters for timestamps / Guid prefixes.

## API / BFF Notes

- New backend file:
  `apps/services/flow/backend/src/Flow.Api/Controllers/V1/AdminWorkflowInstancesController.cs`.
  - Route: `GET /api/v1/admin/workflow-instances`
  - Authorization: `Policies.PlatformOrTenantAdmin`.
  - Query params: `productKey?`, `status?`, `tenantId?`, `search?`,
    `page=1`, `pageSize=20` (capped at 100).
  - Bypasses the tenant query filter via `IgnoreQueryFilters()` and
    applies tenant scoping explicitly:
    - PlatformAdmin: cross-tenant; optionally narrowed by `tenantId`
      query param.
    - TenantAdmin: forced to caller's own tenant via
      `_tenantProvider.GetTenantId()`; the inbound `tenantId` is
      ignored. Returns `Forbid()` if no tenant claim is present.
  - Joins `WorkflowInstance` → `FlowDefinition` (for display name) and
    optional `ProductWorkflowMapping` (for source entity context). Both
    sub-queries also use `IgnoreQueryFilters()` so the join result is
    consistent with the bypassed primary query for PlatformAdmin, **but
    each sub-query re-applies `TenantId == w.TenantId` explicitly** to
    prevent stray cross-tenant rows from leaking metadata onto an
    instance row (defence-in-depth against data-quality edge cases).
  - Mapping sub-query is **deterministic for multi-mapping instances**:
    it prefers `Status == "Active"` and ties-breaks by most-recent
    `UpdatedAt` then `CreatedAt`, so the displayed source entity is
    stable across query plans.
  - Response DTOs (`AdminWorkflowInstanceListItem`,
    `AdminWorkflowInstanceListResponse`) are local to the controller
    file and intentionally read-only — execution / mutation surfaces
    remain on the existing `WorkflowInstancesController`.
- Control Center API client extension:
  `controlCenterServerApi.workflows.list({ ... })` in
  `apps/control-center/src/lib/control-center-api.ts`. Cache: **10s**,
  tag `cc:workflows` (added to `CACHE_TAGS`). Short TTL because workflow
  rows change frequently in production.
- New normaliser `mapWorkflowInstanceListItem` lowercases tenantId,
  coerces empty strings to `null` for optional fields, and tolerates a
  missing/empty `items` array.

## Filter / Search / Status Notes

- Status filter values mirror Flow's `WorkflowEngine` constants
  (`Active`, `Pending`, `Completed`, `Cancelled`, `Failed`).
- Product filter values mirror `Flow.Domain.Common.ProductKeys.All`
  (e.g. `FLOW_GENERIC`, `SYNQ_LIENS`, `SYNQ_FUND`, …). The table uses a
  small label map so future product keys still render legibly even if
  the filter dropdown isn't updated.
- `search` matches (case-insensitive) against `CorrelationKey`,
  `CurrentStepKey`, and the workflow-instance id stringified — these
  cover the most common operator paste-ins (case id, correlation key,
  step name).
- Ordering is `UpdatedAt DESC` falling back to `CreatedAt`, so the most
  recently touched workflows surface first.

## Permission / Visibility Notes

- Page guard: `requirePlatformAdmin()` — Control Center surface is
  platform-only today.
- API guard: `Policies.PlatformOrTenantAdmin` (matches the existing Flow
  policy registration). Cross-tenant visibility is gated by
  `User.IsInRole(Roles.PlatformAdmin)`, never by request input.
- Inbound `tenantId` query param is ignored for non-PlatformAdmins; the
  page never passes it for tenant admins because the page itself is
  platform-only.
- Endpoint is read-only (HTTP GET only); execution endpoints
  (`Start`, `Advance`, `Complete`, `Cancel`) are unchanged on the
  existing `WorkflowInstancesController`.

## Known Issues / Gaps

- The page does not yet expose a "View detail" deep link per row — there
  is no Control Center workflow-detail surface yet. The list shows
  enough context (workflow name, current step, source entity, assigned
  user) to triage; a follow-up can wire in a detail drawer once we
  decide whether it belongs in CC or back in the product surfaces.
- The product filter dropdown is hard-coded against the current
  `ProductKeys` set; if Flow adds a new product the table will still
  render the value as a fallback label, but the dropdown won't list it
  until updated.
- No CSV export — out of scope for E9.1 per brief, but noted as a
  natural next iteration.
- Search is `LIKE %term%` which will hit a table scan on large datasets;
  acceptable today (Flow workflow volume is modest), revisit when row
  counts cross ~100k or if a second high-traffic admin surface appears.

## Recommendation

Ship E9.1 as-is. The new admin endpoint is correctly scoped, the
Control Center surface follows the established tenant-list pattern, and
the read-only contract keeps the blast radius small. The next natural
slice (E9.2-style follow-up) would be a workflow-detail drawer or page
that surfaces the step graph and recent transitions for a selected
instance.
