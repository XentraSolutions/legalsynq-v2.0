# E9.3 Report

> Control Center Exception / Stuck Workflow View — Operational Diagnostics
> and Triage. Read-only triage surface that surfaces workflows currently
> Failed, Cancelled, Stuck, or carrying an engine error message, and
> reuses the E9.2 detail drawer for inspection.

## Scope Executed

- **Backend (Flow.Api)** — extended the existing admin list endpoint
  `GET /api/v1/admin/workflow-instances` with three optional triage
  query parameters and tagged every returned row with the
  classification labels that apply. No new endpoint.
- **Control Center types + API client** — extended
  `WorkflowInstanceListItem` with `lastErrorMessage?` and
  `classifications?: WorkflowClassification[]`; added
  `controlCenterServerApi.workflows.listExceptions(...)` that calls the
  same endpoint with `exceptionOnly=true`.
- **Control Center UI** — new server route `/workflows/exceptions` with
  classification / stale-threshold / product / status / tenant / search
  filters, a dedicated `WorkflowExceptionsTable` showing per-row Reason
  badges, and re-uses the E9.2 `WorkflowDetailDrawer` via the existing
  `?selected=<id>` URL contract.
- **Nav + cross-link** — added "Workflow exceptions" under the existing
  OPERATIONS group in `nav.ts`, plus a small `View exceptions` link in
  the `/workflows` header.

## Assumptions

- A workflow's "last touched" timestamp is `UpdatedAt ?? CreatedAt`.
  The E9.1 list already orders by this, so reusing it for the Stuck
  predicate keeps ordering and detection consistent.
- The Stuck rule only applies to `Status in ("Active","Pending")`. A
  Completed workflow is by definition not stuck even if it hasn't been
  touched in months.
- A row may match multiple classifications (e.g. a Failed workflow that
  also has a `LastErrorMessage`). The backend returns **all** that
  apply as a `Classifications: string[]` and the UI renders one badge
  per label rather than collapsing to a single primary label.
- Classification labels are **deterministic strings** shared verbatim
  between server (`AdminWorkflowInstancesController` constants) and
  client (`WorkflowClassification` union). An unknown value on the
  query string is silently ignored — the response then falls back to
  the broad `exceptionOnly` union or, if neither is set, full E9.1
  behaviour.
- Default stale threshold is **24h**. Clamped at the server to
  `[1h, 720h]` (30d ceiling) to keep the SQL predicate sensible.
- `LastErrorMessage` is the existing column on `WorkflowInstance`
  (entity line 61, also surfaced by E9.2's detail endpoint). E9.3
  promotes it from detail-only to also live on the list row so the UI
  can show a truncated preview without a second roundtrip per row.
- The `?selected=<id>` deep-link contract from E9.2 is reused
  unchanged. The drawer hydration is server-side; failure here renders
  the drawer's compact error state and never affects the list page.
- Page guard remains `requirePlatformAdmin` to match the existing
  `/workflows` page. The Flow admin endpoint additionally allows
  `TenantAdmin` scoped to their own tenant — that capability exists at
  the API for future enablement but is not surfaced in this UI phase.
- The "search" field re-uses the same E9.1 contains-style match
  (instance id / correlationKey / currentStepKey) — operators don't
  need to learn a second search behaviour.

## Repository / Architecture Notes

- Control Center is a Next.js 15 server-rendered app under
  `apps/control-center`. Pages are server components that call the
  shared `apiClient` (10-second tagged cache) which proxies through
  the gateway at `http://localhost:5010` (`/flow/...`). E9.3 uses the
  same path; no direct browser-to-Flow coupling was introduced.
- Auth/visibility is enforced in two layers: the page calls
  `requirePlatformAdmin()` (CC layer) and the Flow controller is
  decorated with `[Authorize(Policy = Policies.PlatformOrTenantAdmin)]`
  and re-applies an explicit `TenantId == w.TenantId` predicate on
  every join — same defence-in-depth pattern fixed in E9.2 round 1.
- Flow's `WorkflowInstance` entity already had `LastErrorMessage`
  (`apps/services/flow/backend/src/Flow.Domain/Entities/WorkflowInstance.cs:61`).
  No new domain field, migration, or workflow-engine behaviour was
  introduced.

## Exception / Stuck View Notes

- Route: `/workflows/exceptions` (separate from `/workflows`). A
  dedicated route was preferred over a tab so the URL itself is
  shareable in incident triage and the filter form can be specialised
  without polluting the E9.1 list.
- Header carries an `Operational triage` chip (subdued amber, not red)
  to set the right tone. A `← Back to all workflows` link is in the
  header; the symmetric `View exceptions` link sits in the `/workflows`
  header so operators can hop between the two surfaces in one click.
- Row layout (in addition to the standard E9.1 columns): a **Reason**
  column with one badge per classification (e.g. `Failed`,
  `Stuck >24h`, `Error present`), a truncated last-error preview under
  the workflow name (90 chars), and an **Age** column rendered as a
  short relative duration (`12m`, `4h`, `3d`) so an operator can answer
  "how long has this been in trouble?" at a glance.
- Row "Open →" reuses the E9.2 detail drawer via `?selected=<id>` —
  open/close keeps every filter and page in the URL.
- States: loading is the standard server-render path; empty renders a
  centred panel ("No workflows currently require attention. Stale
  threshold: Nh."); fetch errors render a compact red banner above the
  table without unmounting the rest of the shell.

## API / BFF Notes

- **Endpoint reused**: `GET /api/v1/admin/workflow-instances` (the same
  E9.1 admin list). New optional query params:

  | Param                 | Type   | Default | Notes                                                       |
  |-----------------------|--------|---------|-------------------------------------------------------------|
  | `exceptionOnly`       | bool   | `false` | When true, narrows to the union of all four classifications |
  | `classification`      | string | (none)  | One of `Failed` / `Cancelled` / `Stuck` / `ErrorPresent`. Implies `exceptionOnly`. Unknown values silently ignored. |
  | `staleThresholdHours` | int    | `24`    | Clamped to `[1, 720]`. Drives the `Stuck` predicate.        |

- **Response shape additions** (additive, non-breaking):
  - `AdminWorkflowInstanceListItem.LastErrorMessage` (string, nullable)
  - `AdminWorkflowInstanceListItem.Classifications` (string[], possibly empty)
  - `AdminWorkflowInstanceListResponse.StaleThresholdHours` (int)

  Existing E9.1 callers receive these new fields as well (always set);
  the existing fields are unchanged.

- **CC client method**: `controlCenterServerApi.workflows.listExceptions`
  returns the new `WorkflowInstancePagedResponse` (paged + echoed
  threshold). 10-second TTL, tag `cc:workflows`, identical to E9.1's
  `list` so they share invalidation.

- **No new endpoint, no new auth surface, no browser → Flow coupling**.
  The existing `[Authorize(Policy = Policies.PlatformOrTenantAdmin)]`
  decoration covers all triage paths.

## Detection Rules / Exception Logic

The four rules are applied in **two places** with deterministic
equivalence:

1. **Server-side LINQ filter** (when `exceptionOnly=true` or
   `classification=<X>`) narrows the query before paging.
2. **Per-row tagging** (`ClassifyRow`, in-memory after projection)
   evaluates the same rules and emits the labels that apply to each
   returned row, including for E9.1 callers that pass no triage params.

| Label          | Rule                                                                                       |
|----------------|--------------------------------------------------------------------------------------------|
| `Failed`       | `Status == "Failed"`                                                                       |
| `Cancelled`    | `Status == "Cancelled"`                                                                    |
| `Stuck`        | `Status in ("Active","Pending")` AND `(UpdatedAt ?? CreatedAt) < (UtcNow - staleHours)`    |
| `ErrorPresent` | `LastErrorMessage` is non-null AND non-empty                                               |

`exceptionOnly=true` with no `classification` is a server-side OR over
all four rules (a single SQL predicate so the index path stays sane).
A `classification=<X>` value implies `exceptionOnly=true`. Unknown
values are dropped to `null` so a stale or hand-mangled URL still
returns a deterministic response.

The stale cut-off (`UtcNow - staleHours`) is computed once per request
and reused both as the SQL constant and by the per-row classifier so
the badge a row carries always agrees with whether it was kept by the
server filter.

## Filter / Search / Status Notes

The filter row on `/workflows/exceptions` exposes:

- **Classification** — `Failed only` / `Cancelled only` / `Stuck only`
  / `Error present only` / All. Drives the `classification` query
  param.
- **Stale threshold** — `24h / 48h / 72h / 7d`. Drives
  `staleThresholdHours` (which both narrows the `Stuck` predicate and
  is echoed back in the response so the badge reads "Stuck >Nh").
- **Product** — same enum as the E9.1 page.
- **Status** — `Active / Pending / Cancelled / Failed`. Stacks with
  the classification filter (e.g. classification=Stuck + status=Active
  → only stuck Active rows).
- **Tenant id** — free-text scoping (PlatformAdmin only).
- **Search** — same contains-style behaviour as E9.1
  (id / correlationKey / currentStepKey).
- **Clear** — drops all filters back to defaults.

All filters are GET-only; the URL is the source of truth, so a triage
URL can be shared verbatim in an incident channel and reproduces the
exact view.

## Permission / Visibility Notes

- Page guard: `requirePlatformAdmin` (matches `/workflows`).
- API guard: `[Authorize(Policy = Policies.PlatformOrTenantAdmin)]`,
  unchanged from E9.1/E9.2.
- Tenant scoping: the controller bypasses the per-tenant EF query
  filter via `IgnoreQueryFilters()` and re-applies the appropriate
  predicate explicitly: PlatformAdmin sees every tenant (optionally
  narrowed by an inbound `tenantId`), TenantAdmin is forced to their
  own tenant regardless of any inbound `tenantId` param. The
  defence-in-depth `TenantId == w.TenantId` predicate on the
  definition and mapping joins is preserved for the new code path.
- Future tenant-admin UI surfacing is **deferred**; the API supports
  it but the page guard intentionally remains PlatformAdmin-only this
  phase.

## Known Issues / Gaps

- **Status filter overlap**. The `status` filter is intentionally not
  removed from the exceptions page even when `classification=Stuck`
  is set, so an operator can narrow Stuck to e.g. only `Active`. A
  contradictory combination (e.g. `classification=Failed`,
  `status=Active`) yields zero rows by design rather than an error.
- **No execution actions** (retry/replay/cancel). Explicitly out of
  scope for E9.3 — those belong on the product surfaces, not Control
  Center.
- **No bulk export, no analytics, no SLA engine integration** —
  deferred per the spec.
- **TenantAdmin UI surfacing is deferred**. The API allows it; the
  page guard is intentionally PlatformAdmin-only this phase.
- **Per-row classification in E9.1 pages**. The `WorkflowOperationsTable`
  on `/workflows` was not changed to render the new `classifications`
  field; this would be a small follow-up if desired.
- The "search" behaviour is contains-style on three string fields
  (id / correlationKey / currentStepKey). Searching the truncated
  `lastErrorMessage` text is not yet supported — a small server-side
  extension would add it.

## Recommendation

**Ready for follow-up admin actions.** E9.3 delivers a coherent,
read-only triage surface that:

- exposes a deterministic, documented set of classification rules and
  surfaces them in-row;
- reuses the E9.1 list endpoint and the E9.2 detail drawer rather
  than duplicating either;
- preserves the tenant scoping and PlatformAdmin guard from the
  prior phases;
- introduces no execution surface and no browser → Flow coupling;
- ships a stable URL contract that is shareable in incident triage.

The natural follow-ups are: (1) extending the row classification chips
into the main `/workflows` table; (2) adding a TenantAdmin-scoped
variant of this page; (3) layering admin actions (retry / cancel /
re-route) once their authorization model is agreed. None of those are
required by E9.3.
