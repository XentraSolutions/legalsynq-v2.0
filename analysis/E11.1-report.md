# E11.1 Report — Task Entity Model

> **Status: complete.** Foundational only. No engine wiring, no APIs,
> no SLA, no execution behaviour change. The new grain is durable,
> tenant-scoped, and queryable on day one so E11.2 can layer creation,
> assignment, and progression on top without a schema redesign.

## Scope Executed

- New domain entity `Flow.Domain.Entities.WorkflowTask` representing a
  first-class human-work item raised against a `WorkflowInstance` +
  `StepKey` pair.
- New string-constant value sets `WorkflowTaskStatus` (Open /
  InProgress / Completed / Cancelled) and `WorkflowTaskPriority`
  (Low / Normal / High / Urgent).
- EF Core mapping registered in `FlowDbContext`: dedicated `DbSet`,
  per-property column constraints, four indexes (operator inbox, user
  queue, role queue, per-instance list), Restrict-on-delete FK to
  `WorkflowInstance`, tenant query filter, server-side defaults for
  `Status` and `Priority`.
- `SaveChangesAsync` save-hook calls `WorkflowTask.EnsureValid()` on
  Added/Modified rows so the documented invariants fail loudly at
  write time.
- EF migration `20260417182606_AddWorkflowTaskE11_1` created and
  applied — table `flow_workflow_tasks` exists with all four indexes
  and the FK in place.

## Assumptions

1. **Distinct grain from the legacy `TaskItem`.** The pre-existing
   `TaskItem` lives at the *definition* layer (its FKs are
   `FlowDefinitionId` + `WorkflowStageId`) and predates the dedicated
   `WorkflowInstance` introduced in MERGE-P4. Renaming or repurposing
   it would have rewritten existing migrations and the Phase-3 surrogate
   semantics still consumed by `ProductWorkflowMapping`. Adding
   `WorkflowTask` alongside it isolates E11.1 entirely from legacy
   wiring and lets a future phase deprecate `TaskItem` on its own
   schedule. The two grains are clearly distinguished in code comments.
2. **Tenant isolation via the existing pattern.** `WorkflowTask`
   inherits from `AuditableEntity` → `BaseEntity` (TenantId), reuses
   the same SaveChangesAsync hook that stamps TenantId from the
   resolved `ITenantProvider`, and carries the same query filter as
   every other Flow entity. No new isolation surface.
3. **`StepKey` as a string, not an FK.** The key mirrors
   `WorkflowStage.Key` / `WorkflowInstance.CurrentStepKey`. Holding it
   as a string survives definition edits (renames / version bumps that
   would break a stage FK) and keeps cross-version task triage
   straightforward. Trade-off: no referential integrity on the step
   side; documented as a known gap.
4. **Multiple tasks per instance over time.** No uniqueness
   constraint on `(WorkflowInstanceId, StepKey)` — a re-entry into a
   step legitimately raises a fresh task. E11.2 may add a *partial*
   open-task uniqueness rule if product use cases demand it.
5. **Assignment placeholders only.** `AssignedUserId`,
   `AssignedRole`, `AssignedOrgId` are all nullable strings. No
   resolver, no mutual-exclusion rule, no notification. E11.2 owns
   that surface.
6. **Status as a string constant set, not an enum.** Matches the
   in-place convention (`WorkflowSlaStatus`, `WorkflowEngine.Status*`)
   so the column stays a short `varchar` and the lifecycle is
   trivially extensible (Blocked / OnHold / etc.) without a schema
   migration.
7. **Domain invariants are minimal.** Only the four required-field
   checks plus the two terminal-timestamp/status pairings the spec
   calls out are enforced. State-transition legality (no re-open of
   terminal, no skipping InProgress, …) is intentionally deferred to
   the lifecycle phase.
8. **No outbox / SLA / engine touch.** The new entity is invisible to
   `WorkflowEngine`, `OutboxProcessor`, `WorkflowSlaEvaluator`,
   `OutboxDispatcher`, and `OutboxWriter`. Smoke verified.

| `AssignedUserId` | `string(256)?` | Placeholder; nullable. |
| `AssignedRole` | `string(128)?` | Placeholder; nullable. |
| `AssignedOrgId` | `string(256)?` | Placeholder; nullable. |
| `CreatedAt` / `UpdatedAt` | `datetime?` | From `AuditableEntity`; stamped by save-hook. |
| `CreatedBy` / `UpdatedBy` | `string(256)?` | From `AuditableEntity`. |
| `StartedAt` | `datetime?` | Lifecycle marker. Nullable. |
| `CompletedAt` | `datetime?` | Lifecycle marker. Pairs with `Status=Completed`. |
| `CancelledAt` | `datetime?` | Lifecycle marker. Pairs with `Status=Cancelled`. |
| `CorrelationKey` | `string(256)?` | Optional external ref / idempotency key. |
| `MetadataJson` | `longtext?` | Forward-compatible JSON bag. |

Domain invariants enforced in `EnsureValid()` (called from save-hook):

- `WorkflowInstanceId` non-empty
- `StepKey` non-empty / non-whitespace
- `Status` non-empty
- `CompletedAt != null` ↔ `Status == "Completed"` (enforced both directions)
- `CancelledAt != null` ↔ `Status == "Cancelled"` (enforced both directions)

(`TenantId` is enforced by the existing save-hook.)

## Status Model Notes

`WorkflowTaskStatus` exposes four string constants:

- `Open` — created, awaiting pickup. Default for new rows.
- `InProgress` — operator actively working it.
- `Completed` — terminal success. Requires `CompletedAt`.
- `Cancelled` — terminal abort. Requires `CancelledAt`.

Helper predicates `IsKnown(...)` and `IsTerminal(...)` are provided
for use by future phases. `WorkflowTaskPriority` mirrors the same
pattern with `Low / Normal / High / Urgent` and a `Normal` default.

State-transition rules (e.g. legal `Open → InProgress → Completed`
graph, no re-open of terminal) are deliberately NOT enforced this
phase. Adding them belongs with the lifecycle handlers in E11.2.

## Indexing Notes

Four indexes were added — exactly the set the spec requires, no more:

| Name | Columns | Use |
| --- | --- | --- |
| `ix_flow_workflow_tasks_tenant_status` | `(TenantId, Status)` | Tenant-wide bucket counts (open/in-progress dashboards). |
| `ix_flow_workflow_tasks_tenant_user_status` | `(TenantId, AssignedUserId, Status)` | Operator inbox (`my open tasks`). |
| `ix_flow_workflow_tasks_tenant_role_status` | `(TenantId, AssignedRole, Status)` | Role queue (`unclaimed reviewer tasks`). |
| `ix_flow_workflow_tasks_instance` | `(WorkflowInstanceId)` | Per-instance task list (control-center detail drawer). |

Avoided over-indexing on `AssignedOrgId`, `Priority`, and
`CorrelationKey` for now. Each can be added in a tightly-scoped
follow-up the moment a real query pattern justifies it; pre-baking
indexes against speculative queries costs write throughput for no
proven read benefit.

## Known Issues / Gaps

1. **No referential integrity on `StepKey`.** The string is not an FK
   to `WorkflowStage.Key`. Trade-off justified above; an
   application-level validator could be added in E11.2 if drift becomes
   a real problem.
2. **No state-transition enforcement.** Only the terminal-timestamp /
   status pairing is enforced. Anything richer (no re-open of
   terminal, no skip of `InProgress`, assignee-only progression) lives
   in the lifecycle phase.
3. **No assignment uniqueness rule.** All three assignment fields are
   independently nullable; nothing forbids setting all three at once.
   The resolver in E11.2 should pick a precedence and enforce it.
4. **No `AssignedOrgId` / `Priority` / `CorrelationKey` index.** None
   added speculatively. Add the moment a real read pattern requires
   one.
5. **`TaskItem` legacy grain coexistence.** `TaskItem` continues to
   exist for `ProductWorkflowMapping` and the Phase-3 surrogate. A
   future phase should consolidate or formally deprecate it; out of
   scope here.
6. **Pre-existing Flow.Api migration auto-apply gap (carried forward
   from the E10.3 report).** Operators must apply Flow migrations
   manually between deploys. Unchanged by this phase but visible to
   anyone applying E11.1.

## Recommendation

**Ready for E11.2.** The data layer is in place, tenant scoping is
identical to existing grains, indexes cover the immediate operator
inbox / role queue / per-instance read patterns, and the platform is
demonstrably unchanged at runtime (engine + outbox + SLA evaluator
all clean). E11.2 can begin by introducing the creation surface
(workflow-driven raise, admin-driven raise) and the assignment
resolver without touching the schema again.
