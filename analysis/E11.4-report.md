# E11.4 Report ‚Äî Task Lifecycle (Start / Complete / Cancel)

> Status: implementation complete; pending architect review.

## Scope Executed

E11.4 activates the lifecycle behaviour for the `WorkflowTask` grain that
E11.1 introduced. A new application-layer service applies the four
allowed transitions ‚Äî

- `Open ‚Üí InProgress`
- `Open ‚Üí Cancelled`
- `InProgress ‚Üí Completed`
- `InProgress ‚Üí Cancelled`

‚Äî atomically, with strict validation and a single concurrency primitive
that guarantees exactly one transition succeeds when callers race. No
workflow-engine, outbox, SLA, notification, assignment, or external-API
behaviour is touched.

## Assumptions

- The four anchor states (`Open`, `InProgress`, `Completed`, `Cancelled`)
  shipped by E11.1 in `WorkflowTaskStatus` are the entire lifecycle this
  phase needs to model. No new states (`Blocked`, `OnHold`, ‚Ä¶) are
  introduced.
- `WorkflowTask.EnsureValid()` from E11.1 already encodes the
  status / terminal-timestamp pairing rule in *both* directions, so the
  invariants this service must respect are already documented and
  enforceable. The lifecycle service mirrors them in code.
- The existing scoped DI lifetime per request is the unit of work. The
  service does not introduce new transactions and does not coordinate
  with the engine ‚Äî there is no shared SaveChanges boundary to honour.
- `IFlowDbContext` already exposes `DbSet<WorkflowTask>`, which is enough
  surface for both the pre-check read (through the global tenant
  query filter) and the conditional UPDATE.
- The single best concurrency primitive available *without a schema
  change* is the EF Core 8 `ExecuteUpdateAsync` extension with a
  `WHERE Status = expected` predicate. This is exactly the
  "safe update pattern with state check in WHERE clause" listed by the
  spec as an acceptable approach. It avoids adding a `RowVersion`
  column / `IsConcurrencyToken()` config (which would require an EF
  migration) while still being atomic.
- `AssignedUserId` / `AssignedRole` / `AssignedOrgId` are NEVER read or
  written by the lifecycle service. The spec is explicit, and the
  `ExecuteUpdateAsync` `SetProperty` chain physically cannot touch them.

## Task Lifecycle Service Notes

**Interface** ‚Äî `IWorkflowTaskLifecycleService`:

```csharp
Task<WorkflowTaskTransitionResult> StartTaskAsync(Guid taskId, CancellationToken ct = default);
Task<WorkflowTaskTransitionResult> CompleteTaskAsync(Guid taskId, CancellationToken ct = default);
Task<WorkflowTaskTransitionResult> CancelTaskAsync(Guid taskId, CancellationToken ct = default);
```

Returns an immutable `WorkflowTaskTransitionResult` carrying
`(TaskId, PreviousStatus, NewStatus, TransitionedAtUtc)`. The record is
intentionally narrow: it exposes only the lifecycle facts the service
has authority over. Workflow-instance state is never included.

**Implementation** ‚Äî `WorkflowTaskLifecycleService`:

- Stateless. Three injected dependencies (`IFlowDbContext`,
  `IFlowUserContext`, `ILogger<‚Ä¶>`). Scoped DI lifetime aligns with the
  rest of `Flow.Application`.
- A single private helper (`TransitionAsync`) performs the
  pre-check + atomic compare-and-swap; the public methods are thin
  routers that fix the source / target / timestamp triple.
- `CancelTaskAsync` is the only entry point with two valid source
  states. It reads the current status first, picks the matching
  source, and then routes through the same helper ‚Äî keeping the
  compare-and-swap path single-source for every individual UPDATE.
- A small private `TimestampField` enum drives which lifecycle column
  is set in the `SetProperty` chain. This keeps the three branches
  visible in one place rather than scattered across helpers.
- Two-tier internals:
  - `TransitionAsync` = pre-check + CAS (used by Start / Complete).
    Mismatch at the pre-check raises
    `InvalidStateTransitionException` because there is exactly ONE
    valid source for these transitions, so any other status is
    genuinely an invalid source.
  - `ApplyCasAsync` = pure compare-and-swap with NO pre-check.
    Used as the inner step of `TransitionAsync` AND directly by
    Cancel. This split is load-bearing: Cancel reads the status
    once to choose between Open and InProgress as the expected
    source, and must NOT re-read inside the CAS step. If the row
    races from `Open` to `InProgress` between Cancel's read and the
    UPDATE, that interleaving is *contention* (`InProgress ‚Üí
    Cancelled` is itself valid), so the CAS will simply match zero
    rows and the caller gets a retryable
    `WorkflowTaskConcurrencyException` instead of a misclassified
    `InvalidStateTransitionException`.

**No mutation of foreign concerns**:

- Does NOT touch `WorkflowInstance`. Workflow progression on task
  completion is reserved for E11.7.
- Does NOT enqueue outbox messages. Notifications are reserved for
  E12.
- Does NOT touch SLA. The SLA worker still owns its own state and
  reads tasks it cares about independently.
- Does NOT touch assignment columns. There is no `SetProperty` for
  `AssignedUserId` / `AssignedRole` / `AssignedOrgId` anywhere in the
  service.

## Transition Rules

### Allowed (with their side-effects)

| From         | To           | Side-effects                                                         |
| ------------ | ------------ | -------------------------------------------------------------------- |
| `Open`       | `InProgress` | `StartedAt = StartedAt ?? now`, `UpdatedAt = now`, `UpdatedBy = ‚Ä¶`   |
| `Open`       | `Cancelled`  | `CancelledAt = now`, `UpdatedAt = now`, `UpdatedBy = ‚Ä¶`              |
| `InProgress` | `Completed`  | `CompletedAt = now`, `UpdatedAt = now`, `UpdatedBy = ‚Ä¶`              |
| `InProgress` | `Cancelled`  | `CancelledAt = now`, `UpdatedAt = now`, `UpdatedBy = ‚Ä¶`              |

### Disallowed (and how each is rejected)

| From         | To           | Outcome                                                                        |
| ------------ | ------------ | ------------------------------------------------------------------------------ |
| `Open`       | `Completed`  | `InvalidStateTransitionException("Open" ‚Üí "Completed")` from `Complete`        |
| `InProgress` | `Open`       | Not exposed ‚Äî no public method routes back to `Open`                           |
| `Completed`  | *anything*   | `InvalidStateTransitionException("Completed" ‚Üí ‚Ä¶)` from any of the 3 methods   |
| `Cancelled`  | *anything*   | `InvalidStateTransitionException("Cancelled" ‚Üí ‚Ä¶)` from any of the 3 methods   |
| `Open`       | `InProgress` *(retry while already InProgress)* | `InvalidStateTransitionException("InProgress" ‚Üí "InProgress")` |

### Failure cases (operational)

| Cause                                                     | Outcome                                                |
| --------------------------------------------------------- | ------------------------------------------------------ |
| Task id does not exist                                    | `NotFoundException(nameof(WorkflowTask), taskId)`      |
| Task exists but belongs to a different tenant              | `NotFoundException(nameof(WorkflowTask), taskId)` *(global query filter masks the row identically to a missing id, preventing cross-tenant id probing)* |
| Source status mismatch at pre-check time                  | `InvalidStateTransitionException(actual, target)`      |
| Source status changed between pre-check and UPDATE         | `WorkflowTaskConcurrencyException(taskId, expected)`   |

## Timestamp Handling Notes

- All timestamps are `DateTime.UtcNow` captured once per call and reused
  across every `SetProperty` in the same UPDATE ‚Äî so `UpdatedAt`,
  `StartedAt` / `CompletedAt` / `CancelledAt`, and the result record's
  `TransitionedAtUtc` all carry the same instant. No clock drift inside
  one transition.
- `StartedAt` uses the idempotent expression `t.StartedAt ?? now`. The
  current public surface cannot reach the InProgress state with
  `StartedAt` already set (Open ‚Üí InProgress is the only path), but the
  defensive expression means a future `Resume` operation could re-use
  the helper without losing the original start instant.
- `CompletedAt` and `CancelledAt` are unconditionally set. The terminal
  status / timestamp pairing rule from `WorkflowTask.EnsureValid()` is
  preserved: every `SetProperty(Status = Completed)` is paired with
  `SetProperty(CompletedAt = now)` in the same `ExecuteUpdateAsync`,
  and likewise for `Cancelled`. There is no code path that flips status
  without setting the matching timestamp.
- `UpdatedAt` is always written. `UpdatedBy` is updated to the current
  user when one is available (`_user.UserId`); when there is no user
  context (e.g. background scope), the existing value is preserved
  via `t => updatedBy ?? t.UpdatedBy`.
- `CreatedAt` / `CreatedBy` are NEVER touched.

The save-hook in `FlowDbContext.SaveChangesAsync` (audit-column
stamping + `EnsureValid`) does not run for `ExecuteUpdateAsync`. This is
why the service writes `UpdatedAt` and `UpdatedBy` explicitly and why
every transition pairs the status flip with the matching terminal
timestamp inline ‚Äî the in-DB state never violates the E11.1 invariants
because the UPDATE sets both columns atomically.

## Concurrency Notes

**Chosen approach**: atomic compare-and-swap at the database level via
EF Core 8 `ExecuteUpdateAsync` with the expected source `Status` in the
`WHERE` clause.

The full pattern per transition:

```csharp
// 1. Pre-check (AsNoTracking + global tenant query filter)
var current = await ReadCurrentStatusAsync(taskId, ct);
if (current != expectedStatus)
    throw new InvalidStateTransitionException(current, newStatus);

// 2. Atomic CAS ‚Äî the race-safe primitive
var affected = await _db.WorkflowTasks
    .Where(t => t.Id == taskId && t.Status == expectedStatus)
    .ExecuteUpdateAsync(s => s
        .SetProperty(t => t.Status, newStatus)
        .SetProperty(t => t.<lifecycleTimestamp>, now)
        .SetProperty(t => t.UpdatedAt, now)
        .SetProperty(t => t.UpdatedBy, t => updatedBy ?? t.UpdatedBy), ct);

if (affected == 0)
    throw new WorkflowTaskConcurrencyException(taskId, expectedStatus);
```

**Why this is safe under contention**:

- The database evaluates the `WHERE Id = @id AND Status = @expected`
  predicate atomically. Two concurrent transitions on the same task ‚Äî
  for example "operator A starts the task" and "operator B cancels the
  task" ‚Äî will both pass the pre-check (it has no isolation guarantee),
  but exactly ONE of the conditional UPDATEs will match a row. The
  loser's UPDATE matches zero rows and we raise
  `WorkflowTaskConcurrencyException`. There is no silent overwrite and
  no "last writer wins".
- The pre-check is purely a UX concern ‚Äî it converts the common case
  ("user double-clicks Complete on an already-Completed task") into a
  clean `InvalidStateTransitionException` with the actual current state
  in the message, instead of a bare concurrency error.
- Tenant scoping is enforced on the read path by the global
  `HasQueryFilter` on `WorkflowTask`. The same query filter is applied
  to `ExecuteUpdateAsync` (EF runs the IQueryable through the standard
  pipeline), so a cross-tenant attempt cannot match a row even if the
  caller knows the task id. The pre-check returns
  `NotFoundException` for the same reason.

**Why not RowVersion / `IsConcurrencyToken()`**:

- Would require a new EF migration and a new column on
  `flow_workflow_tasks`. The compare-and-swap on `Status` covers the
  exact failure mode the spec calls out (concurrent transitions
  corrupting state) without any schema change.
- A `Status`-as-concurrency-token configuration would still need a
  migration (model snapshot diff), even though it produces no DDL on
  MySQL ‚Äî too much surface for a phase whose explicit constraint is
  "no regression in workflow engine, no regression in task creation or
  assignment, background workers unaffected".

**Partial-update guarantee**: every transition sets *all* of
`Status`, the matching terminal timestamp (when applicable),
`UpdatedAt`, and `UpdatedBy` in the SAME `SetProperty` chain ‚Äî i.e.
the SAME single SQL UPDATE. There is no path that writes only some of
these columns. If the predicate matches zero rows, no column is
written at all.

### Functional / behavioural (verified by inspection)

- **Allowed transitions**: every public method routes through
  `TransitionAsync` with a single `(expected, new)` pair that maps to
  one of the four allowed transitions; nothing else is constructible
  through the public surface.
- **Disallowed transitions**: each public method's `expectedStatus` is
  fixed (`Open` for Start, `InProgress` for Complete, `Open|InProgress`
  for Cancel); any other current status fails the pre-check and
  raises `InvalidStateTransitionException` before the UPDATE runs.
- **Timestamps**: every status flip is paired with its matching
  terminal timestamp in the same UPDATE; `UpdatedAt` is always
  written; `StartedAt` is null-coalesced.
- **Concurrency**: two concurrent calls on the same task can both pass
  the pre-check, but the `WHERE ‚Ä¶ AND Status = expected` predicate
  guarantees that exactly one UPDATE matches a row ‚Äî the other gets
  `affected == 0` and raises `WorkflowTaskConcurrencyException`. No
  partial state, no silent overwrite.
- **Tenant safety**: the pre-check uses `_db.WorkflowTasks` directly,
  which carries the global tenant `HasQueryFilter`; the
  `ExecuteUpdateAsync` chain does the same. Cross-tenant id probing
  surfaces as `NotFoundException`, identical to a missing id.
- **Assignment fields untouched**: no `SetProperty` references
  `AssignedUserId` / `AssignedRole` / `AssignedOrgId`; cannot regress.
- **No engine / outbox / SLA / notification touch**: the service
  references `IFlowDbContext` only, no engine / outbox / SLA / notification
  interfaces are imported.

## Known Issues / Gaps

- **No public API / controller surface yet.** The lifecycle service is
  registered in DI and is consumable from any other Application service,
  but no admin or operator endpoint calls it in this phase. The spec
  explicitly defers external task APIs; the operator-facing surface
  ("My Tasks") is reserved for E11.5.
- **No unit / integration tests.** Behaviour is verified by inspection
  + clean DI resolution at boot. Adding focused tests (precedence,
  concurrent CAS, tenant isolation, terminal-state rejection) is a
  natural follow-up that can ship alongside the E11.5 API surface.
- **No outbox event on transition.** `task.started` / `task.completed` /
  `task.cancelled` are reserved for E12 notifications. The service
  intentionally does not stage anything on the outbox.
- **No workflow-instance progression on completion.** This is the
  E11.7 deliverable and is explicitly out of scope here. Operators
  completing a task today move only the task; the workflow instance
  is unaffected.
- **Save-hook bypass for these UPDATEs.** `ExecuteUpdateAsync` does
  not run `EnsureValid` or the audit-column stamper. We compensate by
  always pairing status / terminal-timestamp in the same UPDATE and by
  setting `UpdatedAt` / `UpdatedBy` explicitly. A future hook (e.g.
  outbox staging on transition) would need to use a tracked
  `Update` + `SaveChangesAsync` instead ‚Äî easy to add without
  changing the public interface.
- **`IFlowUserContext.UserId` is preserved when null.** During
  background-scope invocations (none today, but possible in the
  future) the UPDATE leaves the existing `UpdatedBy` in place rather
  than blanking it. Documented inline.

## Recommendation

Ready to proceed to **E11.5 (My Tasks API)**. The lifecycle is
mechanically complete, deterministic, race-safe, tenant-scoped, and
has zero coupling to the workflow engine, outbox, SLA, or assignment
subsystems. E11.5 can build a thin operator-facing controller on top
of the existing service interface without re-opening any of the
invariants documented here.
